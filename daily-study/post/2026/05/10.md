# 📅 TIL 2026-05-10: 셰이더(Shader) 전체 지도 — GPU 위에서 실행되는 렌더링의 언어

> **학습 내용**: 셰이더의 정의부터 GPU 실행 모델, 파이프라인 각 셰이더 종류, 현대 확장(Mesh/RT/Compute), 셰이딩 언어

## 1️⃣ 셰이더란 무엇인가? — 정의와 GPU vs CPU

**셰이더(Shader)** 는 **렌더링 파이프라인을 통과하는 데이터에 적용되는 프로그래밍 가능한 연산**이다. 한마디로 "GPU 위에서 돌아가는 작은 프로그램"이다.

이름의 기원은 1988년 Pixar의 RenderMan에서 빛과 그림자로 입체감을 표현하는(Shading) 절차적 프로그램에서 유래했지만, 오늘날 셰이더가 하는 일은 훨씬 넓다:

| 역할 | 설명 |
|------|------|
| 3D 좌표 변환 | Model → World → View → Clip Space 변환 |
| 조명·음영 계산 | PBR BRDF, Cook-Torrance, Phong 등 |
| 텍스처 샘플링 | 텍스처에서 색상·법선 데이터 읽기 |
| 절차적 효과 | 노이즈, 파동, 디졸브, 셀 셰이딩 |
| 포스트 프로세싱 | 블러, 색보정, 피사계심도, TAA |
| 범용 병렬 연산 | 물리 시뮬레이션, AI 추론(DLSS 등) |

### CPU vs GPU — 근본적인 차이

| 구분 | CPU | GPU(셰이더) |
|------|-----|-----------|
| 병렬성 | 소수 스레드 (8~32코어) | 수천~수만 스레드 동시 실행 |
| 분기(if/else) | 낮은 비용 | 높은 비용 (Warp Divergence) |
| 실행 패러다임 | MIMD (다중 명령·다중 데이터) | SIMT (단일 명령·다중 스레드) |
| 강점 | 복잡한 순서 제어, 분기 많은 로직 | 대규모 동일 연산 병렬 처리 |
| 주요 활용 | 게임 로직, AI, 물리 CPU 시뮬 | 정점 변환, 픽셀 색상, 이미지 처리 |

## 2️⃣ GPU 실행 모델 — SIMT, Warp, Divergence

### Warp / Wavefront — GPU의 기본 실행 단위

GPU는 셰이더 스레드를 **묶음 단위**로 실행한다:

- **Nvidia**: **Warp** — 32개 스레드를 한 묶음으로 동일 명령어 실행
- **AMD**: **Wavefront** — 64개 스레드 (RDNA에서는 32 또는 64)

```
화면 해상도 1920×1080 → 약 207만 픽셀 → Fragment Shader가 동시 실행
Warp 0 : 픽셀 (0,0)~(31,0)   병렬 처리
Warp 1 : 픽셀 (32,0)~(63,0)  병렬 처리
...  (64,800개 Warp가 순차 스케줄링)
```

### Warp Divergence — if/else가 비싼 이유

Warp 내 32 스레드는 **반드시 같은 명령어를 실행**해야 한다. if/else가 있으면:

```hlsl
// 이 코드의 실제 GPU 실행 순서
if (roughness > 0.5) {
    // Path A: 이 경로 해당 픽셀만 활성, 나머지 마스킹
}
else {
    // Path B: 이 경로 해당 픽셀만 활성, 나머지 마스킹
}
// 결과: A+B 두 경로 모두 실행 시간 소비 → 최악 효율 50%
```

**해결 전략**: 분기를 `lerp()`로 대체해 수학적 표현으로 변환.

### 레이턴시 숨기기 (Latency Hiding)

텍스처 샘플링 시 메모리 레이턴시(100~300사이클) 동안:
1. GPU는 즉시 **다른 Warp로 전환**하여 연산 지속
2. 텍스처 데이터 도착 → 원래 Warp 재개

이를 위해 GPU는 코어 수보다 훨씬 많은 **In-Flight Warp**를 동시 유지. 따라서 충분한 스레드 수(Occupancy)가 GPU 효율의 핵심이다.

## 3️⃣ 역사 — Fixed Function에서 Programmable GPU까지

```
1988년   Pixar RenderMan → "셰이더" 용어 최초 도입 (오프라인 렌더링)
  │
1990년대 고정 기능(Fixed-Function) 파이프라인
         3dfx Voodoo, ATI Rage — 조명 공식 수정 불가
  │
2001년   Nvidia GeForce 3 (NV20) → 최초 프로그래밍 가능 픽셀 셰이더
2002년   DirectX 9 + HLSL → 개발자가 직접 조명 공식 작성 가능
  │
2006년   DirectX 10 + 통합 셰이더 모델 → 지오메트리 셰이더, 범용 셰이더 코어
2009년   DirectX 11 + 테셀레이션 셰이더 (Hull + Domain)
  │
2018년   DirectX Raytracing (DXR) → 하드웨어 RT 코어, 레이 트레이싱 셰이더
2018년   Nvidia Turing → 메시 셰이더 & 태스크 셰이더
2020년   DX12 Ultimate → AMD RDNA 2도 메시 셰이더 지원
```

> [!tip] 고정 기능의 한계
> 1990년대에는 Gouraud Shading인지 Flat Shading인지 "스위치"만 선택할 수 있었다. 조명 수식 자체를 바꾸려면 하드웨어를 바꿔야 했다. 프로그래밍 가능한 셰이더의 등장은 이 고정된 공식을 개발자의 코드로 완전히 대체한 혁명이었다.

## 4️⃣ 그래픽 파이프라인 셰이더 종류

### 파이프라인 흐름

```
[CPU: Draw Call]
    ↓
정점 버퍼 (Vertex Buffer)
    ↓
① 정점 셰이더 (Vertex Shader)          ← 프로그래밍 가능
    ↓
② 테셀레이션 (선택)                     ← 프로그래밍 가능
   Hull Shader → Tessellator(고정) → Domain Shader
    ↓
③ 지오메트리 셰이더 (선택)              ← 프로그래밍 가능
    ↓
래스터라이저 (Rasterizer)               ← 고정 기능
    ↓
④ 프래그먼트/픽셀 셰이더               ← 프로그래밍 가능
    ↓
Output Merger / ROP                     ← 고정 기능(설정 가능)
    ↓
프레임 버퍼 → 화면
```

---

### ① 정점 셰이더 (Vertex Shader)

**3D 좌표를 2D 화면 좌표로 변환**. 정점(Vertex)당 1회 실행.

```hlsl
struct VertexOutput {
    float4 position : SV_Position; // 반드시 Clip Space 출력
    float3 worldPos : TEXCOORD0;
    float3 normal   : TEXCOORD1;
    float2 uv       : TEXCOORD2;
};

cbuffer TransformCB : register(b0) {
    float4x4 modelMatrix;
    float4x4 viewMatrix;
    float4x4 projMatrix;
};

VertexOutput VS_Main(VertexInput input) {
    VertexOutput o;
    float4 world    = mul(modelMatrix, float4(input.position, 1.0));
    float4 view     = mul(viewMatrix,  world);
    o.position      = mul(projMatrix,  view);   // Clip Space
    o.worldPos      = world.xyz;
    o.normal        = mul((float3x3)modelMatrix, input.normal);
    o.uv            = input.uv;
    return o;
}
```

좌표 공간 변환 흐름:
```
Model Space → (Model Matrix) → World Space
→ (View Matrix) → View Space → (Projection Matrix) → Clip Space
→ (÷w, 고정기능) → NDC(-1~1) → (Viewport Transform) → Screen Space
```

정점 셰이더의 제약: **새로운 정점 생성 불가** (이는 테셀레이션/지오메트리 셰이더의 역할).

---

### ② 테셀레이션 셰이더 (Hull + Tessellator + Domain)

런타임에 폴리곤을 동적으로 세분화 — **거리 기반 자동 LOD**의 핵심.

| 단계 | 역할 | 프로그래밍 여부 |
|------|------|-------------|
| Hull Shader | Tessellation Level 결정 (카메라 거리 기반) | 프로그래밍 가능 |
| Tessellator | 실제 분할 수행 (새 정점 UV 생성) | 고정 기능 |
| Domain Shader | 새 정점 위치 계산 + Displacement 적용 | 프로그래밍 가능 |

```
카메라 가까이: Level 64 → 삼각형 수 ×64
카메라 멀리: Level 4  → 삼각형 수 ×4   (자동 조정)
```

활용: **Displacement Mapping** — Normal Map이 조명만 속이는 것과 달리, 테셀레이션과 결합하면 실루엣까지 바뀌는 실제 지오메트리 변형 가능.

---

### ③ 지오메트리 셰이더 (Geometry Shader)

**입력 프리미티브 → 0개 이상의 새 프리미티브 생성**. 삼각형 전체를 보며 작동.

```hlsl
[maxvertexcount(4)]
void GS_BillboardParticle(point VertexOutput input[1],
                           inout TriangleStream<VertexOutput> stream) {
    // 점(Point) 하나 → 카메라를 향하는 사각형(Quad) 4개 정점 생성
    float3 up    = float3(0, 1, 0);
    float3 right = normalize(cross(up, CameraDir)) * ParticleSize;
    float3 pos   = input[0].worldPos;
    // 4개 코너 정점 Append → 삼각형 스트립으로 출력
    AppendQuadVertex(stream, pos - right - up * ParticleSize);
    AppendQuadVertex(stream, pos + right - up * ParticleSize);
    AppendQuadVertex(stream, pos - right + up * ParticleSize);
    AppendQuadVertex(stream, pos + right + up * ParticleSize);
}
```

주요 활용: 파티클 시스템(점→빌보드 쿼드), 섀도우 볼륨, 큐브맵 단일 패스 렌더링.

> [!warning] 지오메트리 셰이더의 한계
> 많은 프리미티브를 생성할수록 GPU 캐시 효율이 급격히 저하. 현대에는 **Mesh Shader**로 대체 추세.

### ④ 프래그먼트/픽셀 셰이더 (Fragment/Pixel Shader)

래스터화 후 생성된 **각 프래그먼트의 최종 색상을 계산**. 셰이더 개발자가 가장 많이 작성하는 단계.

```hlsl
float4 PS_PBR(FragmentInput input) : SV_Target {
    // 텍스처 샘플링
    float3 albedo    = AlbedoTex.Sample(Sampler, input.uv).rgb;
    float  roughness = ORMTex.Sample(Sampler, input.uv).g;
    float  metallic  = ORMTex.Sample(Sampler, input.uv).b;
    float3 N         = UnpackNormal(NormalTex.Sample(Sampler, input.uv));

    // Cook-Torrance BRDF
    float3 L  = normalize(LightDir);
    float3 V  = normalize(CameraPos - input.worldPos);
    float3 H  = normalize(L + V);

    float3 F0 = lerp(float3(0.04, 0.04, 0.04), albedo, metallic);
    float3 F  = F_Schlick(F0, max(dot(V, H), 0.0));
    float  D  = D_GGX(max(dot(N, H), 0.0), roughness);
    float  G  = G_Smith(max(dot(N, V), 0.0001), max(dot(N, L), 0.0), roughness);

    float3 specular = (D * G * F) / (4.0 * max(dot(N,V),0.0001) * max(dot(N,L),0.0) + 0.0001);
    float3 diffuse  = (1.0 - F) * (1.0 - metallic) * albedo / 3.14159;
    float3 Lo       = (diffuse + specular) * LightColor * max(dot(N, L), 0.0);

    return float4(Lo, 1.0);
}
```

픽셀 셰이더의 다양한 출력:

| 출력 | 용도 |
|------|------|
| SV_Target0 (단일 출력) | 화면 직접 출력 (Forward Rendering) |
| MRT (다중 렌더 타겟) | G-Buffer — Albedo·Normal·Roughness 동시 출력 (Deferred Rendering) |
| discard | 해당 프래그먼트 폐기 (알파 클리핑, 나뭇잎·철망) |
| SV_Depth 수정 | 커스텀 깊이값 출력 (ImpostorRendering 등) |

## 5️⃣ 현대 셰이더 확장 — Mesh Shader와 Ray Tracing

### 메시 셰이더 & 태스크 셰이더 (2018~)

**기존 파이프라인의 문제**: CPU가 정점 데이터를 준비 → CPU 병목 발생. Frustum Culling, LOD 결정도 CPU에서 처리 → 비효율.

**메시 셰이더의 해결**:

```
[태스크 셰이더]  → GPU에서 메시렛 단위 Frustum/Occlusion Culling
    ↓ (필요한 메시렛만 전달)
[메시 셰이더]   → GPU에서 직접 지오메트리 생성·변환
    ↓
[래스터라이저] → 이후 동일
```

- **메시렛(Meshlet)**: 메시를 64~128개 정점 청크로 분할한 단위
- CPU 드로우 콜 없이 GPU가 독립적으로 컬링·LOD 수행
- 전통 파이프라인 대비 삼각형 처리 성능 **최대 10배 향상** (Nvidia 발표)

### 레이 트레이싱 셰이더 (DXR, 2018~)

광선 기반 렌더링을 위한 새로운 셰이더 타입 5종:

| 셰이더 타입 | 역할 |
|-----------|------|
| Ray Generation | 각 픽셀에서 광선 발사 |
| Closest-Hit | 가장 가까운 충돌 지점에서 조명 계산 |
| Any-Hit | 반투명·알파 테스트 처리 |
| Miss | 광선이 아무것도 맞지 않을 때 (스카이박스) |
| Intersection | 커스텀 기하(구·NURBS) 교차 판정 |

핵심 기반: **BVH(Bounding Volume Hierarchy)** — 메시를 계층적 경계 박스로 구성해 광선-삼각형 검사를 O(n) → O(log n)으로 가속. Nvidia RT Core / AMD Ray Accelerator가 BVH 순회를 하드웨어로 처리.

## 6️⃣ 컴퓨트 셰이더와 GPGPU

그래픽 파이프라인 **외부**에서 GPU를 범용 병렬 연산에 활용하는 셰이더.

```hlsl
// 컴퓨트 셰이더 예시: Bloom 밝기 추출
[numthreads(8, 8, 1)]
void CS_BloomThreshold(uint3 id : SV_DispatchThreadID) {
    float3 color        = InputTexture[id.xy].rgb;
    float  brightness   = dot(color, float3(0.2126, 0.7152, 0.0722));
    // 임계값 초과 픽셀만 통과
    OutputTexture[id.xy] = float4(brightness > Threshold ? color : 0, 1);
}
```

**디스패치(Dispatch)** 모델:
```
Dispatch(128, 128, 1)     → 16,384 Work Group
[numthreads(8, 8, 1)]    → Work Group당 64 스레드
총 스레드: 1,048,576 스레드 동시 실행
```

현대 렌더러에서 컴퓨트 셰이더로 구현하는 것들:

| 용도 | 예시 |
|------|------|
| 포스트 프로세싱 | Bloom, SSAO, DoF, TAA |
| 조명 컬링 | Tiled/Clustered Deferred Lighting |
| 물리 시뮬레이션 | 천, 유체, 파티클 |
| AI 추론 | DLSS, FSR — GPU 업스케일링 |
| GPU 스키닝 | CPU 대신 GPU에서 골격 변형 |

**Shared Memory** — 워크그룹 내 공유하는 온칩 메모리(L1 수준 속도). 병렬 합산(Reduction), 컨볼루션 등 인접 데이터를 자주 접근하는 알고리즘에 필수.

## 7️⃣ 셰이딩 언어와 SPIR-V

### 주요 셰이딩 언어

| 언어 | 개발사 | 대상 API | 특징 |
|------|--------|---------|------|
| **HLSL** | Microsoft | DirectX 9~12, Vulkan | C++ 유사, 게임 산업 표준 |
| **GLSL** | Khronos | OpenGL, Vulkan | C 기반, 크로스 플랫폼 |
| **GLSL ES** | Khronos | OpenGL ES, WebGL | 모바일·웹 |
| **MSL** | Apple | Metal | Swift/ObjC 문법 유사 |
| **WGSL** | W3C | WebGPU | 웹 전용 현대적 언어 |

### SPIR-V — 중간 표현의 역할

```
HLSL 소스 ─→ [DXC 컴파일러]    ─┐
GLSL 소스 ─→ [glslang 컴파일러] ─┼─→ SPIR-V 바이너리 ─→ [드라이버] ─→ GPU ISA
Slang 소스 ─→ [Slang 컴파일러] ─┘
```

SPIR-V의 가치:
- **플랫폼 독립**: 동일 SPIR-V를 Nvidia, AMD, Intel GPU에서 모두 실행
- **빠른 드라이버 컴파일**: 소스 파싱 생략, 최적화에만 집중
- **보안**: 소스 코드 노출 없이 배포 가능
- **2024년**: Microsoft DirectX 12도 SPIR-V를 기본 교환 형식으로 채택 예정

## 8️⃣ 셰이더 최적화 핵심 원칙

### 원칙 1. 분기 최소화 → lerp 대체

```hlsl
// 비효율: Warp Divergence 유발
float shadow;
if (isInShadow)
    shadow = 0.0;
else
    shadow = 1.0;

// 효율: 수학 표현으로 분기 제거
float shadow = ShadowMap.SampleCmpLevelZero(ShadowSampler, uv, depth);
```

### 원칙 2. 텍스처 샘플링 수 최소화 (채널 패킹)

```hlsl
// 나쁜 예: 3번 샘플링
float ao        = AOTex.Sample(s, uv).r;
float roughness = RoughnessTex.Sample(s, uv).r;
float metallic  = MetallicTex.Sample(s, uv).r;

// 좋은 예: ORM 채널 패킹 → 1번 샘플링
float3 orm      = ORMTex.Sample(s, uv).rgb;
// R:AO, G:Roughness, B:Metallic
```

### 원칙 3. 정밀도 선택

```hlsl
// 위치·행렬: float (float32) 필수 — 정밀도 손실 시 깜박임/떨림
float4 worldPos = mul(modelMatrix, float4(pos, 1.0));

// 색상·UV: half (float16) 충분 — 모바일에서 약 2배 빠름
half3 albedo    = (half3)AlbedoTex.Sample(s, uv).rgb;
```

### 원칙 4. Permutation Explosion 주의

```hlsl
// Static Branch (컴파일 타임 분기) → 셰이더 변형(Permutation) 생성
#ifdef USE_NORMAL_MAP
    float3 N = UnpackNormal(NormalMap.Sample(s, uv));
#else
    float3 N = input.normal;
#endif
```

Static Branch가 k개면 최대 2^k 변형 생성 → 컴파일 시간·메모리 폭발. UE5의 Static Switch Parameter도 동일 원리. **필요한 분기만 최소화**가 핵심.

## 9️⃣ 셰이더 디자이너 · 그래픽스 엔지니어 관점

> [!example] 관점 1: 셰이더는 표면 재질의 수학적 언어다
> 셰이더 디자이너에게 셰이더는 "어떤 표면이 어떻게 빛과 상호작용하는가"를 수식으로 표현하는 도구다. 금속이 반짝이는 이유, 물이 굴절하는 이유, 피부가 반투명하게 빛나는 이유 — 이 모든 것이 BRDF 수식으로 귀결된다. UE5 Material Editor의 노드 그래프는 이 수식의 시각화이지, 본질이 아니다. 노드를 연결할 줄 알더라도, 그 뒤에서 무슨 HLSL이 생성되고, GPU에서 어떻게 실행되는지를 이해하는 것이 **아마추어와 전문가의 경계**다.

> [!example] 관점 2: 그래픽스 엔지니어에게 셰이더는 성능 예산과의 타협이다
> 이론적으로 완벽한 PBR 공식이 있어도, 60fps 유지를 위해서는 근사(Approximation), LOD, Baked Lighting, Precomputed Data 등으로 타협해야 한다. RenderDoc, PIX, Nsight 같은 GPU 프로파일러로 **ALU 바운드인지, 텍스처 바운드인지, 메모리 바운드인지**를 정확히 진단한 후 올바른 방향을 잡는 것이 엔지니어의 역할이다. "느리다"고 무작정 셰이더 코드를 줄이는 것은 Fragment-bound일 때만 효과가 있다.

> [!example] 관점 3: 컴퓨트 셰이더를 모르면 현대 렌더러의 절반을 모른다
> Deferred Lighting의 타일 컬링, TAA의 히스토리 누적, SSAO, Bloom, DoF — 현대 AAA 렌더러의 핵심 알고리즘 대부분이 컴퓨트 셰이더다. 메시 셰이더의 태스크 셰이더도 컴퓨트 모델 기반이다. 그래픽스 엔지니어 취업 인터뷰에서 "Compute Shader로 뭘 구현해봤나요?"는 이제 기본 질문이다.

> [!example] 관점 4: Mesh Shader는 단순 기능 추가가 아닌 패러다임 전환이다
> Mesh Shader의 진정한 의미는 **CPU 병목을 GPU로 옮기는 아키텍처 전환**이다. 전통 파이프라인에서 CPU가 담당하던 Frustum Culling, LOD 선택, 드로우 콜 생성을 이제 GPU의 태스크 셰이더가 병렬로 처리한다. 이는 Nanite(UE5)가 수억 개의 삼각형을 실시간 처리하는 핵심 원리 중 하나다. 셰이더 디자이너도 이 패러다임 변화를 이해하고 있어야 엔진 팀과 올바른 대화가 가능하다.

## 🔟 Key Insights

> [!success] 인사이트 1: Warp Divergence가 "왜 셰이더에서 if/else를 피하는가"의 답이다
> HLSL 셰이더 튜토리얼에서 "if/else를 lerp로 대체하라"는 조언의 이유를 이전까지는 막연히 따랐다. 오늘 SIMT 실행 모델과 Warp Divergence를 이해하고 나서야, 이 규칙이 **32개 스레드가 동일 명령어를 공유해야 하는 하드웨어 필연성**에서 나온다는 것을 알았다. 앞으로 셰이더를 작성할 때 분기 최소화는 "규칙"이 아니라 "이유가 있는 선택"으로 다가올 것이다.

> [!success] 인사이트 2: SPIR-V는 크로스 플랫폼 셰이더 생태계의 언어 공통분모다
> "HLSL이냐 GLSL이냐"는 논쟁이 사실 크게 중요하지 않아졌다. 둘 다 SPIR-V로 컴파일되고, SPIR-V에서 각 플랫폼 드라이버가 최적화된 GPU 코드를 생성한다. 이 구조를 이해하면 멀티 플랫폼 셰이더 개발의 전략이 명확해진다 — "어떤 언어로 쓸지"보다 "어떤 IR로 수렴하는지"가 더 중요한 질문이다.

> [!success] 인사이트 3: 렌더링 방정식은 모든 셰이더 코드의 근원이다
> Cook-Torrance BRDF의 D·G·F 세 항, PBR의 BaseColor·Roughness·Metallic — 이 모든 파라미터가 1986년 Kajiya의 렌더링 방정식에서 출발했다. 수식 하나가 수십 년의 그래픽스 기술로 구체화된 것이다. 셰이더를 깊이 이해하려면 결국 이 방정식으로 돌아와야 한다. 석사 과정에서 수학적 기반을 제대로 익히는 것이 장기적으로 가장 중요한 투자임을 다시 느꼈다.

> [!success] 인사이트 4: 셰이더 타입의 다양화는 렌더링 한계를 GPU 쪽으로 밀어내는 역사다
> 고정 기능 → 정점/픽셀 셰이더 → 테셀레이션 → 지오메트리 → 메시/태스크 → 레이 트레이싱 → 텐서. 각 셰이더 타입의 추가는 기존 파이프라인의 특정 병목(CPU 컬링, 지오메트리 생성, 픽셀 정확 조명)을 GPU의 병렬 처리로 옮기는 과정이었다. 이 역사적 맥락을 이해하면 앞으로 어떤 새로운 셰이더 타입이 등장할지도 예상할 수 있다.
