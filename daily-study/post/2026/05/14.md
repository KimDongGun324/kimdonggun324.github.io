## UV 매핑 — 3D 표면을 2D로 펼치다

**UV 매핑(UV Mapping)** 은 3D 모델의 표면을 2D 평면으로 "언래핑"하여 텍스처를 입힐 좌표 체계를 만드는 작업이다. 종이 상자를 가위로 잘라 펼치는 과정의 역방향이다.

핵심 개념:
- **심(Seam)**: 3D 표면을 자르는 봉제선. 시각적으로 덜 보이는 위치에 배치해야 아티팩트가 줄어든다.
- **UV 아일랜드(UV Island)**: 심으로 잘린 각각의 조각. 0~1 UV 공간 안에 배치된다.
- **텍셀 밀도(Texel Density)**: 메시 표면 단위 면적당 할당된 픽셀 수. 균일하지 않으면 일부 영역이 흐리거나 과도하게 선명하게 보인다.

주요 언래핑 알고리즘:

| 알고리즘 | 원리 | 특징 |
|---|---|---|
| **LSCM** | 최소 제곱 등각 맵 | 각도 보존 우선, 면적 왜곡 허용 |
| **ABF** | 각도 기반 평탄화 | 각도 왜곡 최소, 연산 비용 높음 |
| **Smart UV** | 휴리스틱 자동 분할 | 빠르지만 품질 불균일 |
| **Triplanar** | XYZ 3방향 투영 혼합 | UV 없이 실시간 처리 — 지형에 적합 |

```hlsl
// Triplanar Mapping — UV 없는 지형·절벽에 활용
float3 Triplanar(Texture2D tex, SamplerState s, float3 worldPos, float3 worldNormal)
{
    float3 blend   = abs(worldNormal);
    blend          = pow(blend, 8.0);
    blend          = blend / (blend.x + blend.y + blend.z);

    float3 xProj   = tex.Sample(s, worldPos.yz).rgb;
    float3 yProj   = tex.Sample(s, worldPos.xz).rgb;
    float3 zProj   = tex.Sample(s, worldPos.xy).rgb;
    return xProj * blend.x + yProj * blend.y + zProj * blend.z;
}
```

> [!tip] UV 고급 시스템
> - **UDIM**: U축 방향으로 타일 번호(1001~1110)를 늘려 고해상도 에셋(VFX·영화)을 관리하는 시스템
> - **라이트맵 UV (UV1)**: 라이팅 정보 전용 채널. 겹침 없이, 비율 비례로, 4픽셀 이상 패딩 필수
> - **Trim Sheet**: 몰딩·프레임 등 모듈형 요소를 가로 스트립으로 배열한 텍스처

## UV 좌표계와 텍스처 공간

**UV 좌표** 는 3D 정점에 할당되는 2D 좌표 쌍이다. 텍스처 해상도에 독립적인 \[0, 1\] 정규화 범위를 사용한다.

```
UV 좌표 (0.5, 0.5)  →  항상 텍스처 중앙
                        (512px 텍스처든 4096px 텍스처든 동일)
```

**OpenGL vs DirectX V축 차이**:

| API | 원점(0,0) 위치 | 비고 |
|---|---|---|
| OpenGL / Vulkan | 왼쪽 하단 | 수학적 좌표계 |
| DirectX / Unreal Engine | 왼쪽 상단 | 화면 좌표계 |

GLSL ↔ HLSL 간 텍스처 이식 시 V축 반전(`1.0 - v`)이 필요하다.

**구면 UV 수식**: 행성·눈동자처럼 구형 오브젝트에 투영할 때 다음 수식을 사용한다.

$$u = 0.5 + \frac{\arctan2(d_z,\, d_x)}{2\pi}, \qquad v = 0.5 + \frac{\arcsin(d_y)}{\pi}$$

극점(pole) 근방에서 U축이 수렴하며 텍셀이 과도하게 늘어나는 **폴 아티팩트**가 발생한다. UV 심을 극점 근방에만 배치하거나 큐브맵으로 해결한다.

## 텍스처 맵의 종류 — PBR 파이프라인의 재료들

현대 PBR 파이프라인에서 단일 텍스처가 아닌 복수의 맵이 협력하여 물리적 표면 속성을 재현한다.

| 맵 | 저장 정보 | BRDF에서의 역할 |
|---|---|---|
| **Albedo / Base Color** | 순수 표면색 (sRGB) | F0 기저 반사율 계산 입력 |
| **Normal Map** | 접선 공간 법선 (Linear) | 법선 N 교란 → 조명 방향 변경 |
| **Roughness** | 미세면 거칠기 (Linear) | GGX 분포함수 D의 α값 |
| **Metallic** | 금속성 (Linear) | F0 선택: 비금속 0.04 / 금속 Albedo |
| **AO** | 주변광 차폐 (Linear) | 간접광 감쇠 |
| **Emission** | 자체 발광 (sRGB) | 조명 계산과 독립적으로 더해짐 |
| **Displacement** | 표면 높낮이 (Linear) | Tessellation과 연동, 실루엣 변형 |
| **SSS** | 내부 산란 색·거리 | 피부·밀랍·잎사귀의 반투과 표현 |

> [!warning] Linear vs sRGB — 혼동하면 물리적으로 틀린 렌더링이 나온다
> - **sRGB 감마 보정 적용**: Albedo, Emission
> - **반드시 Linear 공간**: Normal, Roughness, Metallic, AO, Displacement
> - ORM 채널 패킹 시 Linear 강제 설정 필수 — 이를 놓치면 GGX의 α값이 왜곡됨

**ORM 채널 패킹**: Occlusion(R), Roughness(G), Metallic(B)을 하나의 텍스처에 묶어 텍스처 슬롯(DX11 최대 16개)과 VRAM을 절약한다.

## 멀티텍스처링과 범프 매핑 계열

**멀티텍스처링** 은 하나의 폴리곤에 두 개 이상의 텍스처를 동시에 적용하는 기법이다. 현대 PBR 셰이더는 10개 이상의 텍스처 레이어를 결합한다.

주요 활용:
- **라이트맵**: GI 결과를 UV1에 사전 계산하여 런타임 조명 연산 절감
- **디테일 텍스처**: 주 텍스처의 반복 패턴을 보완하는 고주파 디테일 중첩
- **더트맵**: 풍화·오염·마모를 예술적으로 표현

표면 요철 표현 기법 비교:

| 기법 | 원리 | 기하 변형 | 자기 차폐 |
|---|---|---|---|
| Bump Mapping | 높이 맵으로 법선 추론 | 없음 | 없음 |
| Normal Mapping | 접선 공간 법선 직접 저장 | 없음 | 없음 |
| Parallax Mapping | 높이 맵으로 UV 오프셋 | 없음 | 없음 |
| POM | 레이마칭으로 교차점 탐색 | 없음 | 있음 |
| Displacement | 테셀레이션 + 메시 변형 | 있음 | 있음 |

## 텍스처 필터링과 밉맵

**밉맵(Mipmap)** 은 원본 텍스처를 절반씩 축소한 이미지 체인이다. 전체 저장 용량은 원본의 약 1.33배다.

밉맵 LOD($\lambda$) 자동 계산 수식:

$$\lambda = \log_2\!\left(\max\!\left(\sqrt{\left(\frac{\partial u}{\partial x}\right)^{\!2}+\left(\frac{\partial v}{\partial x}\right)^{\!2}},\ \sqrt{\left(\frac{\partial u}{\partial y}\right)^{\!2}+\left(\frac{\partial v}{\partial y}\right)^{\!2}}\right)\right)$$

GPU는 **2×2 픽셀 쿼드**를 동시에 처리하며 `ddx()`, `ddy()` 편미분을 픽셀 간 차분으로 근사한다. 삼각형이 1픽셀만 덮어도 최소 2×2 단위로 셰이더가 실행되며, 나머지 3픽셀을 **Helper Invocations**라 한다.

필터링 방식 비교:

| 방식 | 샘플 수 | 특징 |
|---|---|---|
| Nearest Neighbor | 1 | 픽셀화 아티팩트 발생 |
| Bilinear | 4 | 부드러움, 밉 경계 선명 |
| Trilinear | 8 | 밉맵 전환 매끄러움 |
| Anisotropic 16x | 최대 8 | 경사면에서 선명 유지 |

이중선형 보간 수식 ($s, t \in [0,1]$):

$$C = (1-s)(1-t)\,C_{00} + s(1-t)\,C_{10} + (1-s)t\,C_{01} + st\,C_{11}$$

## 텍스처 베이킹 — 복잡함을 텍스처에 굽는다

**베이킹(Baking)** 은 전역 조명·AO·고해상도 기하 정보 등 연산 비용이 높은 결과를 텍스처에 사전 계산·저장하는 최적화 기법이다.

```
[High-Poly Sculpt (수백만 폴리곤)]
        ↓ 레이캐스팅
[Normal Map — UV 공간에 법선 저장]
        ↓ 적용
[Low-Poly Mesh (수천 폴리곤)]   ←   시각적으로 구별 불가
```

베이킹 품질을 결정하는 핵심 요소:

| 요소 | 설명 |
|---|---|
| **케이지(Cage)** | 투사 범위. 너무 작으면 법선 누락, 너무 크면 반대면 포함 오류 |
| **레이캐스팅 방향** | 평균 법선 vs 케이지 기반 |
| **패딩(Gutter)** | UV 심 경계에 2~4픽셀 복제 → 밉맵 필터링 경계 아티팩트 방지 |

> [!info] 자동화가 필수인 이유
> 대형 프로젝트에서 수천 개 에셋에 베이킹을 수동으로 수행하는 것은 불가능하다. Unreal Engine 베이킹 파이프라인, Substance Bakers, xNormal 등 툴의 케이지·패딩 설정을 표준화하면 품질 일관성이 보장된다.

## 원근 보정 수학 — 왜 u/z 를 보간하는가

원근 투영 후 화면 공간에서 $u$, $v$를 직접 선형 보간하면 텍스처가 비틀린다. 이유는 $u$, $v$ 자체가 화면 공간에서 **선형이 아니기** 때문이다.

화면 공간에서 선형인 양:

$$\frac{1}{z},\quad \frac{u}{z},\quad \frac{v}{z}$$

**아핀 보간** (깊이 무시, 오류):

$$u_\alpha = (1-\alpha)\,u_0 + \alpha\,u_1$$

**원근 보정 보간** (GPU 래스터라이저 실제 동작):

$$u_\alpha = \frac{(1-\alpha)\dfrac{u_0}{z_0} + \alpha\dfrac{u_1}{z_1}}{(1-\alpha)\dfrac{1}{z_0} + \alpha\dfrac{1}{z_1}}$$

무게중심 좌표 $(\lambda_0, \lambda_1, \lambda_2)$를 삼각형에 적용하면:

$$u_{\text{correct}} = \frac{\lambda_0 \dfrac{u_0}{w_0} + \lambda_1 \dfrac{u_1}{w_1} + \lambda_2 \dfrac{u_2}{w_2}}{\lambda_0 \dfrac{1}{w_0} + \lambda_1 \dfrac{1}{w_1} + \lambda_2 \dfrac{1}{w_2}}$$

> [!warning] 역사적 실패 사례 — PlayStation의 아핀 왜곡
> 원본 PlayStation은 하드웨어 원근 보정 없이 아핀 매핑만 지원했다. 카메라가 기울어지면 텍스처가 눈에 띄게 비틀렸다. Sony는 **삼각형 세분화** (작은 폴리곤일수록 아핀 오차 감소)로 보완했다. Quake는 16픽셀마다 한 번만 정밀 계산하고 사이는 선형 보간하는 하이브리드로 성능을 확보했다.

## GPU 하드웨어 구현 — TMU와 역·포워드 매핑

### TMU (Texture Mapping Unit)

GPU에 탑재된 전용 고정 기능 하드웨어로, 텍스처 샘플링의 전 과정을 처리한다.

- 텍스처 좌표 어드레싱 (Repeat / Clamp / Mirror)
- 이중선형·삼선형·이방성 필터링
- BC/ASTC 압축 포맷 실시간 하드웨어 디코딩
- 밉맵 LOD 선택

NVIDIA RTX 4090 기준 512개의 TMU를 병렬 탑재한다.

### 역 텍스처 매핑 vs 포워드 텍스처 매핑

```
역 매핑 (현대 표준):
  화면 픽셀 → 래스터라이저 → UV → TMU → 텍셀

포워드 매핑 (레거시):
  텍셀 → 변환 → 화면 픽셀 → 프레임 버퍼
```

| 방식 | 장점 | 단점 | 사용 예 |
|---|---|---|---|
| **역 매핑** | 픽셀 수에 비례한 작업량 | 비선형 메모리 접근 (캐시 미스) | 현대 GPU 전체 |
| **포워드 매핑** | 텍스처 순차 접근 (캐시 효율) | 도형 크기 무관하게 텍셀 전체 순회 | 3DO, 세가 새턴, NV1 |

**Morton Order 스위즐링**: 역 매핑의 캐시 미스를 줄이기 위해 텍스처를 Z-order Curve로 저장하여 공간적으로 인접한 텍셀들이 메모리상에서도 가깝도록 배치한다.

## 텍스처 압축 — BC 포맷과 ASTC

GPU 텍스처 압축은 일반 이미지 압축과 달리 **4×4 텍셀 블록 단위로 독립 압축**하여 임의 접근이 가능하다.

| 포맷 | 압축비 | 주 용도 |
|---|---|---|
| **BC1 (DXT1)** | 6:1 | Albedo (알파 없음) |
| **BC3 (DXT5)** | 4:1 | 부드러운 알파 |
| **BC4** | 8:1 | 단일 채널 (Height, AO) |
| **BC5** | 4:1 | Normal Map XY 채널 |
| **BC6H** | 6:1 | HDR 텍스처 |
| **BC7** | 4:1 | 고품질 RGB/RGBA |
| **ASTC** | 가변 | 모바일/Apple (블록 2×2~12×12) |

> [!tip] Normal Map 압축 핵심
> BC5는 XY 채널만 저장한다. Z 채널은 셰이더에서 법선이 단위 벡터임을 이용해 재계산한다.
> $$n_z = \sqrt{\max\!\left(0,\; 1 - n_x^2 - n_y^2\right)}$$
> BC7보다 대역폭을 아끼면서 Normal Map에 최적화된 선택이다.

## [AI 생성 참고용] 셰이더 디자이너·그래픽스 엔지니어 연결점

> [!example] 관점 ① — UV 왜곡을 셰이더로 극복하기
> 셰이더 디자이너는 UV 언래핑의 한계를 코드로 보완하는 능력이 필수다. 지형처럼 UV가 없거나 타일링이 필요한 표면에는 월드 공간 XYZ 좌표를 활용하는 **트리플레이너 매핑**을 적용한다. 반복 패턴이 눈에 띄는 타일링 아티팩트는 두 UV 오프셋 샘플을 무작위 블렌딩하는 **Stochastic Tiling**으로 해결한다. POM은 레이마칭 루프를 셰이더 내에서 실행하며, 반복 횟수와 품질의 트레이드오프를 제어한다. 이 능력들은 UE5 머티리얼 그래프에서 "커스텀 노드"로 구현되며, 포트폴리오에서 기술적 깊이를 증명하는 직접적 증거가 된다.

> [!example] 관점 ② — 텍스처 메모리 대역폭은 곧 프레임 시간
> 그래픽스 엔지니어 관점에서 텍스처 샘플링은 GPU 대역폭 병목의 주요 원인이다. 같은 해상도라도 BC1(8B/4×4블록)과 BC7(16B/4×4블록)의 대역폭 차이는 2배이며, 이것이 프레임 시간에 직접 영향을 준다. 텍스처 아틀라스는 여러 텍스처를 하나로 합쳐 Draw Call 수를 줄이고 캐시 지역성을 높인다. VRAM 예산이 정해진 콘솔 타이틀에서 텍스처 메모리 관리는 렌더러 엔지니어의 핵심 책임이다. 이 사고방식은 그래픽스 엔지니어 면접에서 "최적화 경험"을 설명할 때 핵심 프레임이 된다.

> [!example] 관점 ③ — PBR 채널 설계와 Linear 공간 관리
> Albedo만 sRGB 감마 보정을 적용하고, Roughness·Metallic·AO는 반드시 Linear 공간으로 저장해야 한다. 이 원칙이 무너지면 GGX 분포함수 D와 Smith 가려짐 함수 G의 α값이 왜곡되어 물리적으로 부정확한 광택이 나온다. ORM 채널 패킹 시 텍스처 임포트 세팅에서 sRGB 옵션을 반드시 해제해야 하며, 이를 자동화하는 에셋 파이프라인 스크립트가 대형 프로젝트에서 필수다.

> [!example] 관점 ④ — 렌더러 수준에서의 원근 보정 이해
> 소프트웨어 래스터라이저를 구현하거나 임베디드 렌더러를 최적화할 때 원근 보정 수식을 직접 구현해야 한다. u/z, v/z, 1/z를 정점에서 계산해 선형 보간한 뒤, 픽셀마다 나눗셈으로 복원하는 구조다. Quake 방식처럼 16픽셀마다 한 번만 정밀 계산을 수행하고 사이는 선형 보간으로 채우는 하이브리드 접근은 성능과 품질의 균형을 맞추는 고전적이지만 여전히 유효한 전략이다. 커스텀 렌더러 구현 경험은 취업 포트폴리오에서 강력한 차별점이 된다.

## Key Insights

> [!success] Insight 1 — UV는 해상도 독립적 추상화 레이어다
> UV 좌표는 [0, 1] 정규화 범위를 사용하므로, 512px 텍스처든 4096px 텍스처든 동일한 UV 좌표가 동일한 상대적 위치를 가리킨다. 이 추상화 덕분에 텍스처 해상도를 변경해도 UV를 재언래핑할 필요가 없다. 셰이더에서 UV 스크롤·회전·왜곡 연산도 이 [0, 1] 공간에서 이루어진다.

> [!success] Insight 2 — 원근 보정의 핵심은 1/z의 선형성이다
> 화면 공간에서 선형인 것은 $u$, $v$가 아니라 $u/z$, $v/z$, $1/z$다. 이 성질을 이해하면 아핀 매핑 왜곡의 원인, GPU 래스터라이저의 작동 방식, 소프트웨어 래스터라이저 구현법이 한 번에 이해된다. 렌더링 파이프라인의 가장 기초적이면서 가장 중요한 수학 원리 중 하나다.

> [!success] Insight 3 — 베이킹은 "연산을 시간으로 사는" 트레이드오프다
> 노멀 맵 베이킹은 수백만 폴리곤의 고해상도 조각 메시를 수천 폴리곤 로우폴리에서 재현하게 한다. 사전 계산 비용을 지불하고 런타임 비용을 0으로 만드는 명확한 트레이드오프다. 라이트맵·AO·Displacement 베이킹도 같은 원리다. 이 트레이드오프를 인식하는 것이 렌더링 최적화 사고의 출발점이다.

> [!success] Insight 4 — GPU TMU는 캐시 지역성을 위해 Morton Order를 사용한다
> 역 텍스처 매핑은 텍스처 메모리에 비선형으로 접근하여 캐시 미스를 유발한다. GPU는 텍스처를 Morton Order(Z-order Curve)로 저장하여 공간적으로 인접한 텍셀들이 메모리상에서도 가깝도록 배치한다. 이것이 고성능 텍스처 샘플링의 물리적 기반이며, GPU 하드웨어 설계 이해의 핵심이다.
