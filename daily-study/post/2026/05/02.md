# 📅 TIL 2026-05-02: Texture · Material · Shader의 계층

> **출처**: YouTube — "Textures vs Materials vs Shaders: What you Need to Know!" ([영상 링크](https://www.youtube.com/watch?v=CBX9Esfnx5I))
> 3D 아티스트가 혼용하는 세 용어의 정확한 정의 / Shading의 역사(Flat→Gouraud→Phong) / 소프트웨어별 Shader 언어 대응 / Procedural Shader / PBR 기초
> "Shader는 GPU에서 실행되는 프로그램이고, Material은 그 프로그램의 파라미터 인스턴스이며, Texture는 그 파라미터에 연결되는 이미지 데이터다"

---

오늘 가장 명확히 깨달은 것은 **세 개념의 포함 관계**다: `Shader ⊃ Material ⊃ Texture`.
많은 3D 아티스트들이 Material과 Shader를 혼용하는 이유가 소프트웨어마다 UI 구조가 달라서라는 점도 이해했다(UE5에서 "Material"은 사실상 HLSL 셰이더 소스코드다).
Shading의 역사를 Flat → Gouraud → Phong → Blinn-Phong → PBR 순서로 짚으면서, 각 기법이 단순한 역사적 사실이 아니라
**"버텍스에서 계산할 것인가, 픽셀에서 계산할 것인가"라는 성능-품질 트레이드오프의 진화**임을 체감했다.
특히 Phong의 핵심이 "색상이 아닌 Normal 벡터를 보간"한다는 한 줄 개념이 모든 현대 Fragment Shader의 원형임을 연결지었다.

## 1️⃣ Today I Learned

### ① 세 개념의 포함 관계와 혼용 문제

```
SHADER  (GPU 프로그램 — "어떻게 그릴지" 알고리즘)
  └─ MATERIAL  (Shader + Parameter 인스턴스 — "어떤 값을 넣을지")
       └─ TEXTURE  (이미지 데이터 — "색상/법선/거칠기 공급")
```

**혼용 문제의 원인**: 소프트웨어마다 UI 레이어가 다르다.
- Blender: "Material" 탭 안에서 Shader 설정 → 혼용
- Unreal Engine 5: "Material Editor"가 사실상 HLSL 셰이더 소스 작성 공간 → 혼용
- Substance Painter: Texture를 "Material"이라 부름 → 혼용

같은 Shader 코드로, Material 파라미터만 바꾸면 나무/금속/플라스틱이 완전히 다르게 표현된다.

---

### ② Texture — 이미지 데이터의 한계

텍스처는 **3D 모델 표면에 입히는 이미지**다. 폴리곤 수를 늘리지 않고 시각적 디테일을 추가하는 방법.

**텍스처만으로는 불충분한 이유**: 색상 데이터만 제공한다. 빛이 어떻게 반응할지, 광택이 있는지, 반사가 얼마나 강한지 — 이것은 Material과 Shader가 담당한다.

**제작 방법 4가지**: Photoshop(2D 소프트웨어) / 사진 촬영(Photo-based) / 손 그림 후 스캔 / **Photogrammetry**(최신, 실사 3D 자산)

**핵심 포맷:**

| 포맷 | 특징 |
|------|------|
| `.png` | 무손실, 알파 채널, 일반 텍스처 |
| `.exr` | HDR 부동소수점, Normal Map·라이트맵 |
| `.dds` | GPU 친화적 압축(DXT/BC7), 게임 런타임 |

---

### ③ Material — 빛 반응의 레시피: PBR Map 구성

Material은 **"빛이 닿았을 때 이 표면이 어떻게 보여야 하는가"를 정의하는 텍스처 + 파라미터 집합**이다.

**PBR Material Map 전체:**

| Map | 역할 | 값 범위 |
|-----|------|---------|
| **Albedo / Base Color** | 순수 고유 색상 (조명 영향 없음) | RGB 0~1 |
| **Normal Map** | 법선 벡터를 RGB로 인코딩 → 요철 표현 | RGB → XYZ |
| **Roughness** | 표면 거칠기 | 0 매끄러움 → 1 거침 |
| **Metallic** | 금속성 여부 | 0 비금속 → 1 금속 |
| **AO** | 구석 자체 그림자 | 1 밝음 → 0 어두움 |
| **Emissive** | 자체 발광 | 발광 색상 + 강도 |
| **Displacement** | 실제 geometry 변형 | 높이값 |

---

### ④ Normal Map의 작동 원리

Normal Map은 많은 초보자가 오해하는 개념이다.

```
실제 로우 폴리곤 메쉬: 평평한 삼각형들
Normal Map 적용 후: 울퉁불퉁하게 보임 (geometry 변화 없음!)

RGB 픽셀 → XYZ 법선 방향 벡터 인코딩:
  중립(128,128,255) → N(0, 0, 1) 수직 → 파란색 계열 (Tangent Space)
  (255,128,128)     → N(1, 0, 0) 오른쪽 기운 면
```

**Tangent Space vs Object Space:**
- **Tangent Space** (파란색 계열): UV 공간 기준. 모델 이동/회전해도 올바르게 작동. 실무 표준.
- **Object Space** (형형색색): 로컬 좌표 기준. 타일링 불가. 특수 용도.

> [!warning] OpenGL vs DirectX Normal Map 방향
> Substance Painter에서 Export 시 반드시 대상 엔진의 Normal Map 규칙(Y축 방향)을 확인할 것. 잘못 설정하면 조명이 반대로 들어온 것처럼 보인다.

### ⑤ Shader — GPU에서 실행되는 프로그램

셰이더는 GPU에서 실행되는 **소규모 프로그램**이다. 각 픽셀/버텍스 단위로 조명, 그림자, 반사를 실시간 계산한다.

**현대 GPU 셰이더 종류:**

| 종류 | 단위 | 역할 |
|------|------|------|
| **Vertex Shader** | 버텍스 | 3D → 클립 공간 좌표 변환 |
| **Fragment (Pixel) Shader** | 픽셀 | 최종 색상 계산, 텍스처 샘플링 |
| **Geometry Shader** | 프리미티브 | 동적 지오메트리 생성 |
| **Tessellation Shader** | 패치 | LOD, Displacement |
| **Compute Shader** | 스레드 그룹 | GPGPU, 물리 시뮬레이션 |

```
Shader와 Material의 관계:
┌─────────────────────────────────────────┐
│  Fragment Shader 코드 (한 번만 컴파일)  │
│  • albedo = texture(albedo_map, uv)     │
│  • roughness = texture(rough_map, uv)   │
│  • ... PBR 조명 계산 ...                │
└─────────────────────────────────────────┘
         ↑ 같은 코드
Material A (Wood):  albedo_map=wood.png, roughness=0.8
Material B (Metal): albedo_map=steel.png, roughness=0.1, metallic=1.0
→ 완전히 다른 표면 표현
```

Blender의 기본 큐브조차 화면에 보이려면 Shading이 필요하다 — 이것이 "Shader 없이는 아무것도 렌더링되지 않는다"는 뜻이다.

### ⑥ Shading의 역사: Flat → Gouraud → Phong

**이 역사는 면접에서 자주 출제되며, 각 기법의 트레이드오프를 이해하는 틀이다.**

**Flat Shading (초창기):**
- 폴리곤 하나에 단 하나의 색상. 각진(faceted) 외형.
- 계산: 폴리곤의 Face Normal 하나로 조명 계산 → 폴리곤 전체에 균일 적용.

**Gouraud Shading (1971, Henri Gouraud):**
- 각 **버텍스**에서 조명 계산 → 버텍스 사이를 **색상값으로 보간**.
- 개선: 부드러운 그라데이션. 한계: Specular highlight가 버텍스 사이에서 소실 (Mach band).
- 계산 위치: **Vertex Shader**.

**Phong Shading (1973, Bui Tuong Phong):**
- 버텍스의 **Normal 벡터를 보간** → 각 픽셀마다 독립적으로 조명 계산.
- 핵심 차이: 색상이 아닌 **Normal을 보간** → 픽셀 단위 조명.
- 계산 위치: **Fragment Shader**. 더 정확한 Specular, 더 부드러운 표면.

```
수식 비교:

Gouraud: I_P = α×I_v0 + β×I_v1 + γ×I_v2  (보간 대상: 조명 결과)
Phong:   N_P = normalize(α×N_v0 + β×N_v1 + γ×N_v2)  (보간 대상: Normal)
         I_P = k_a×I_a + k_d×(N_P·L)×I_d + k_s×(R·V)^α×I_s  (픽셀마다 계산)
```

---

### ⑦ Blinn-Phong 개선과 현대 Programmable Shading

**Blinn-Phong (Jim Blinn):**
Phong의 반사 벡터 R 계산을 Half Vector H로 대체 → 약 30% 빠름, 더 자연스러운 하이라이트.

```
H = normalize(L + V)    (광원 방향 + 시선 방향의 중간 벡터)
I_spec = k_s × (N · H)^α × I_s
→ R 계산 불필요, 현재도 실시간 렌더링에서 광범위하게 사용
```

**현대 Programmable Shading:** HLSL/GLSL/OSL/VEX 등 셰이딩 언어로 위 이론을 직접 구현. PBR, Ray Tracing, Neural Rendering 모두 이 토대 위에 구축되어 있다.

### ⑨ Procedural Shader — 수학으로 만드는 텍스처

영상에서 보여준 **GLSL 대리석(Marble) 셰이더**의 핵심:

```glsl
// Domain Warping: 좌표를 Noise로 왜곡 → 다시 Noise 계산
float d = perlin(p.x, p.y) * scale;     // 1차 왜곡
float u = p.x + d;  float v = p.y + d;
d = perlin(u, v) * scale;               // 2차 왜곡
float noise = perlin(p.x + d, p.y + d); // 최종 패턴
color = vec3(... noise ...);            // 색상 매핑
```

| 항목 | Image Texture | Procedural Texture |
|------|--------------|-------------------|
| 메모리 | 파일 크기에 비례 | 거의 0 (코드만) |
| 해상도 | 고정 | 이론상 무제한 |
| 타일링 | 반복 패턴 보임 | 자연스러운 반복 |
| 편집 | 이미지 편집 필요 | 파라미터만 조정 |

**Perlin Noise**: Ken Perlin (1983) 개발. 대리석, 연기, 구름, 지형에 활용되는 자연스러운 랜덤 패턴 생성 알고리즘.

### ⑩ PBR의 핵심 원칙과 Cook-Torrance BRDF

**PBR의 두 가지 핵심 원칙:**
1. **에너지 보존**: 반사된 빛의 총량은 입사한 빛을 초과 불가 (Diffuse + Specular ≤ Incoming)
2. **Microfacet Theory**: 실제 표면은 수많은 미세면(microfacet)으로 이루어져 있고, Roughness는 이 미세면들의 방향 분포를 나타냄

**Cook-Torrance BRDF (물리 기반 반사 모델):**

```
f_cook-torrance = [D(h) × F(v,h) × G(l,v,h)] / [4(n·l)(n·v)]

D (GGX Distribution):     미세면 방향 분포 — Roughness에 따라 결정
F (Fresnel-Schlick):       입사 각도에 따른 반사율 변화 (grazing angle → 강한 반사)
G (Smith Masking-Shadow):  미세면 간 자가 차폐 — 거칠수록 자가 차폐 증가
```

**Metallic vs Dielectric:**

| 특성 | 금속 (Metallic 1) | 비금속 (Metallic 0) |
|------|-----------------|---------------------|
| Diffuse | 없음 | 있음 |
| Specular F0 | 높음 (RGB 색상) | 낮음 (~4%, 단색) |
| 예시 | 금, 구리, 알루미늄 | 나무, 플라스틱, 피부 |

## 2️⃣ Key Insights — [AI 생성 참고용] 셰이더 디자이너, 그래픽스 엔지니어 관점에서 연결되는 것들

> [!success] 인사이트 1: `glBufferData`의 `const void*` — Texture가 Shader에 도달하는 경로
> 어제 C++에서 배운 `void*` 의 타입 소거가 오늘 직접 연결됐다. `glBufferData(GL_ARRAY_BUFFER, size, const void* data, usage)` 의 세 번째 인자가 바로 `void*` 다. float 버텍스 배열이든, Vertex 구조체 배열이든 어떤 타입의 CPU 데이터도 이 함수를 통해 GPU VRAM으로 업로드할 수 있는 이유다. Texture 데이터도 마찬가지로 `glTexImage2D(..., const void* pixels)` 를 통해 전달된다.

> [!success] 인사이트 2: Gouraud vs Phong — Vertex Shader와 Fragment Shader의 역할 분리
> 오늘 Shading 역사를 통해 Vertex Shader와 Fragment Shader가 왜 나뉘어 있는지의 뿌리를 이해했다. Gouraud는 Vertex Shader에서 완결, Phong은 Fragment Shader에서 완결. 현대 렌더링 파이프라인의 두 주요 셰이더 단계가 실제로 1970년대 두 수학자의 통찰에 대응한다. 어떤 계산을 Vertex에서, 어떤 계산을 Fragment에서 해야 할지 판단하는 능력이 셰이더 최적화의 핵심이다.

> [!success] 인사이트 3: Fresnel 효과 — 일상에서 항상 보이는 물리 현상
> Cook-Torrance의 F(Fresnel) 항이 설명하는 것은 일상적으로 관찰할 수 있다. 수면을 정면으로 바라보면 투명하게 보이지만, 수평(grazing angle)으로 바라보면 거울처럼 반사된다. 플라스틱 표면도 마찬가지다. 이것이 Fresnel-Schlick 근사식 `F0 + (1-F0)(1-v·h)^5` 가 캡처하는 현상이다. 셰이더에서 이 항을 빠뜨리면 PBR 머티리얼이 "가짜처럼" 보이는 직접적인 원인이 된다.

> [!success] 인사이트 4: Normal Map의 Tangent Space — C++ 행렬 연산과의 연결
> Normal Map의 Tangent Space 변환은 사실 행렬 곱이다. 버텍스마다 Tangent, Bitangent, Normal 벡터로 구성된 **TBN 행렬**을 만들어 Tangent Space의 법선을 World Space로 변환한다. Fragment Shader에서 `mat3 TBN = mat3(T, B, N); vec3 normal = normalize(TBN * normalFromMap)` 형태로 사용된다. 선형대수의 행렬 변환이 셰이더 코드에 직접 적용되는 순간이다.

> [!success] 인사이트 5: Procedural Shader와 GPU SIMT 아키텍처
> Perlin Noise 기반 Procedural Shader는 모든 픽셀이 **동일한 수학 공식을 서로 다른 UV 좌표에 대해 동시에 실행**한다. 이것이 GPU의 SIMT(Single Instruction Multiple Thread) 아키텍처와 완벽히 맞아떨어지는 이유다. 텍스처 없이 수학만으로 패턴을 생성하면 텍스처 캐시 미스가 없어 고해상도 렌더링에서 오히려 Image Texture보다 빠를 수 있다.
