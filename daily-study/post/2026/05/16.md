## 그래픽스 파이프라인 — 왜 파이프라인인가

**컴퓨터 그래픽스**는 **3차원 데이터(3D Model)를 렌더링하여 2차원 스크린(2D Screen)에서 이미지로 만드는 과정**이다. 이 과정을 효율적으로 처리하기 위한 설계가 바로 **그래픽스 파이프라인(Graphics Pipeline)** 이다.

3D → 2D 변환은 세 단계로 구성된다.

| 단계 | 역할 | 실행 위치 |
|---|---|---|
| **Data Preparation** | GPU가 처리하기 좋은 자료구조로 변환 (VBO, EBO) | CPU |
| **Compute Data** | 정점 좌표 변환 + 조명 계산 | GPU |
| **Generate Images** | 삼각형 → 픽셀 래스터라이제이션 | GPU |

> [!tip] 왜 삼각형 메시가 표준인가?
> 삼각형은 항상 **평면**이 보장되고 3개의 점만으로 면이 결정되므로 **앰비규티가 없다**. GPU 하드웨어가 삼각형 처리에 특화되어 있어, 다른 표현 방식(Voxel, Point Cloud 등)보다 계산이 월등히 빠르다.

## Sequential vs Parallel — 두 방식과 절충안

### Sequential Pipeline (순차 처리)의 문제

```
while (rendering):
    1단계: Preparation
    2단계: Computing
    3단계: Image Generation
    (다시 1단계부터 반복)
```

이미지를 생성하는 동안 Preparation·Computing 단계는 **아무것도 하지 않는다**. 전체 하드웨어 자원의 대부분이 유휴 상태가 되어 효율이 극히 낮다.

### Parallel Pipeline (병렬 처리)의 문제

여러 객체를 동시에 처리하면 속도는 빠르지만 두 가지 **Race Condition**이 발생한다.

1. **모델 읽기 충돌**: 여러 객체가 동일한 메모리 영역에 동시에 접근
2. **프레임 버퍼 쓰기 충돌**: 여러 객체가 **같은 픽셀 위치**를 점유 → 깊이 판단 또는 알파 블렌딩 순서 필요

> [!warning] 순서 의존성의 함정
> 알파 블렌딩은 **뒤에서 앞으로(Back-to-Front)** 순서로 처리해야 올바른 합성이 된다. 이것이 GPU의 병렬 처리와 근본적으로 충돌한다. 이 문제를 해결하는 것이 **OIT(Order-Independent Transparency)** 연구의 출발점이다.

### 파이프라인 — 절충안

파이프라인은 Sequential과 Parallel의 중간이다. **각 단계가 서로 다른 객체를 동시에 처리**한다.

```
[Application Stage] → [Geometry Stage] → [Rasterization Stage]
     (나무 준비)           (바위 계산)          (사람 픽셀 생성)
        ← 다음 프레임에서는 각 단계가 다른 객체를 처리 →
```

## 세 단계 개요 — Application · Geometry · Rasterization

| 단계 | 주 실행 위치 | 역할 | 출력 |
|---|---|---|---|
| **Application Stage** | CPU (일부 GPU) | 씬 구성, 애니메이션, 물리 | 정점(Vertex) 위치 |
| **Geometry Stage** | GPU | 좌표 변환, 조명, 프리미티브 조립 | Fragment 스트림 |
| **Rasterization Stage** | GPU | 픽셀 변환, 색상 계산, 프레임 버퍼 기록 | 최종 픽셀 |

---

## Application Stage — 씬을 만든다

Application Stage는 **CPU에서 수행**되며, 씬을 구성하고 GPU에 전달할 정점 데이터를 준비하는 전 과정을 담당한다. 최근에는 Compute Shader를 통해 **Culling, 물리 시뮬레이션이 GPU에서도 수행**된다(GPU-Driven Rendering).

### Scene Setup — 씬 구성 요소

| 구성 요소 | 내용 |
|---|---|
| **객체 생성·조작** | 3D 모델 로드, 위치·회전·크기(Transform) 설정 |
| **구조 모델링** | 캐릭터 관절(Bone/Joint) 계층 구조 정의 |
| **애니메이션 처리** | Keyframe 보간, 물리 기반 애니메이션 |
| **물리 계산** | 충돌(Collision), 중력, 변형(Deformation) |
| **카메라 파라미터** | FoV, Aspect Ratio, Near/Far Plane |
| **조명 파라미터** | 광원 색상·위치·방향, 재질(Material) |

### Scene Graph — 계층 자료구조

```
World
 ├── Building
 │    ├── Room A
 │    │    ├── Chair
 │    │    └── Table
 │    └── Room B
 └── Camera
```

- **계층 구조**: 부모 노드의 Transform이 자식 노드에 누적 적용됨
- **Culling 지원**: Scene Graph 탐색으로 카메라 시야 밖 노드를 조기 제거

### Shape Representation — 형상 표현 방식 비교

| 표현 방식 | 장점 | 단점 | 주 용도 |
|---|---|---|---|
| **Mesh (Triangle)** | GPU 가속, 계산 단순 | 곡면 근사 필요 | 실시간 렌더링 표준 |
| **Voxel** | 볼륨 표현 정확 | 메모리 방대 | CT, MRI, 마인크래프트 |
| **Point Cloud** | 스캐닝 직접 활용 | 표면 재구성 필요 | LiDAR, 3D 스캐닝 |
| **Octree** | 공간 분할 효율적 | 실시간 렌더링 직접 불가 | 충돌 감지, 공간 인덱싱 |
| **NeRF (Projections)** | 기존 카메라 활용 | 실시간 처리 어려움 | 포토그래메트리, 신경 렌더링 |

### Animation Processing — 애니메이션 기술

| 유형 | 기술 | 설명 |
|---|---|---|
| **공의 움직임** | Keyframe 보간 | 위치·회전의 시간 변화 |
| **천(Cloth)** | Cloth Simulation (질량-스프링) | 각 버텍스를 질량점으로 연결 |
| **사람의 움직임** | Skeletal Animation (Skinning) | 뼈대에 메시 스키닝 |
| **머리카락** | Strand-based Simulation | 가이드 스트랜드 + 보간 |
| **표정** | Blend Shape / Facial Rig | 표정 간 버텍스 보간 |

**Skinning 원리**:

```
vertex_world = Σ(weight_i × bone_matrix_i × vertex_local)
```

각 정점은 여러 뼈의 변환 행렬을 가중합으로 받아 최종 위치가 결정된다.

### Physics Calculation — 물리 알고리즘

| 물리 현상 | 알고리즘 | 예시 |
|---|---|---|
| **충돌 감지** | BVH, GJK, SAT | 두 물체 충돌 여부 판단 |
| **충돌 응답** | Impulse-based, Constraint-based | 충돌 후 반발 방향·속도 |
| **중력·운동** | Euler, Verlet, RK4 수치 적분 | 포물선 운동, 낙하 |
| **변형** | FEM(유한요소법), PBD | 진흙·고무 형태 변화 |
| **유체** | SPH, FLIP, Grid-based | 물·연기의 흐름 |

> [!success] Application Stage의 결론
> 모든 과정의 **최종 출력은 정점(Vertex)의 위치**다. 물리, 애니메이션, Scene Setup의 모든 계산이 결국 "각 정점이 월드 공간 어디에 있는가"로 수렴하여 Geometry Stage로 전달된다.

## Geometry Stage — 정점을 변환한다

### 5단계 좌표 변환 체인

$$\mathbf{v}_{clip} = \mathbf{M}_{proj} \cdot \mathbf{M}_{view} \cdot \mathbf{M}_{model} \cdot \mathbf{v}_{local}$$

| 단계 | 변환 | 입력 공간 | 출력 공간 |
|---|---|---|---|
| **Model Transform** | 모델 행렬 | Object Space | World Space |
| **View Transform** | 뷰 행렬 | World Space | Camera Space |
| **Perspective Projection** | 투영 행렬 | Camera Space | Clip Space |
| **Perspective Division** | 각 성분을 w로 나눔 | Clip Space | NDC \[-1, 1\] |
| **Viewport Transform** | 뷰포트 스케일링 | NDC | Screen Coordinates |

- $\mathbf{M}_{model}$: 오브젝트의 위치·회전·크기를 월드 공간으로 변환
- $\mathbf{M}_{view}$: 카메라를 원점에 놓는 변환 (카메라의 역변환)
- $\mathbf{M}_{proj}$: 원근 투영 — 멀수록 작게 보이게 하는 변환

> [!info] NDC와 동차 좌표
> 클립 공간의 w 성분이 1/z에 비례하므로, x/w, y/w, z/w가 NDC 좌표가 된다. 이 w 성분이 UV 원근 보정 보간의 수학적 근거이기도 하다. 3D 벡터 (x, y, z)를 (x, y, z, 1)로 확장하면 평행이동도 행렬 곱으로 표현할 수 있다.

### Primitive Assembly — 정점 → 삼각형

Primitive Assembly는 변환된 정점들을 연결 정보(Connectivity)에 따라 삼각형으로 조립하는 단계다. 2012년 이후 **Geometry Shader**를 통해 하나의 삼각형에서 여러 프리미티브를 생성할 수 있다(잔디, 털, 파티클 등).

### Triangle Types — 프리미티브 타입 비교

| 타입 | 설명 | 정점 재사용 | 사용 예 |
|---|---|---|---|
| **GL_TRIANGLES** | 정점 3개씩 독립 삼각형 | 없음 | 기본, 가장 범용적 |
| **GL_TRIANGLE_STRIP** | 이전 2개 정점 재사용 | 2개씩 | 지형 메시, 대역폭 절약 |
| **GL_TRIANGLE_FAN** | 첫 정점을 모두 공유 | 중심점 | 원형 면, 부채꼴 |
| **GL_TRIANGLES_ADJACENCY** | 인접 정점 정보 포함 | — | Geometry Shader 실루엣 처리 |

> [!info] 메모리 효율
> N개 삼각형을 GL_TRIANGLES로 표현하면 3N개 정점이 필요하지만, GL_TRIANGLE_STRIP은 N+2개 정점으로 표현 가능하다. 대규모 지형 렌더링에서 대역폭 절약이 크다.

## Rasterization Stage — 픽셀로 만든다

### Rasterization — Edge Function

**Coverage Test**: Edge Function으로 픽셀 샘플이 삼각형 내부인지 판단한다.

```
E(x,y) = (x - x0)(y1 - y0) - (y - y0)(x1 - x0)
픽셀이 삼각형 내부라면 세 에지 모두 같은 부호
```

삼각형 내부로 판정된 각 픽셀 위치마다 **Fragment(픽셀 후보)** 가 생성된다.

### Fragment Processing — 전체 셰이더 파이프라인

```
[1] Raw Vertex Data
     ↓
[2] Vertex Shader      → 좌표 변환 (MVP)
     ↓
[3] Geometry & Tessellation  → 프리미티브 생성/세분화
     ↓
[4] Rasterization      → 삼각형 → Fragment
     ↓
[5] Fragment Shader    → 각 Fragment의 색상 계산
     ↓
[6] Per-fragment Operations → 깊이 테스트, 블렌딩 → 최종 픽셀
```

**왜 Pixel이 아닌 Fragment인가?** 반투명 물체가 있을 경우 하나의 픽셀 위치에 여러 Fragment가 존재할 수 있다. 반투명 유리 뒤에 나무가 있으면 두 Fragment가 **합성**되어 최종 색상이 결정된다.

### Per-Fragment Operations — 최종 픽셀 결정

| 연산 | 역할 | 비고 |
|---|---|---|
| **Fragment Shader** | 색상·투명도·깊이 계산 | Texture Sampling 포함 |
| **Z-Test (Depth Test)** | Z-Buffer와 비교, 가려진 Fragment 폐기 | ROP에서 수행 |
| **Alpha Blending** | 투명도에 따라 기존 픽셀과 합성 | 순서 의존성 주의 |
| **Stencil Test** | 마스크 기반 렌더링 제어 | 거울·포털 효과 |
| **Framebuffer 기록** | 최종 색상을 프레임 버퍼에 기록 | ROP → VRAM |

### Alpha Blending 모드

| 블렌딩 모드 | 수식 | 사용 예 |
|---|---|---|
| **Opaque** | 덮어쓰기 | 불투명 오브젝트 |
| **Transparent** | src·α + dst·(1-α) | 유리, 물 |
| **Additive** | src + dst | 파티클, 발광 효과 |
| **Multiply** | src × dst | 그림자, 색조 변환 |

> [!warning] Painter's Algorithm과 OIT
> 알파 블렌딩은 **Back-to-Front 순서**로 렌더링해야 올바른 결과를 얻는다. 이를 Painter's Algorithm이라 한다. 순서가 보장되지 않으면 투명 아티팩트가 발생하며, 이를 근본적으로 해결하는 기법이 **Weighted Blended OIT**, **Depth Peeling** 등이다.

## Graphics Pipeline 종합 — 데이터 흐름과 병목

### 전체 데이터 흐름

```
[Application]
    Vertices (정점들)
         ↓
[Vertex Shader]
    Transformed Vertices
         ↓
[Geometry Shader] (선택적)
    Emitted Primitives
         ↓
[Clipping & Rasterization]
    Fragments
         ↓
[Fragment Shader]
    Shaded Fragments
         ↓
[Framebuffer Operations]
    Pixels → Display
```

### 파이프라인 병목 분석

| 병목 단계 | 증상 | 해결책 |
|---|---|---|
| **CPU / Application** | CPU 100%, GPU 대기 | Draw Call 감소, GPU Instancing, Compute Shader 이전 |
| **Vertex Bound** | Vertex Shader 처리 과다 | LOD 강화, 정점 수 감소 |
| **Geometry Bound** | Geometry Shader 오버헤드 | Geometry Shader 최소화, Mesh Shader 도입 |
| **Fragment Bound** | Fragment Shader 처리 과다 | Shader 복잡도 감소, Early Z 활성화 |
| **Bandwidth Bound** | 텍스처/버퍼 대역폭 포화 | 텍스처 압축(BC1~BC7), LOD 활용 |

> [!info] Early Z — 보이지 않는 곳에 시간을 쓰지 마라
> **Early Z(Early Depth Test)** 는 Fragment Shader 실행 **이전에** 깊이 테스트를 수행하여, 화면에 보이지 않는 Fragment는 아예 셰이더를 실행하지 않는 최적화다. Fragment Shader가 복잡할수록 Early Z의 성능 이득이 커진다.

## Graphics API — 하드웨어와 소프트웨어 사이

Graphics API는 파이프라인 전체를 **함수 호출로 추상화**한 라이브러리다. 개발자는 GPU 하드웨어 차이에 무관하게 동일한 코드를 작성할 수 있다 (디바이스 드라이버가 추상화 담당).

### Graphics API가 하는 5가지 일

1. **데이터 준비**: VBO·EBO·텍스처를 GPU VRAM에 업로드
2. **자료구조 정의**: Vertex Attribute Layout 설정
3. **파라미터 설정**: 셰이더 바인딩, Uniform 변수, 텍스처 슬롯
4. **렌더 커맨드 제출**: Draw Call 발행 → GPU 파이프라인 실행
5. **프레임 버퍼 관리**: Render Target 설정, Swap Chain 제어

### API 계층 구조

**Low-Level** (직접 제어, 고성능):

| API | 플랫폼 | 특징 |
|---|---|---|
| **OpenGL** | Cross-platform | 오래된 표준, 학습 용이, 드라이버 오버헤드 있음 |
| **Vulkan** | Cross-platform | 저수준 직접 제어, CPU 오버헤드 최소, 복잡 |
| **DirectX 12** | Windows/Xbox | Microsoft 표준, Vulkan과 유사한 저수준 |
| **Metal** | Apple | Apple 전용 최적화 |

**High-Level** (편의성, 생산성):

| 엔진/라이브러리 | 기반 | 특징 |
|---|---|---|
| **Unreal Engine 5** | DX12/Vulkan/Metal | 고품질 AAA, Nanite·Lumen |
| **Unity** | DX12/Vulkan/Metal/OpenGL | 범용 게임 엔진 |
| **three.js** | WebGL | 브라우저 기반 3D |

### 파이프라인 단계별 프로그래머블 분류

| 분류 | 의미 | 해당 단계 |
|---|---|---|
| 🟩 **Programmable** | 셰이더 코드로 직접 제어 | Vertex Shader, Geometry Shader, Fragment Shader |
| 🟨 **Configurable** | 파라미터로 동작 설정 | Alpha Blending, Texture Filtering, Anti-aliasing |
| 🟥 **Hardwired** | 고정 하드웨어, 변경 불가 | Rasterization, Early Z, Texture Cache |

> [!info] Vulkan vs OpenGL 핵심 차이
> OpenGL은 **상태 머신(State Machine)** 기반으로 매 Draw Call마다 드라이버가 상태를 검증한다. Vulkan은 개발자가 **Command Buffer**에 명령을 기록한 후 일괄 제출하므로 드라이버 오버헤드가 최소화된다. CPU-GPU 동기화를 위한 Fence·Semaphore·Pipeline Barrier 개념이 Vulkan의 핵심이다.

## [AI 생성 참고용] 셰이더 디자이너·그래픽스 엔지니어 연결점

> [!example] 관점 ① — "어느 셰이더에서 이 연산이 가능한가"를 알아야 한다
> 셰이더 디자이너는 파이프라인 각 단계가 어느 셰이더에 해당하는지를 명확히 알아야 한다. Vertex Shader는 MVP 행렬 변환을 담당하고, Fragment Shader는 각 픽셀의 최종 색상을 결정한다. "왜 이 효과는 Vertex Shader에서 불가능하고 Fragment Shader에서만 가능한가" — 이 질문에 답할 수 있어야 UE5 머티리얼 그래프에서 노드 배치 실수를 피하고, 커스텀 셰이더 코드를 올바른 위치에 작성할 수 있다. Geometry Shader는 잔디·털·파티클의 GPU 기반 생성에 사용되지만, 현대에는 Mesh Shader로 대체되는 추세임을 알아야 한다.

> [!example] 관점 ② — GPU-Driven Rendering과 Application Stage의 진화
> 전통적으로 Application Stage는 CPU가 담당했지만, 현대 렌더링에서는 이 경계가 허물어지고 있다. GPU-Driven Rendering은 Draw Call 결정, Culling, LOD 선택까지 GPU의 Compute Shader에서 수행한다. Unreal Engine 5의 Nanite는 수억 폴리곤을 GPU에서 직접 처리하며, Mesh Shader(Task Shader + Mesh Shader)는 Vertex/Geometry Shader를 통합하여 Meshlet 단위로 Culling·LOD까지 처리한다. 그래픽스 엔지니어는 CPU-GPU 작업 분배와 동기화 설계가 핵심 역량이다.

> [!example] 관점 ③ — 병목 프로파일링이 곧 최적화의 시작점
> 실무에서 렌더링 성능 문제는 항상 파이프라인의 특정 단계에서 발생한다. RenderDoc, PIX, NSight 등 GPU 프로파일러로 어느 단계가 병목인지 파악하는 것이 첫 단계다. Fragment Bound라면 Shader 복잡도나 텍스처 샘플링 횟수를 줄이고, Bandwidth Bound라면 텍스처 압축을 도입하며, CPU Bound라면 Draw Call 감소와 GPU Instancing을 적용한다. 파이프라인 각 단계를 독립적으로 이해해야 병목 원인을 정확히 진단할 수 있다.

> [!example] 관점 ④ — Deferred Rendering과 G-Buffer의 연결
> Forward Rendering에서 Early Z는 성능 최적화다. Deferred Rendering은 이를 한 단계 발전시켜 Geometry Pass에서 G-Buffer(위치·법선·알베도 등)를 채우고, Lighting Pass에서 가시적 픽셀에 대해서만 조명 계산을 수행한다. UE5는 Deferred Rendering을 기본으로 사용하며, Lumen(GI), Shadow Map, SSAO 모두 G-Buffer를 기반으로 동작한다. 셰이더 디자이너는 Translucent 머티리얼이 별도 렌더 패스에서 처리되는 이유를 이 구조에서 이해해야 한다.

## Key Insights

> [!success] Insight 1 — 파이프라인은 "시간과 병렬성의 균형" 문제다
> Sequential Pipeline은 하드웨어를 낭비하고, Parallel Pipeline은 Race Condition을 유발한다. 파이프라인 방식은 "각 단계가 서로 다른 데이터를 동시에 처리"함으로써 두 극단의 절충점을 달성한다. 이 원리는 CPU 명령어 파이프라인, 디스크 I/O 파이프라인 등 컴퓨터 시스템 전반에 적용되는 보편적 설계 패턴이다.

> [!success] Insight 2 — Application Stage의 최종 출력은 항상 정점 위치다
> 애니메이션, 물리, Scene Graph 등 복잡한 연산이 Application Stage에서 수행되지만, 최종적으로 Geometry Stage에 전달되는 것은 **정점의 위치**뿐이다. 이 명확한 인터페이스 덕분에 Application Stage의 내부 구현(물리 알고리즘, 애니메이션 방식)을 바꾸어도 파이프라인 하위 단계에 영향을 주지 않는다.

> [!success] Insight 3 — Fragment와 Pixel의 구분이 알파 블렌딩을 이해하는 열쇠다
> "Fragment는 픽셀 후보"라는 정의는 단순히 용어 구분이 아니다. 반투명 유리 뒤의 나무처럼 하나의 픽셀 위치에 여러 Fragment가 존재할 수 있으며, 이 합성이 Per-Fragment Operation에서 결정된다. 이 개념을 이해하면 OIT의 필요성, Depth Peeling의 원리, 투명 렌더링 패스 분리의 이유가 자연스럽게 연결된다.

> [!success] Insight 4 — Hardwired vs Programmable 분류가 GPU 설계의 핵심 철학이다
> Rasterization은 왜 고정 하드웨어인가? 극도로 단순하고 반복적인 연산이므로 전용 하드웨어로 구현하는 것이 전력·속도 면에서 훨씬 효율적이기 때문이다. 반면 Vertex·Fragment Shader는 다양한 효과를 표현해야 하므로 프로그래머블로 열려 있다. 이 "무엇을 고정하고 무엇을 열어둘 것인가"라는 판단이 GPU 하드웨어 설계의 핵심 철학이며, Mesh Shader가 등장하면서 이 경계가 다시 이동하고 있다.
