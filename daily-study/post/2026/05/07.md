## 1️⃣ 디자인씽킹이란 무엇인가 — 3가지 관점

디자인씽킹은 아직 학술계에서도 하나의 명확한 개념적 정의가 없다. 다양한 진술들을 정리하면 다음 **3가지 관점**으로 분류된다.

| 관점 | 핵심 내용 | 대표 기관 |
|------|---------|---------|
| **혁신 방법론** | 공감→정의→아이디어→프로토타입→테스트의 반복 프로세스. 후천적 훈련으로 습득 가능 | IDEO |
| **마인드셋** | 공감·협업·낙관주의·통합적 사고·실험주의의 태도와 자세 | d.school |
| **사고방식** | 수렴적 사고와 확산적 사고가 반복 교차하는 인지 과정. 귀추논리 기반 | 로저 마틴 |

> [!tip] 핵심 포인트
> 세 관점은 상호 배타적이지 않다. **방법론(How)**·**마인드셋(Who)**·**사고방식(Why)** 으로 디자인씽킹의 다른 층위를 설명한다고 보는 것이 정확하다.

### 스탠포드 d.school의 혁신 프레임

```
Technology (기술적 가능성, Feasibility)
     +
Business (비즈니스 타당성, Viability)
     +
Human Value (사용자 가치, Usability)
     ↓
  Design Innovation
```

세 요소의 교차점에서 진정한 혁신이 발생한다. 실리콘밸리의 크리에이티브 허브인 d.school은 2005년 SAP의 Hasso Plattner가 설립하였으며, 다학제적 접근으로 혁신을 교육한다.

## 2️⃣ 디자인씽킹의 핵심 정의

정정호(2013) 박사 논문에서 제시한 정의:

> **"창의적 혁신을 견인하기 위해 디자이너의 감각적 사고방식을 효율적으로 활용하며, 이성과 감성 간의 균형을 이루게 하는 인지과정이며 지적과정이다."**
>
> **"최적의 문제해결을 위한 규명·검증 등의 논리적 사고와, 추측·가정 등의 직관적 사고가 통합된 사고방식이다."**

이를 구조화하면:

| 좌뇌 (이성) | 구분 | 우뇌 (감성) |
|-----------|------|-----------|
| 논리·분석적 사고 | 사고 유형 | 감각·상상적 사고 |
| 수렴적 사고 (Convergent) | 방향성 | 확산적 사고 (Divergent) |
| 비판적·검증적 | 특성 | 직관적·가정적 |

두 사고가 **반복·교차·균형**을 이루는 것이 디자인씽킹의 본질이다.

> [!warning] 오해 주의
> 디자인씽킹은 "상상만 하는 것"이 아니다. 상상(확산적)과 비판적 검토(수렴적)가 함께 작동할 때 실현 가능한 혁신 아이디어가 나온다. 날아다니는 자동차 아이디어도 비용·기술·법규 검토(수렴적 사고) 없이는 공상에 그친다.

### 공동의 선 (Common Good)

> 창의적 문제해결의 윤리적 기반: **디자인적 사고는 공동의 선을 위해 인간 중심의 디자인을 실행하는 것이다.**

해악적·기만적 목적의 창의적 아이디어는 진정한 창의적 문제해결이 아니다. 서비스디자인은 이 원칙을 실천하는 구체적인 방법론이다.

## 3️⃣ 문제의 유형 — Well-defined vs Wicked Problem

| 유형 | 정의 | 특성 | 사례 |
|------|------|------|------|
| **잘 정의된 문제** (Well-defined) | 결과와 목표가 명확하여 해결안 도출이 용이 | 알고리즘으로 해결 가능 | 오늘 저녁 메뉴, 내일 날씨 |
| **잘 정의되지 않은 문제** (Ill-defined) | 결과·방법이 불명확, 요구조건 구체화에 시간·비용 필요 | 다양한 해석 가능 | 마케팅 전략, 브랜드 아이덴티티 |
| **심술궂은 문제** (Wicked Problem) | 명확한 결과·체계적 수단 없음. 해결안의 진위 판단도 어려움 | 비결정론적 | 기후변화, 빈곤, 사회 불평등 |

> [!success] 핵심
> **Ill-defined Problem과 Wicked Problem이 바로 디자인씽킹이 적합한 영역**이다. Nigel Cross(1982), Buchanan(1992), Lawson(2006)은 공통적으로 디자인이 심술궂은 문제를 해결하기 위한 수단임을 강조했다.

**Designerly Ways of Knowing** — Nigel Cross(1982)의 핵심 명제:
- 디자이너는 최종 결과(End Image)를 먼저 상상하고, 역방향으로 사고한다
- 이 사고방식은 알고리즘적이지 않으나 결국 맞을 수 있다 — "귀추적 논리"
- 추상적 요구사항을 구체적 사물로 변환할 때 디자이너 나름의 "사물의 언어"를 읽고 쓴다

## 4️⃣ Knowledge Funnel — Mystery → Heuristic → Algorithm

Roger Martin(2009)이 제시한 디자인씽킹 과정 모델:

```
┌─────────────────────────────┐
│         Mystery             │  ← 불명확한 문제 현상들이 산재
│  (무엇이 문제인지도 모르는 상태) │
└──────────┬──────────────────┘
           │ 탐구·관찰·직관
           ▼
┌─────────────────────────────┐
│         Heuristic           │  ← 어림짐작으로 가능성을 좁혀나감
│   (주먹구구식 경험적 규칙 적용)   │
└──────────┬──────────────────┘
           │ 반복 수행·검증
           ▼
┌─────────────────────────────┐
│         Algorithm           │  ← 표준화된 절차로 해결
│    (정형화된 공식·프로세스)       │
└─────────────────────────────┘
```

- **Heuristics**: 시간·정보가 불충분한 상황에서 신속하게 사용하는 어림짐작의 기술. 그리스어 "발견하다(to find)"에서 유래
- **Algorithm**: 문제를 해결하기 위해 정해진 일련의 절차

> [!tip] 불교 도오점수(頓悟漸修)와의 연결
> 갑작스런 깨달음(頓悟)도 반드시 점진적 수행(漸修)의 축적 위에 선다. 아르키메데스의 유레카도, 스티브 잡스의 혁신도 — 경험적 지식의 반복 수행 없이는 불가능했다. **디자인씽킹은 논리적이고 체계화된 휴리스틱이다.**

## 5️⃣ 논리적 추론의 3가지 방식

문제해결 시 디자인적 사고가 활용하는 핵심 사고법은 **귀추논리(Abductive Logic)** 이다. 비교를 위해 3가지를 나란히 본다.

### 연역법 (Deduction)

**흐름: 규칙 → 사례 → 결과**

```
모든 사람은 죽는다.        ← 규칙 (대전제)
소크라테스는 사람이다.     ← 사례 (소전제)
소크라테스는 죽는다.       ← 결과 (결론)
```

특성: **필연적 사실** — 전제가 참이면 결론도 반드시 참

---

### 귀납법 (Induction)

**흐름: 결과 → 사례 → 규칙**

```
소크라테스가 죽었다. 공자도 죽었다. 석가도 죽었다.  ← 결과 (관찰)
소크라테스, 공자, 석가는 사람이다.               ← 사례
모든 사람은 죽는다.                             ← 규칙 (도출)
```

특성: **개연적 사실** — 대부분 그렇지만 예외 가능 (통계적 유의성과 연결)

---

### 귀추법/가추법 (Abduction) ← 디자인씽킹의 핵심

**흐름: 결과 → 규칙 → 사례**

```
굴뚝에 연기가 난다.                    ← 결과 (관찰/현상)
아궁이에 불을 때면 굴뚝에 연기가 난다.   ← 규칙
아궁이에 불을 때는구나.                ← 사례 (추론)
  ↓
아궁이에 불을 때고 있으니, 밥을 짓겠구나. ← 새로운 지식 생산
```

특성: **이미 일어났지만 아직 모르는 사실** — 새로운 지식을 창출할 수 있는 유일한 논증 방식

| 구분 | 연역법 | 귀납법 | 귀추법 |
|------|------|------|------|
| 흐름 | 규칙→사례→결과 | 결과→사례→규칙 | 결과→규칙→사례 |
| 특성 | 필연적 사실 | 개연적 사실 | 이미 일어난 미지의 사실 |
| 역할 | 기존 지식 검증 | 일반 법칙 도출 | 새로운 지식 창출 |
| 대표 활용 | 수학적 증명 | 논문·실험 연구 | 디자인씽킹·진단·디버깅 |

## 6️⃣ 귀추법 심화 — 퍼스와 기호학

**찰스 샌더스 퍼스(Charles Sanders Peirce, 1839~1914)**
- 미국 프래그머티즘(실용주의) 철학의 창시자
- 귀추법 개념을 체계적으로 정립한 현대 철학의 거장
- 은둔 천재 — 하버드에서 쫓겨나 지하 서재에서 연구했고, 사후에 그 가치가 인정됨

**퍼스의 기호학 삼분류**:

| 기호 유형 | 영문 | 정의 | 예시 |
|----------|------|------|------|
| **상징** | Symbol | 사회문화적 약속에 의한 자의적 의미 | 비둘기 → 평화, 해골 → 위험 |
| **지표** | Index | 대상을 직접 가리키는 지시적 의미 | 화장실 픽토그램, 방향 화살표 |
| **도상** | Icon | 대상과 시각적으로 유사한 기호 | 앱 아이콘, UI 아이콘 |

> [!info] 존 듀이(John Dewey)와의 연결
> "인간이 무언가를 이해할 수 있는 것은 오직 경험을 통해서만 가능하다." — 경험 없이는 의미 있는 휴리스틱도, 귀추적 추론도 불가능하다. 경험의 축적이 디자인씽킹의 기반이다.

## 7️⃣ [AI 생성 참고용] 셰이더 디자이너 · 그래픽스 엔지니어 관점

> [!example] 귀추법은 셰이더 디버깅의 일상 언어다

### ① 귀추법과 GPU 디버깅

그래픽스 엔지니어가 매일 수행하는 디버깅 프로세스는 귀추법의 정확한 구현이다.

```
[Shadow Acne 발생] (결과/관찰)
     ↓
Self-shadowing bias가 너무 작으면 Shadow Acne가 생긴다 (규칙)
     ↓
Depth bias 값이 너무 작은 것이다 (가설/사례)
     ↓
→ bias 값을 올리면 Shadow Acne가 사라진다 (새로운 지식 생산)

[Z-fighting 발생] (결과)
     ↓
Depth buffer 정밀도가 낮거나 두 면이 같은 깊이일 때 Z-fighting 발생 (규칙)
     ↓
이 두 폴리곤이 거의 같은 Z 값을 가진다 (가설)
     ↓
→ Polygon offset 또는 카메라 near plane 조정으로 해결 (지식 생산)
```

철학자 퍼스가 1880년대에 정의한 귀추법과, 21세기 그래픽스 엔지니어의 디버깅 사고는 구조적으로 동일하다.

### ② Designerly Ways of Knowing과 셰이더 아티스트

Nigel Cross(1982): "디자이너는 최종 결과(End Image)를 먼저 상상하고 역방향으로 사고한다."

```
셰이더 아티스트의 작업 방식:
  1. 원하는 재질 표현을 머릿속에 시각화 (End Image)
  2. "이 느낌은 SSS + thin-film iridescence 조합일 것" (귀추적 가설)
  3. BRDF 수식과 파라미터를 역방향으로 설계
  4. 프로토타입 → 수정 → 반복

이것이 Cross가 말한 "디자이너의 사물의 언어를 읽고 쓰는 방식"이다.
```

### ③ Wicked Problem과 실시간 렌더링 최적화

실시간 렌더링 최적화는 전형적인 **Wicked Problem**이다.

```
"이 씬의 퍼포먼스를 어떻게 개선할 것인가?"

→ 정답이 없다 (Well-defined가 아님)
→ 해결안이 새 문제를 만든다 (Overdraw 줄이면 배칭 복잡도 증가)
→ 타겟 플랫폼·아트 디렉션·예산에 따라 해결책이 완전히 달라짐
→ "최선"인지 확인하기 어려움 (검증 기준 불명확)
```

디자인씽킹적 접근: 확산(여러 최적화 방안 브레인스토밍) → 수렴(프로파일링으로 병목 특정) → 프로토타이핑(A/B 테스트) → 반복

### ④ Knowledge Funnel과 렌더링 파이프라인 개발

```
Mystery:    "왜 이 씬이 어두워 보이는가?" (RenderDoc 열기 전)
     ↓
Heuristic:  "이전 경험상 Exposure 설정이 원인일 가능성이 높다"
            "또는 Tone mapping curve가 달라졌을 것"
     ↓
Algorithm:  EV 값 검증 → Auto Exposure 디버그 뷰 → 원인 특정
            → 표준화된 Exposure 검증 체크리스트로 문서화
```

- 숙련된 그래픽스 엔지니어의 **휴리스틱**이 바로 경험에서 나온다
- 같은 버그 유형을 10번 마주치면 → 다음에는 즉시 원인을 추론 (직관적 귀추)
- 그 경험이 축적되어 팀 공용 체크리스트(Algorithm)가 된다

### ⑤ 기호학과 셰이더/UI 디자인

퍼스의 기호학 삼분류는 셰이더 및 UI 디자인에 직접 적용된다.

| 기호 유형 | 셰이더·그래픽스 적용 사례 |
|----------|----------------------|
| **Symbol (상징)** | 빨간색 HP 바, 황금색 레어 아이템 테두리 — 문화적 약속에 의한 의미 |
| **Index (지표)** | 대미지 방향 표시 화살표, 미니맵 적 위치 표시 — 직접 지시하는 기호 |
| **Icon (도상)** | UI 아이콘, 3D 아이템 프리뷰 — 대상과 시각적으로 닮은 기호 |

셰이더 아티스트가 재질(Material)을 설계할 때도 이 삼분류를 직관적으로 사용한다. Metal은 빛의 물리적 반응(Index), 황금색은 문화적 가치(Symbol), 금속 질감의 texture는 실제 금속과 유사한 도상(Icon)이다.

## 8️⃣ Key Insights

> [!success] 인사이트 1: 귀추법은 그래픽스 엔지니어의 모국어였다
> 퍼스가 1880년대에 정의한 귀추법 — "관찰된 현상에서 가장 그럴듯한 설명을 찾는 추론" — 이 GPU 버그 트래킹, 렌더링 아티팩트 진단, 셰이더 최적화 방향 결정에서 내가 매일 사용하는 사고방식이었다. 이름 없이 쓰던 것에 이름이 생겼다. 귀추법을 의식적으로 훈련하면 디버깅 속도와 정확도가 올라갈 것이다.

> [!success] 인사이트 2: Designerly Ways of Knowing이 셰이더 아티스트를 설명한다
> Nigel Cross(1982)가 "디자이너는 알고리즘적으로 생각하지 않고, 마지막 이미지를 먼저 상상한 뒤 역방향으로 끼워 맞춘다"고 했는데, 이것이 셰이더 아티스트의 창작 과정 그대로다. 원하는 비주얼을 먼저 그리고, 그 비주얼을 만들 BRDF 수식·노드 그래프를 역추적하는 방식. 이 방식이 "설명이 안 된다"고 느껴지지만 결국 맞는 이유를 Cross가 학술적으로 증명했다.

> [!success] 인사이트 3: 실시간 렌더링 최적화는 Wicked Problem이다
> "최적의 렌더링 성능"은 Well-defined가 아니다. 플랫폼·아트 스타일·예산에 따라 해결책이 완전히 달라지고, 한 문제를 해결하면 다른 문제가 생긴다. 디자인씽킹의 수렴-확산 사이클이 렌더링 최적화 작업에서도 그대로 필요하다. "왜 우리 팀의 최적화 작업이 항상 끝이 없는가" — Wicked Problem이기 때문이다.

> [!success] 인사이트 4: Knowledge Funnel이 팀 기술 성장 구조를 설명한다
> Mystery(새 기술 접하는 주니어) → Heuristic(경험 쌓이는 미드레벨) → Algorithm(팀 공용 체크리스트·문서화하는 시니어). 그래픽스 팀의 기술 성장도 이 깔때기 구조를 따른다. 내가 지금 어느 단계인지 인식하고, 경험을 문서화하여 Heuristic을 Algorithm으로 끌어올리는 것이 시니어로 가는 길이다.
