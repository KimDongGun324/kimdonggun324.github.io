## 1️⃣ 렌더링 파이프라인이란 — 개요와 아키텍처

**렌더링 파이프라인(Graphics Rendering Pipeline)**은 가상 카메라, 3D 오브젝트, 광원 등을 입력으로 받아 **2D 이미지를 생성하는 실시간 렌더링의 핵심 도구**이다.

파이프라인은 공장 조립 라인과 같이 **여러 단계(stage)가 순차·병렬로 처리**된다:

- 단계들은 병렬로 실행되며, 각 단계는 이전 단계의 결과에 의존한다
- n개의 단계로 나뉘면 이상적으로 n배의 속도 향상을 기대할 수 있다
- 전체 속도는 **병목(bottleneck)** 단계에 의해 결정된다

> [!tip] 병목(Bottleneck) 개념
> 고기를 추가하는 단계가 30초 걸리면 빵 준비(20초)·토핑(20초)이 아무리 빨라도 분당 2개 이상 만들 수 없다. GPU 렌더링도 동일하다. **Vertex-bound, Fragment-bound, Bandwidth-bound** — 어느 단계가 병목인지 파악하는 것이 최적화의 첫 걸음이다.

## 2️⃣ 파이프라인 4단계 개요

실시간 렌더링 파이프라인은 크게 **4단계**로 구성된다.

```
Application  →  Geometry Processing  →  Rasterization  →  Pixel Processing
   (CPU)              (GPU)                  (GPU)              (GPU)
```

| 단계 | 실행 위치 | 주요 역할 |
|------|----------|---------|
| **애플리케이션(Application)** | CPU | 충돌 감지, 물리, 애니메이션, 컬링 알고리즘, 렌더링 프리미티브 전달 |
| **지오메트리 처리(Geometry Processing)** | GPU | 버텍스 셰이딩, 투영, 클리핑, 스크린 매핑 |
| **래스터화(Rasterization)** | GPU | 삼각형 설정(Triangle Setup), 삼각형 순회(Triangle Traversal) |
| **픽셀 처리(Pixel Processing)** | GPU | 픽셀 셰이딩, 병합(z-버퍼, 블렌딩, 스텐실) |

> [!info] 렌더링 속도 단위
> - **FPS(Frames Per Second)**: 초당 렌더링 프레임 수. 특정 프레임 또는 평균 성능 표현
> - **Hz**: 하드웨어 고정 주파수 (모니터 주사율 등)
> - **ms(밀리초)**: 단일 프레임 렌더링 시간. 성능 분석 시 가장 직관적

## 3️⃣ 지오메트리 처리 — 버텍스 셰이딩과 좌표 변환

### 좌표 공간 변환 흐름

모델이 화면에 그려지기까지 다음 좌표 공간을 순차로 거친다.

```
모델 공간          월드 공간          카메라/뷰 공간       클립 공간            NDC               화면 공간
(Model Space)  →  (World Space)  →  (View/Eye Space)  →  (Clip Space)  →  (NDC, ÷w)  →  (Screen Space)
   모델 변환           뷰 변환            투영 변환          원근 나눗셈          뷰포트 변환
  (4×4 Matrix)      (4×4 Matrix)       (4×4 Matrix)
```

| 공간 | 설명 | 변환 행렬 |
|------|------|---------|
| **모델 공간** | 오브젝트 고유의 로컬 좌표. 변환 미적용 | — |
| **월드 공간** | 모든 오브젝트가 공유하는 전역 공간. 위치·방향·크기 결정 | Model Transform |
| **뷰 공간(카메라 공간)** | 카메라를 원점에, 음의 z축 방향으로 바라보도록 재배치 | View Transform |
| **클립 공간** | 동차 좌표계(Homogeneous). w 나눗셈 이전 상태 | Projection Matrix |
| **NDC** | 원근 나눗셈(÷w) 후 (-1,-1,-1) ~ (1,1,1) 정규화 범위 | — |
| **화면 공간** | 실제 픽셀 좌표로 매핑된 최종 공간 | Viewport Transform |

> [!warning] 버텍스 셰이더의 의무
> GPU의 버텍스 셰이더는 반드시 **클립 공간 좌표(Clip Coordinates)**를 출력해야 한다. 이는 API 규칙이 아니라, 다음 단계인 클리핑이 항상 **단위 정육면체(Canonical View Volume)**를 기준으로 동작하도록 파이프라인이 설계되었기 때문이다.

### 두 가지 투영 방식

| 구분 | 직교 투영(Orthographic) | 원근 투영(Perspective) |
|------|----------------------|----------------------|
| 뷰 볼륨 형태 | 직사각형 박스 | 절두체(Frustum) |
| 평행선 보존 | ✅ 유지됨 | ❌ 지평선에서 수렴 |
| 원근감 | 없음 | 있음 (멀수록 작게) |
| 주요 사용처 | 건축, 2D 게임, UI | 대부분의 3D 렌더링 |

## 4️⃣ 선택적 지오메트리 단계 — Tessellation · Geometry Shader · Stream Output

버텍스 셰이딩 후, 세 가지 선택적 단계가 이 순서로 사용될 수 있다.

### 테셀레이션(Tessellation)

곡면을 **카메라 거리에 따라 동적으로 적절한 삼각형 수**로 분할하는 단계.

```
입력 패치(Patch)
  ↓ Hull Shader        — 패치의 분할 수준(LOD) 결정
  ↓ Tessellator        — 실제 분할 수행 (고정 기능 하드웨어)
  ↓ Domain Shader      — 분할된 각 정점의 최종 위치 계산
  ↓ 더 많은 삼각형 출력
```

가까이 있으면 많은 삼각형, 멀리 있으면 적은 삼각형 — **동적 LOD(Level of Detail)**의 핵심 메커니즘이다.

### 지오메트리 셰이더(Geometry Shader)

프리미티브(점·선·삼각형)를 입력받아 **새로운 프리미티브를 생성**할 수 있는 단계. 대표 활용: 파티클 시스템에서 점(point) → 카메라를 향한 쿼드(quad) 변환.

### 스트림 출력(Stream Output)

처리된 정점을 화면 렌더링 대신 **버퍼로 출력**하는 단계. GPU를 순수 지오메트리 엔진으로 활용하며, 파티클 시뮬레이션에서 활용된다.

## 5️⃣ 클리핑과 스크린 매핑

### 클리핑(Clipping)

뷰 볼륨 **외부의 프리미티브는 제거**, 걸쳐 있는 프리미티브는 **경계면에서 잘라낸다**.

```
단위 정육면체 내부  →  그대로 전달
단위 정육면체 외부  →  버림
경계에 걸침         →  클리핑 후 새 정점 생성
```

투영 행렬을 사용하는 이유 중 하나는 클리핑을 항상 **단위 정육면체 기준으로 일관되게** 처리할 수 있기 때문이다. 이후 **원근 나눗셈(÷w)**을 통해 NDC로 변환된다.

### 스크린 매핑(Screen Mapping)

NDC의 x·y 좌표를 **실제 윈도우 픽셀 좌표로 변환**. z좌표([−1,+1] 또는 [0,1])도 [z1, z2] 범위로 재매핑된다.

> [!info] OpenGL vs DirectX 픽셀 좌표 원점 차이
> - **OpenGL**: (0,0) → 이미지 **좌측 하단**
> - **DirectX**: (0,0) → 이미지 **좌측 상단**
>
> API를 전환할 때 텍스처 UV 플립(Flip) 이슈가 발생하는 원인이 바로 여기에 있다.

## 6️⃣ 래스터화 — 삼각형 설정과 순회

래스터화(**Rasterization**, 스캔 변환(Scan Conversion))는 **3D 정점 → 화면 픽셀**로의 변환이다. 두 하위 단계로 구성된다.

### 삼각형 설정(Triangle Setup)

삼각형의 미분값, 에지 방정식, 기타 데이터를 계산. **고정 기능 하드웨어**가 담당.

### 삼각형 순회(Triangle Traversal)

삼각형 내부에 중심(또는 샘플)이 포함된 픽셀을 찾아 **프래그먼트(Fragment)**를 생성. 세 정점의 속성(깊이, 색상, UV 등)을 **보간(Interpolation)**하여 각 프래그먼트에 할당한다.

**픽셀 포함 판정 방식:**

| 방식 | 설명 | 특성 |
|------|------|------|
| 포인트 샘플링 | 픽셀 중심 1개만 검사 | 계단 현상(Aliasing) 발생 |
| MSAA (Multisampling) | 픽셀당 여러 샘플로 판정 | 안티앨리어싱 효과 |
| 보수적 래스터화 | 픽셀과 조금이라도 겹치면 포함 | 정확한 오버랩 감지 필요 시 |

> [!tip] 원근 보정 보간 (Perspective-Correct Interpolation)
> 삼각형 순회에서 UV·색상 등을 단순 선형 보간하면 원근 왜곡이 발생한다. 이를 방지하기 위해 **원근 나눗셈을 고려한 보정 보간(Rational Linear Interpolation)**이 수행된다. 이것이 텍스처가 3D 공간에서 올바르게 보이는 이유이다.

## 7️⃣ 픽셀 처리 — 픽셀 셰이딩과 병합

### 픽셀 셰이딩(Pixel Shading)

보간된 셰이딩 데이터를 입력으로, **프래그먼트당 색상을 계산**하는 단계. 프로그래머블 GPU 코어가 실행하며, 개발자가 작성하는 **픽셀 셰이더(HLSL/GLSL의 fragment shader)**가 바로 이 단계이다.

대표 기법: **텍스처링(Texturing)** — 2D 이미지를 3D 표면에 "붙이는" 작업 (6장에서 상세 논의).

### 병합(Merging) — ROP

픽셀 셰이더 출력 색상을 컬러 버퍼와 결합하는 단계. **ROP(Render Output Unit)**라고도 불린다.

#### z-버퍼(Depth Buffer) 알고리즘

```
새로운 프래그먼트의 z값  <  z-버퍼의 현재 z값
        ↓ Yes                   ↓ No
  컬러 버퍼 + z-버퍼 업데이트      변경 없음 (뒤에 가려져 있음)
```

| 특성 | 내용 |
|------|------|
| 시간 복잡도 | O(n) — 프리미티브 수에 선형 비례 |
| 렌더링 순서 | 대부분의 불투명 오브젝트는 **임의 순서**로 가능 |
| 한계 | **투명(Transparent) 오브젝트 처리 불가** — 단일 깊이만 저장 |

> [!warning] 투명 오브젝트 처리 규칙
> 투명 오브젝트는 모든 **불투명 오브젝트 렌더링 후**, **뒤에서 앞(back-to-front)** 순서로 렌더링해야 한다. 또는 **OIT(Order-Independent Transparency)** 알고리즘(5.5절)을 사용한다. 투명도는 기본 z-버퍼의 대표적 약점이다.

#### 주요 버퍼 및 채널

| 버퍼/채널 | 저장 내용 | 주요 용도 |
|----------|---------|---------|
| **컬러 버퍼** | RGB 색상 | 최종 픽셀 색상 |
| **z-버퍼(Depth Buffer)** | 깊이값(z) | 가시성 판단 |
| **알파 채널** | 불투명도(opacity) | 투명도, discard 연산 |
| **스텐실 버퍼** | 8bit 마스크 | 그림자, 반사, 특수 효과 |
| **프레임버퍼** | 위 모두를 포함 | 최종 출력 |

**더블 버퍼링(Double Buffering):** 렌더링은 **백 버퍼(Back Buffer)**에서 수행되고, 완료되면 **프론트 버퍼(Front Buffer)**와 교환된다. 수직 귀선 소거(Vertical Retrace) 타이밍에 스왑하여 화면 찢김(Tearing)을 방지한다.

## 8️⃣ 셰이더 디자이너 · 그래픽스 엔지니어 관점

> [!example] 파이프라인을 이해해야 셰이더를 제대로 쓸 수 있다

### ① 버텍스 셰이더가 Clip Space를 출력하는 이유

셰이더 코드를 처음 배울 때 "왜 `SV_Position`은 clip space여야 하는가?"가 의문이었다. RTR 2장은 그 답을 명확히 한다.

```hlsl
// HLSL Vertex Shader 출력
struct VertexOutput {
    float4 position : SV_Position; // 반드시 Clip Space 좌표
    float2 uv       : TEXCOORD0;
    float3 normal   : NORMAL;
};

VertexOutput VS(VertexInput v) {
    VertexOutput o;
    // Model → World → View → Clip 변환
    o.position = mul(UNITY_MATRIX_MVP, float4(v.position, 1.0));
    // 이 출력이 Clip Space인 이유:
    // 다음 단계인 클리핑이 항상 단위 정육면체 기준으로 작동하기 때문
    ...
}
```

클리핑 단계는 단위 정육면체에 대해 일관되게 동작해야 하므로, 버텍스 셰이더는 반드시 클립 공간 좌표를 출력해야 한다. API 규칙이 아닌 **파이프라인 설계 필연성**이다.

---

### ② 테셀레이션 셰이더와 동적 LOD

테셀레이션 단계는 셰이더 아티스트에게 매우 강력한 도구를 제공한다.

```
[Hull Shader]   : "이 패치를 카메라 거리에 따라 몇 등분으로 나눌까?"
                   → 카메라에 가까우면 64분할, 멀면 4분할
[Tessellator]   : 실제 분할 (고정 기능, 프로그래밍 불필요)
[Domain Shader] : 분할된 각 정점에 displacement map 적용
                   → 표면에 실제 요철(Bump) 생성
```

셰이더 그래프(ShaderGraph, MaterialX)에서 테셀레이션을 활성화하면 내부적으로 이 세 단계가 순서대로 실행된다. **LOD를 자동화하면서도 디스플레이스먼트 매핑을 사용**하는 것이 현대 AAA 게임의 지형 렌더링 표준이다.

---

### ③ 픽셀 셰이더와 ROP — 블렌딩 연산의 위치

```
픽셀 셰이더 (Programmable)  →  ROP/병합 단계 (Configurable, 비프로그래머블)
       ↓                              ↓
  색상 계산 (BRDF, 텍스처링)      z-테스트, 알파 블렌딩, 스텐실 테스트
  "이 픽셀의 최종 색상은?"         "이 픽셀을 버퍼에 어떻게 합칠까?"
```

알파 블렌딩 설정(`Blend SrcAlpha OneMinusSrcAlpha`)이 픽셀 셰이더 코드 안에 없는 이유가 바로 여기에 있다. 블렌딩은 **ROP(병합) 단계의 역할**이며, API 레벨에서 구성(configurable)하는 것이지 셰이더 코드에서 직접 계산하는 것이 아니다.

### ④ z-버퍼와 Depth 관련 셰이더 테크닉

z-버퍼 알고리즘을 이해하면 다음 테크닉들의 원리가 명확해진다.

| 테크닉 | 파이프라인 원리 | 실전 적용 |
|--------|--------------|---------|
| **Depth Prepass** | 불투명 오브젝트 z값만 먼저 기록 → 이후 pass에서 z-test로 불필요한 픽셀 셰이더 skip | 복잡한 PBR 셰이더의 Overdraw 감소 |
| **Reversed-Z** | z-버퍼를 [1→0]으로 뒤집어 정밀도 향상 | 먼 거리 Z-fighting 방지 |
| **Z-fighting** | 두 폴리곤의 z값이 거의 같을 때 깜박임 발생 | Polygon Offset 또는 카메라 Near Plane 조정으로 해결 |
| **Early-Z** | 픽셀 셰이더 실행 전에 z-test를 통과한 픽셀만 처리 | GPU가 자동으로 최적화. `discard`가 있으면 비활성화됨 |
| **투명 오브젝트 정렬** | z-버퍼는 단일 깊이만 저장하므로 투명체는 back-to-front 정렬 필요 | OIT(A-Buffer, Per-Pixel Linked List) 로 대체 가능 |

### ⑤ 래스터화와 안티앨리어싱 — 그래픽스 아티스트의 선택

```
[포인트 샘플링]    픽셀당 1개 샘플 → 계단 현상(Jaggies) 발생
       ↓ 비용 낮음 · 퀄리티 낮음

[MSAA]           픽셀당 4x/8x 샘플 → 엣지 부드러움 · 메모리 증가
       ↓ 비용 중간 · 지오메트리 에지에만 효과

[TAA]            시간축 누적 샘플링 → 고품질 · 고스팅(Ghosting) 발생
       ↓ 현재 AAA 표준. UE5/Unity 기본 옵션

[보수적 래스터화] 픽셀이 조금이라도 겹치면 포함 → Voxelization, Ray Tracing 가속에 활용
```

셰이더 아티스트는 안티앨리어싱 방식에 따라 **엣지의 소프트니스(edge softness)와 모션 블러(ghosting)** 사이의 트레이드오프를 이해하고 아트 디렉션을 조율해야 한다.

## 9️⃣ Key Insights

> [!success] 인사이트 1: 파이프라인은 "왜 이렇게 써야 하는가"의 답이다
> `SV_Position`이 클립 공간이어야 하는 이유, `Blend` 설정이 셰이더 밖에 있는 이유, 투명 오브젝트를 마지막에 그려야 하는 이유 — 이 모든 것이 파이프라인 구조에서 필연적으로 나온다. 파이프라인을 이해하면 API의 수많은 "관례"가 "설계 필연성"으로 보인다.

> [!success] 인사이트 2: 병목 분석은 파이프라인 4단계의 언어로 한다
> "왜 이 씬이 느린가?"를 분석할 때 — CPU-bound(애플리케이션 단계), Vertex-bound(지오메트리 처리), Rasterization-bound(래스터화), Fragment-bound(픽셀 처리), Bandwidth-bound(데이터 전송) — 4단계 프레임워크로 분류해야 올바른 최적화 방향을 찾을 수 있다. 무작정 셰이더 복잡도를 줄이는 것은 Fragment-bound일 때만 효과가 있다.

> [!success] 인사이트 3: 테셀레이션은 LOD 자동화의 열쇠다
> 메시 LOD를 수작업으로 만드는 것은 비효율적이다. 테셀레이션 셰이더(Hull + Tessellator + Domain)를 이해하고, Displacement Map과 결합하면 카메라 거리에 따라 자동으로 디테일이 조절되는 **동적 LOD 시스템**을 구현할 수 있다. 이는 현대 AAA 지형·캐릭터 렌더링의 기반이다.

> [!success] 인사이트 4: z-버퍼 이해 없이는 투명 렌더링을 할 수 없다
> z-버퍼가 "픽셀당 하나의 깊이값만 저장"한다는 근본 제약을 이해해야, 투명 오브젝트의 back-to-front 정렬이 왜 필요한지, OIT가 왜 어려운지를 설명할 수 있다. 이 제약을 모르면 투명 렌더링 버그 앞에서 원인을 알 수 없다.
