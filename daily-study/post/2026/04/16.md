### ✅ [C++] 제어 흐름 개요 Control Flow (5.1)

프로그램이 실행되는 **순서를 제어하는 방법**은 총 5가지로 분류된다.

| 분류 | 종류 | 예시 |
|------|------|------|
| **순차 실행 (Sequential)** | 기본 흐름 | 위에서 아래로 한 줄씩 |
| **점프 (Jump)** | `goto`, `break`, `continue`, `return` | 특정 위치로 이동 |
| **조건 분기 (Conditional Branch)** | `if`, `switch` | 조건에 따라 다른 경로 |
| **반복 (Loop)** | `for`, `while`, `do-while` | 블록을 반복 실행 |
| **예외 처리 (Exception Handling)** | `try`, `catch`, `throw` | 런타임 오류 대응 |

**`exit()` vs `return`의 결정적 차이:**

```cpp
#include <iostream>
#include <cstdlib>

int main()
{
    std::cout << "I love you" << std::endl;
    exit(0);                              // 프로그램 전체 즉시 종료
    std::cout << "i love you, too" << std::endl;  // 실행 안 됨
    return 0;
}
```

| 항목 | `return` | `exit()` |
|------|----------|----------|
| 종료 범위 | 현재 함수만 종료 | 프로그램 전체 종료 |
| 소멸자(destructor) 호출 | ✅ 정상 호출 | ❌ 호출 안 됨 (주의!) |
| 헤더 필요 | 불필요 | `<cstdlib>` 필요 |
| 사용 시나리오 | 일반 함수 종료 | 치명적 오류, 디버깅 강제 종료 |

**exit 코드 의미:**

```cpp
exit(0);             // 정상 종료
exit(1);             // 비정상 종료 (오류)
exit(EXIT_SUCCESS);  // exit(0)과 동일 — 권장
exit(EXIT_FAILURE);  // exit(1)과 동일 — 권장
```

> [!warning] exit() 사용 주의
> `exit()`는 지역 객체의 소멸자를 호출하지 않는다. RAII 패턴이나 파일 핸들, 동적 메모리 관리 코드가 있다면 리소스 누수가 발생할 수 있다. 가급적 `return`을 사용하고, `exit()`는 정말 필요한 경우에만 쓸 것.

### ✅ [C++] 조건문 if (5.2)

**기본 구조와 else if 체이닝:**

```cpp
if (x > 10)
{
    std::cout << "x는 10 초과" << std::endl;
}
else if (x > 5)
{
    std::cout << "x는 5 초과 10 이하" << std::endl;
}
else
{
    std::cout << "x는 5 이하" << std::endl;
}
```

**Dangling else — `else`는 항상 가장 가까운 `if`에 붙는다:**

```cpp
// 들여쓰기가 속인다!
if (x > 0)
    if (x > 10)
        std::cout << "10 초과" << std::endl;
else                  // ← 이 else는 'if (x > 0)'이 아니라 'if (x > 10)'에 붙음!
    std::cout << "10 이하" << std::endl;

// 해결: {} 반드시 사용
if (x > 0)
{
    if (x > 10)
        std::cout << "10 초과" << std::endl;
}
else
{
    std::cout << "0 이하" << std::endl;
}
```

**null statement — 세미콜론이 if를 즉시 끝낸다:**

```cpp
if (x > 10);          // ← 세미콜론이 if 블록을 비웁니다 (if 본체 = 빈 문장)
{
    std::cout << "항상 실행됨!" << std::endl;  // if와 무관, 무조건 실행
}
```

**대입과 비교 혼동 + Yoda Condition:**

```cpp
// 잘못된 코드 — 대입(assign)이 발생, x가 0이 되고 조건은 false
if (x = 0) { ... }    // 버그! x에 0을 대입하고, 0은 false로 평가

// 올바른 비교
if (x == 0) { ... }

// Yoda 조건: 상수를 왼쪽에 → 실수로 단일 = 를 쓰면 컴파일 에러가 발생해 바로 알 수 있음
if (0 == x) { ... }   // 더 안전한 방어 패턴
```

**단락 평가 (Short-circuit Evaluation):**

```cpp
// &&: 첫 번째가 false면 뒤는 평가 안 함
if (ptr != nullptr && ptr->value > 0)  // ptr이 null이면 ptr->value는 읽지 않음 (안전)

// ||: 첫 번째가 true면 뒤는 평가 안 함
if (x == 0 || calculateExpensive())   // x가 0이면 calculateExpensive() 호출 안 됨
```

**Early Return — 중첩 줄이기 패턴:**

```cpp
// 중첩이 많은 코드 (bad)
void process(int x)
{
    if (x > 0)
    {
        if (x < 100)
        {
            // 실제 로직...
        }
    }
}

// Early Return으로 정리 (good)
void process(int x)
{
    if (x <= 0) return;     // 조건 미충족 시 즉시 반환
    if (x >= 100) return;

    // 실제 로직 — 들여쓰기 줄어들고 읽기 쉬움
}
```

**삼항 연산자 (Ternary Operator):**

```cpp
// (조건) ? 참일 때 값 : 거짓일 때 값
int min = (x < y) ? x : y;

int getMin(int x, int y) { return (x < y) ? x : y; }
```

### ✅ [C++] Switch-case (5.3)

**기본 구조 — 정수/enum 타입 전용:**

```cpp
int x;
std::cin >> x;

switch (x)
{
case 0:
    std::cout << "Zero" << std::endl;
    break;
case 1:
    std::cout << "One" << std::endl;
    break;
default:
    std::cout << "Other" << std::endl;
    break;
}
```

**Fall-through — `break` 없으면 다음 case로 계속 실행:**

```cpp
switch (x)
{
case 0:
    std::cout << "Zero";   // break 없음 → fall-through
case 1:
    std::cout << "One";    // break 없음 → fall-through
case 2:
    std::cout << "Two";
    break;
}
// x가 0이면 출력: ZeroOneTwo
// x가 1이면 출력: OneTwo
```

> Fall-through는 의도적으로 쓸 때는 `// intentional fall-through` 주석을 반드시 달 것.

**error C2360 — case label에 의한 초기화 건너뜀:**

```cpp
switch (x)
{
case 0:
    int b = 5;    // ❌ error C2360: case 1로 점프 시 b의 초기화를 건너뜀
    break;
case 1:
    b = 6;        // 컴파일러가 거부
    break;
}

// 해결 1: 선언만, 초기화 나중에
switch (x)
{
case 0:
    int b;        // ✅ 선언만 — 초기화 없으면 건너뛰어도 OK
    b = 5;
    break;

// 해결 2 (권장): case마다 {} 블록으로 스코프 분리
case 0: { int b = 5; std::cout << b; break; }
case 1: { int b = 9; std::cout << b; break; }  // 독립적인 b
}
```

**`enum class` + switch:**

```cpp
enum class Colors { BLACK, WHITE, RED, GREEN, BLUE };

Colors color = Colors::BLACK;

switch (color)                         // enum class는 직접 switch 가능
{
case Colors::BLACK: std::cout << "BLACK"; break;
case Colors::WHITE: std::cout << "WHITE"; break;
case Colors::RED:   std::cout << "RED";   break;
default: break;
}
```

| if-else vs switch | if-else | switch |
|-------------------|---------|--------|
| 조건 타입 | 모든 타입 (bool, 범위 비교 등) | 정수/enum만 |
| 가독성 | 복잡한 조건에 유리 | 동일 변수 다값 비교에 유리 |
| 컴파일러 최적화 | 일반적 | jump table 최적화 가능 |
| Fall-through | 없음 | 의도적으로 활용 가능 |

### ✅ [셰이더] PBR(물리 기반 렌더링) 머티리얼의 이해 (4강)

**PBR(Physically Based Rendering)**: Disney가 주도하여 개발한 물리 기반 렌더링 방식. 언리얼, 유니티 모두 채택.

**설계 철학 3가지:**
- **단순성**: 아티스트가 다루기 쉬운 소수의 직관적 파라미터
- **견고성**: 실내·실외·역광 등 어떤 조명에서도 일관된 결과
- **적절한 예술적 제어**: 100% 물리 시뮬레이션이 아닌 실용적 근사

**핵심 파라미터 4가지:**

| 파라미터 | 범위 | 역할 | 실무 비중 |
|----------|------|------|----------|
| **Base Color** | RGB [0,1] | 표면 고유 색상 (조명 제거된 순수 색) | 높음 |
| **Metallic** | [0,1] | 금속/비금속 분기 (실질적으로 0 or 1) | 높음 |
| **Roughness** | [0,1] | 미세 표면 거칠기 (0:거울, 1:무광) | 높음 |
| **Specular** | [0,1] | 비금속 기본 반사율 (기본 0.5, 거의 수정 안 함) | 낮음 |

**최종 렌더링 공식:**
```
Final Color = 직접광 Diffuse + 직접광 Specular
            + 간접광 Diffuse + 간접광 Specular
```

**난반사(Diffuse) vs 정반사(Specular):**

| 반사 종류 | 특성 | 시각적 결과 |
|-----------|------|------------|
| **난반사 (Diffuse)** | 모든 방향으로 균등 산란 | 표면 전체가 표면 색으로 보임 |
| **정반사 (Specular)** | 입사각 = 반사각으로 반사 | 특정 각도에서만 광원 형태가 보임 |

**금속(Metal) vs 비금속(Dielectric):**

```
금속:    빛 입사 → 미세 표면에서 전부 흡수 → 표면에서 고유 컬러로 정반사만
비금속:  빛 입사 → 내부 굴절·산란 → 난반사 + 표면에서 약한 정반사(F0 약 4%)
```

| 특성 | 금속 (Metallic 1) | 비금속 (Metallic 0) |
|------|------------------|-------------------|
| 난반사 | ❌ 없음 | ✅ Base Color 색상 |
| 정반사 | ✅ Base Color 컬러로 틴팅 | ✅ 무채색 약 4% |
| 간접광 없는 환경 | 거의 검정 | Base Color 색으로 보임 |

**에너지 보존 법칙:**
- 과거 방식: 아티스트가 스펙큘러 컬러를 HDR로 강제 설정 → 에너지 생성 (물리 부정확)
- PBR 방식: 출력 에너지 ≤ 입력 에너지 — 번쩍이는 반사를 원하면 **환경 자체가 밝아야** 함

**실제 금속 측정값 예시:**

| 금속 | R | G | B |
|------|-----|-----|-----|
| 구리 (Copper) | 0.955 | 0.637 | 0.538 |
| 크롬 (Chrome) | 0.550 | 0.556 | 0.554 |
| 금 (Gold) | 1.000 | 0.766 | 0.336 |

> 크롬이 "거울처럼" 보이는 이유 — 고유 컬러가 회색 계열이어서 주변 환경을 거의 그대로 반사하기 때문.

## 2️⃣ Key Insights

- **제어 흐름 5분류를 알면 모든 코드가 보인다** — 순차/점프/조건분기/반복/예외, 이 5개로 모든 프로그램 로직을 설명할 수 있다
- **`exit()`는 소멸자를 호출하지 않는다** — C++ RAII(Resource Acquisition Is Initialization) 패턴을 사용하는 코드에서 `exit()`를 쓰면 리소스 누수 발생. 가급적 `return` 사용
- **Dangling else는 들여쓰기가 아니라 `{}`로 해결** — `else`는 의미상 가장 가까운 `if`에 붙는다. 들여쓰기는 컴파일러가 무시
- **null statement `if(x);`는 눈에 보이지 않는 버그** — 빈 세미콜론 하나가 if 블록 전체를 무력화. Yoda condition과 `{}`를 습관화
- **Fall-through는 의도인지 버그인지 구분하라** — `break` 생략은 실수인 경우가 많다. 의도적 fall-through는 반드시 주석 표시
- **switch case `{}`로 독립 스코프 — error C2360 방지** — 각 case를 `{}`로 감싸면 초기화 건너뜀 에러를 원천 차단하고 가독성도 향상
- **PBR에서 아티스트가 실제로 다루는 값은 3가지** — Base Color + Metallic + Roughness. Specular는 기본값(0.5)에서 거의 변경하지 않음
- **에너지 보존 법칙 = "환경이 밝아야 반사도 밝다"** — 어두운 환경에서 메탈릭을 아무리 높여도 반사가 어둡다. 과거 방식처럼 스펙큘러를 임의로 뻥튀기 불가
