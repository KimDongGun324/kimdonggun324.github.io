## 📝 초록 (Abstract)

PBR(Physically Based Rendering) 메터리얼은 현실 세계의 광학적 물리 법칙을 시뮬레이션하여 더 사실적인 표면 표현을 가능하게 한다. 기존의 수작업(Hand Painting) 텍스처링과 달리 **절차적 생성(Procedural Generation)** 방식은 컴퓨터 알고리즘으로 텍스처를 생성하므로 다음과 같은 이점을 제공한다:

- **해상도 독립성** — 어떤 해상도에서도 품질 유지
- **비파괴적 편집** — 언제든지 파라미터 조정 가능
- **아티스트 간 호환성** — 동일한 재질 정의를 공유

이 논문은 **Substance Painter**를 사용하여 Frame, Ball, Base 세 가지 오브젝트에 PBR 절차적 텍스처링을 적용하고, 이를 **Maya의 Arnold 렌더러**로 최종 렌더링하는 전체 파이프라인을 분석한다.

## 1️⃣ 서론

### 배경

3D 그래픽스 산업에서 사실적인 표면 표현은 게임, 영화, VR/AR 등 모든 분야의 핵심 과제다. 과거에는 아티스트가 포토샵 등으로 텍스처를 직접 "그리는" 방식이 주를 이루었으나, 이 방식은 다음과 같은 한계를 가진다:

1. **조명 의존성** — 수작업 텍스처는 특정 조명 환경에 맞게 그려지므로, 다른 조명에서는 부자연스럽게 보인다.
2. **확장성 부족** — 새로운 오브젝트나 재질을 추가할 때마다 처음부터 작업해야 한다.
3. **일관성 유지의 어려움** — 여러 아티스트가 협업할 때 스타일이 달라질 수 있다.

### PBR의 등장과 절차적 생성의 결합

PBR은 에너지 보존 법칙(Energy Conservation)에 기반한 수학적 모델로, **조명에 관계없이 일관된 표면 반응**을 보장한다. 여기에 절차적 생성 방식을 결합하면, 물리적으로 정확하면서도 효율적이고 재사용 가능한 텍스처링 파이프라인을 구축할 수 있다.

### 연구 목적

본 논문의 목적은:
1. PBR 메터리얼의 이론적 기반을 정리하고
2. Substance Painter를 활용한 절차적 텍스처링 워크플로우를 체계화하며
3. 실제 제작 결과물로 파이프라인의 유효성을 검증하는 것이다.

## 2️⃣ PBR 메터리얼 (Physically Based Rendering Materials)

### 2.1 렌더링 모델의 발전사 — Phong에서 PBR로

#### Phong 반사 모델 (1975)

Phong 모델은 표면의 반사를 세 가지 성분으로 분해한다:

```
최종 색상 = 주변광(Ambient) + 난반사(Diffuse) + 정반사(Specular)
```

| 성분 | 설명 | 한계 |
|------|------|------|
| **Ambient** | 환경에서 오는 전방향 조명 | 물리적 근거 없음 |
| **Diffuse** | 난산란 — Lambert의 코사인 법칙 | 에너지 보존 미적용 |
| **Specular** | 정반사 하이라이트 | 하이라이트가 과장됨 |

**Phong 모델의 근본적 한계:**
- 재질 표현이 직관적이지 않다 (Ambient/Diffuse/Specular 값은 물리적 의미가 없음)
- 에너지 보존 법칙을 만족하지 않아 — 반사된 빛이 입사된 빛보다 많아질 수 있다
- 조명 조건이 바뀌면 완전히 다른 결과를 내어 "튜닝"이 필요하다

#### Blinn-Phong (1977)

Phong의 Half-Vector 최적화 버전. 연산량은 줄었지만 근본적 한계는 동일하다.

#### Physically Based Rendering의 등장 (2000년대 후반~2010년대 초)

PBR은 다음 수학적 모델에 기반한다:

**반사 방정식 (Rendering Equation):**

```
Lo(p, ωo) = ∫ fr(p, ωi, ωo) · Li(p, ωi) · (n · ωi) dωi
```

- `Lo` — 점 p에서 방향 ωo로 나가는 빛의 양
- `fr` — BRDF (Bidirectional Reflectance Distribution Function, 양방향 반사 분포 함수)
- `Li` — 입사광의 양
- `(n · ωi)` — Lambert의 코사인 법칙

**핵심 물리 법칙:**
1. **에너지 보존** — 반사된 빛의 총량은 입사된 빛의 총량을 초과할 수 없다
2. **헬름홀츠 상반성** — 입사와 출사 방향을 바꿔도 BRDF 값은 동일하다

#### PBR이 산업 표준이 된 계기

| 연도 | 사건 |
|------|------|
| 2012 | Disney의 "Physically-Based Shading at Disney" 발표 — 실무 가능성 입증 |
| 2013 | Unreal Engine 4, Unity가 PBR 채택 |
| 2014 | Allegorithmic(현 Adobe)의 Substance Painter 출시 |
| 현재 | 모든 주요 게임/영화 파이프라인의 표준 |

### 2.2 Metallic / Roughness 워크플로우

게임 엔진(Unreal Engine, Unity)과 Substance Painter에서 기본으로 채택하는 워크플로우다.

#### 사용하는 텍스처 맵

| 맵 이름 | 채널 | 범위 | 역할 |
|---------|------|------|------|
| **Base Color** | RGB | 0–1 | 재질의 기본 색상. 비금속은 알베도, 금속은 F0 값 |
| **Metallic** | Grayscale | 0 (비금속) or 1 (금속) | 해당 픽셀이 금속인지 비금속인지 정의 |
| **Roughness** | Grayscale | 0 (매끄러움) ~ 1 (거침) | 미세 표면 거칠기. Glossiness의 반대 |
| **Normal** | RGB | [-1, 1] | 표면의 미세 요철을 법선 벡터로 표현 |
| **AO (Ambient Occlusion)** | Grayscale | 0–1 | 틈새나 구석에서 차단되는 간접광의 양 |
| **Height / Displacement** | Grayscale | 0–1 | 표면의 실제 높이 변위 (테셀레이션 필요) |

#### Metallic 값의 물리적 의미

```
Metallic = 0 (비금속, Dielectric):
  - 정반사(F0) ≈ 0.04 (약 4%) — 셰이더가 자동 설정
  - Base Color = 알베도 (난반사 색상)
  - 난반사 + 정반사 모두 존재

Metallic = 1 (금속, Conductor):
  - 정반사(F0) = Base Color에서 읽어옴 (금속마다 다름)
  - 난반사 없음 (금속은 빛을 흡수하거나 반사)
  - 오직 정반사만 존재
```

> [!important] 왜 Metallic은 0 또는 1만 사용해야 하는가?
> 물리적으로 재질은 금속이거나 비금속이다. 중간값(0.1~0.9)은 금속과 비금속의 경계(녹이 슬거나 도금이 벗겨진 부분)에만 제한적으로 사용해야 한다. 임의로 중간값을 남용하면 비물리적(non-physical) 결과가 나온다.

#### Roughness 값의 물리적 의미

Roughness는 **미세면(Microfacet)** 이론에 기반한다. 표면을 매우 작은 스케일로 보면 수많은 미세 거울(Microfacet)로 구성되며, 이 미세 거울들의 방향 분포가 Roughness를 결정한다:

```
Roughness = 0 (완전 매끄러움):
  모든 미세 거울이 같은 방향 → 날카로운 정반사 하이라이트
  예) 광택 대리석, 맑은 물

Roughness = 1 (완전 거침):
  미세 거울이 무작위 방향 → 하이라이트가 퍼져 흐릿해짐
  예) 거친 콘크리트, 매트 페인트
```

### 2.3 Specular / Glossiness 워크플로우

VFX 파이프라인(Houdini, 일부 Maya 설정)에서 주로 사용하는 워크플로우다.

#### 사용하는 텍스처 맵

| 맵 이름 | 채널 | 역할 |
|---------|------|------|
| **Diffuse** | RGB | 난반사 색상 (금속은 검정) |
| **Specular** | RGB | **정반사 색상을 직접 정의** (F0 값) |
| **Glossiness** | Grayscale | Roughness의 반전값 (1 - Roughness) |

#### Metallic vs Specular/Glossiness 차이점

| 항목 | Metallic/Roughness | Specular/Glossiness |
|------|--------------------|---------------------|
| **금속 표현** | Metallic 맵으로 자동 처리 | Specular 맵에서 직접 정의 |
| **오류 가능성** | F0가 자동화되어 실수 가능성 낮음 | F0를 직접 제어 — 비물리적 값 입력 가능 |
| **게임 엔진** | 주로 사용 | 거의 사용 안 함 |
| **VFX/영화** | 점점 증가 | 여전히 많이 사용 |
| **텍스처 메모리** | 적음 | 많음 (Specular가 RGB라 3채널 사용) |

---

### 2.4 두 워크플로우 비교 및 선택 기준

**Metallic/Roughness를 선택해야 하는 경우:**
- 게임 엔진(Unreal, Unity) 타겟
- 실시간 렌더링 환경
- Substance Painter를 사용하는 경우
- 메모리와 성능이 제한된 환경

**Specular/Glossiness를 선택해야 하는 경우:**
- 레거시 파이프라인 호환성이 필요한 경우
- VFX 스튜디오의 기존 워크플로우를 따르는 경우
- F0 값을 색상별로 세밀하게 제어해야 하는 경우

> [!tip] 본 논문의 선택
> 이 연구에서는 **Metallic/Roughness 워크플로우**를 사용하며, 이를 Maya의 **aiStandardSurface** 셰이더에 연결한다. aiStandardSurface는 Arnold 렌더러의 기본 물리 기반 셰이더다.

## 3️⃣ 절차적 텍스처링 (Procedural Texturing)

### 3.1 UV 맵핑과 UDIM 시스템

#### UV 맵핑의 기본

3D 메시의 표면을 2D 평면으로 펼치는 과정을 **UV 언래핑(UV Unwrapping)**이라 한다.

```
3D 공간 (X, Y, Z)  →  UV 공간 (U, V)
각 버텍스는 UV 좌표 (0~1 범위)를 갖는다
텍스처 이미지의 픽셀이 이 UV 좌표에 맵핑됨
```

**UV 언래핑의 핵심 원칙:**
- **시임(Seam) 최소화** — 눈에 보이는 이음새를 줄여야 한다
- **스트레칭 최소화** — 폴리곤이 찌그러지지 않아야 한다
- **UV 공간 효율 최대화** — 빈 공간 없이 UV를 배치해야 한다

#### Texel Density (텍셀 밀도)

텍셀(Texel)은 텍스처 이미지의 한 픽셀이다. **Texel Density**는 3D 오브젝트의 단위 면적당 텍셀 수를 의미한다.

```
Texel Density = 텍스처 해상도(px) / 실제 표면 면적(cm²)

예) 2048×2048 텍스처를 1m² 표면에 적용 → 2048 texels/m
```

**Texel Density가 일관되지 않으면:**
- 가까운 오브젝트는 흐릿하고, 먼 오브젝트는 선명하게 보이는 불균형 발생
- 텍스처 보간(Interpolation) 아티팩트 발생 → 경계 부분에서 픽셀이 번지거나 계단 현상

이 논문에서는 Texel Density 불일치로 인한 **인터폴레이션 아티팩트**를 언급하며, 이를 해결하기 위해 UDIM 시스템을 도입한다.

#### UDIM (U-Dimension) 시스템

UDIM은 하나의 UV 레이아웃이 아닌 **여러 UV 타일(Tile)**을 사용하는 시스템이다.

```
일반 UV:
  ┌────────┐
  │  0~1   │  단일 타일
  └────────┘

UDIM UV:
  ┌────────┬────────┬────────┐
  │1001    │1002    │1003    │
  │(0~1)   │(1~2)   │(2~3)   │
  ├────────┼────────┼────────┤
  │1011    │1012    │1013    │
  │(0~1)   │(1~2)   │(2~3)   │
  │(v:1~2) │(v:1~2) │(v:1~2) │
  └────────┴────────┴────────┘

각 타일은 독립적인 텍스처 파일을 가짐:
  texture_BaseColor.1001.png
  texture_BaseColor.1002.png
  ...
```

**UDIM의 장점:**
- 고해상도 텍스처를 여러 타일에 분산 → 메모리 한계 극복
- 각 파트에 다른 Texel Density 적용 가능
- VFX 파이프라인(Katana, Mari, Houdini)과 호환

**UDIM 타일 번호 계산법:**

```
UDIM = 1001 + u_index + (v_index × 10)

예) U: 0~1, V: 0~1 → 1001
    U: 1~2, V: 0~1 → 1002
    U: 0~1, V: 1~2 → 1011
```

### 3.2 베이킹 (Baking) — 고폴리에서 저폴리로

#### 베이킹의 개념

고폴리(High Poly) 메시의 세밀한 기하학 정보를 저폴리(Low Poly) 메시의 텍스처로 "굽는(Bake)" 작업이다. 게임에서는 폴리곤 수를 최소화해야 하지만 시각적 디테일은 유지해야 하므로 베이킹이 필수적이다.

```
High Poly (조각 수준 디테일)  →  Baking  →  Low Poly + 텍스처 맵들
수백만 폴리곤                              수천 폴리곤 + 텍스처로 디테일 표현
```

#### Substance Painter에서 베이킹하는 5가지 맵

**1. Normal Map (법선 맵)**

```
High Poly의 표면 법선 방향을 RGB로 인코딩:
  R → X축 법선 성분
  G → Y축 법선 성분
  B → Z축 법선 성분

값 범위: 0~255 (픽셀) → -1~1 (벡터)
중간값(128, 128, 255)는 완전히 평평한 표면 (법선이 위를 향함)
```

**Normal Map의 두 가지 포맷:**

| 포맷 | G채널 | 사용처 |
|------|-------|--------|
| **OpenGL** | 위가 밝음 | Substance, Unreal, Blender |
| **DirectX** | 아래가 밝음 | Unity (기본), Maya |

> [!warning] 포맷 불일치 주의
> Substance Painter에서 OpenGL Normal Map을 익스포트해 DirectX 타겟에 적용하면 표면이 뒤집어진 것처럼 보인다. 익스포트 설정에서 반드시 확인해야 한다.

**2. Ambient Occlusion (AO) Map**

```
빛이 닿기 어려운 틈새, 구석, 오목한 부분을 어둡게 표현
값: 0 (완전 차단, 검정) ~ 1 (완전 노출, 흰색)

물리적 의미: 반구(hemisphere)에서 오는 빛이 차단되는 비율
```

AO는 주변광의 가려짐을 사전 계산(Pre-computed)한 텍스처다. 실시간 렌더링에서 AO를 베이킹해 사용하면 연산 비용 없이 디테일한 그림자 효과를 얻을 수 있다.

**3. Curvature Map (곡률 맵)**

```
볼록한 부분(Convex): 흰색 (1.0)
평평한 부분: 중간 회색 (0.5)
오목한 부분(Concave): 검정 (0.0)
```

Curvature Map의 핵심 용도는 **엣지 웨어(Edge Wear)** 표현이다. 실제 금속 물체는 사용하면서 볼록한 엣지가 마모되어 금속이 드러나고, 오목한 곳에는 때나 먼지가 쌓인다. Substance Painter의 Generator가 이 정보를 자동으로 활용한다:

```
Metal Edge Wear Generator:
  → Curvature의 볼록한 부분(흰색)에 마모 효과 적용
  → 도장이 벗겨져 금속 원재료 노출

Dirt Generator:
  → AO의 오목한 부분(어두운 부분)에 오염 효과 적용
  → 먼지, 기름때, 녹 표현
```

**4. Position Map (위치 맵)**

```
메시의 각 점이 3D 공간에서의 위치를 RGB로 인코딩:
  R → X 위치
  G → Y 위치
  B → Z 위치

오브젝트 바운딩 박스를 0~1로 정규화
```

Position Map은 위치 기반 마스킹에 사용된다. 예를 들어 오브젝트의 하단부(Y가 낮은 부분)에만 물때나 진흙 효과를 적용할 때 유용하다.

**5. Thickness Map (두께 맵)**

```
메시의 두께를 측정 (다른 쪽 표면까지의 거리):
  얇은 부분: 밝음 (빛이 투과하기 쉬움)
  두꺼운 부분: 어두움 (빛이 투과하기 어려움)
```

Thickness Map의 주요 용도:
- **Subsurface Scattering(SSS, 표면하 산란)** — 피부, 초, 이파리처럼 빛이 재질 내부를 통과해 산란되는 효과
- 얇은 부분은 빛이 더 많이 투과 → 더 밝고 따뜻한 색상으로 표현

#### 베이킹 품질에 영향을 주는 요인

1. **Cage(케이지)** — Low Poly에서 법선을 쏘는 방향을 제어하는 껍질. 케이지 설정이 잘못되면 아티팩트 발생
2. **Max Ray Distance** — 광선이 High Poly에 닿는 최대 거리. 너무 작으면 일부 디테일 누락
3. **해상도** — 4K(4096px) 이상 권장. 낮으면 날카로운 엣지에서 계단 현상
4. **앤티앨리어싱** — 베이킹 시 멀티샘플링으로 경계 부드럽게 처리

### 3.3 Substance Painter 워크플로우

#### 레이어 구조 이해

Substance Painter의 레이어는 포토샵과 비슷하지만, 각 레이어가 RGBA 하나의 채널이 아닌 **여러 PBR 채널(Base Color, Metallic, Roughness, Normal 등)을 동시에 제어**한다.

```
레이어 스택 (위에서 아래로 적용):
  ┌─────────────────────────────┐
  │ [Fill Layer] Edge Wear      │  ← 최상단 (가장 나중에 적용)
  │   Mask: Metal Edge Wear     │
  ├─────────────────────────────┤
  │ [Fill Layer] Dirt           │
  │   Mask: Dirt Generator      │
  ├─────────────────────────────┤
  │ [Fill Layer] Base Material  │  ← 기본 재질
  └─────────────────────────────┘
```

#### Fill Layer (필 레이어)

캔버스 전체를 단일 재질 값으로 채우는 레이어다. Paint Layer와 달리 브러시로 직접 그리지 않고 파라미터로 재질을 정의한다.

```
Fill Layer 사용 예:
  Base Color: [0.5, 0.3, 0.1]  ← 갈색 도장
  Metallic: 0                   ← 비금속
  Roughness: 0.7                ← 약간 거친 도장 표면
```

#### Generator (제너레이터)

Generator는 **베이킹된 맵을 참조해 자동으로 마스크를 생성**하는 알고리즘이다. 손으로 마스크를 그릴 필요 없이 물리적으로 자연스러운 마모, 오염 패턴을 만든다.

**Metal Edge Wear Generator:**
```
입력: Curvature Map (볼록한 엣지 = 흰색)
출력: 엣지 위치에 마모 효과가 있는 마스크

파라미터:
  - Wear Amount: 마모 강도
  - Wear Sharpness: 마모 경계의 선명도
  - Noise Scale: 노이즈 패턴의 크기 (자연스러운 불규칙성)
```

**Dirt Generator:**
```
입력: AO Map (틈새 = 어두움) + Curvature Map (오목한 부분 = 어두움)
출력: 오목하고 좁은 부분에 오염이 쌓인 마스크

파라미터:
  - Dirt Level: 오염 강도
  - Dirt Contrast: 오염과 청결 부분의 대비
```

#### 절차적 텍스처링의 핵심 장점

```
수작업(Hand Painting) 방식:
  아티스트 A → 스타일 A의 마모 패턴
  아티스트 B → 스타일 B의 마모 패턴
  = 일관성 없음, 비물리적

Generator 방식:
  메시의 기하학(곡률, AO)에서 자동 계산
  = 물리적으로 정확, 아티스트 무관 일관성
  = 메시가 바뀌어도 파라미터 재적용으로 즉시 업데이트
```

### 3.4 결과물 — Frame / Ball / Base 오브젝트

#### Frame 오브젝트

금속 프레임 구조물에 적용된 텍스처:
- **기본 재질**: 도장된 금속 (Metallic: 1, Roughness: 0.4)
- **Metal Edge Wear Generator** 적용: 프레임 엣지에 도장이 벗겨져 금속 원재료 노출
- **Dirt Generator** 적용: 볼트 구멍, 코너, 오목한 부분에 오염 표현
- **Fill Layer**로 레이어별 재질 분리 (도장 레이어 / 노출 금속 레이어 / 오염 레이어)

#### Ball 오브젝트

구형 오브젝트에 적용된 텍스처:
- **기본 재질**: 매끄러운 금속 공 (Metallic: 1, Roughness: 0.1~0.2)
- 구형 특성상 엣지 웨어가 균일하게 분포
- **Position Map** 활용: 공의 아래쪽(바닥과 닿는 부분)에 오염 집중 표현
- **Curvature Map** 정보가 구형에서 균일하게 나타남 → 미세하고 고르게 마모

#### Base 오브젝트

받침대/기저부 오브젝트:
- **기본 재질**: 두꺼운 금속 판 (Metallic: 1, Roughness: 0.6)
- **AO Map** 활용: 나사, 볼트, 홈 부분에 강한 Dirt 효과
- **Position Map** 활용: 하단부에 더 강한 오염
- **Thickness Map** 적용: 얇은 돌출부의 약한 SSS 표현

### 3.5 텍스처 익스포트 및 Maya Arnold 렌더링

#### Substance Painter에서 익스포트

Substance Painter는 다양한 타겟 파이프라인에 맞는 익스포트 프리셋을 제공한다:

```
익스포트 프리셋 선택: Arnold (AiStandard)

생성되는 파일:
  오브젝트_BaseColor.png     → aiStandardSurface: Base Color
  오브젝트_Metalness.png     → aiStandardSurface: Metalness
  오브젝트_Roughness.png     → aiStandardSurface: Roughness
  오브젝트_Normal.png        → aiStandardSurface: Normal Camera
  오브젝트_Height.png        → Displacement Shader 연결
```

#### Maya에서 aiStandardSurface 셰이더 연결

**Table 1에서 발췌한 텍스처-셰이더 매핑:**

| 텍스처 맵 | aiStandardSurface 입력 | 비고 |
|-----------|----------------------|------|
| Base Color | Base Color | 직접 연결 |
| Metallic | Metalness | 직접 연결 |
| Roughness | Roughness | 직접 연결 |
| Normal | Normal Camera | aiNormalMap 노드를 통해 연결 |
| Height | Displacement Shader | aiDisplacement 노드를 통해 연결 |

> [!important] Normal과 Height의 연결 방법 차이
> **Normal Map**: `File` 노드 → `aiNormalMap` 노드 → `aiStandardSurface.normalCamera`
> **Height Map**: 노말과 달리 변위(Displacement)이므로 셰이딩 그룹의 `Displacement Shader`에 `aiDisplacement` 노드를 통해 연결해야 한다. aiStandardSurface의 채널에 직접 연결하는 것이 아님을 주의.

#### Arnold 렌더러의 역할

Arnold는 **경로 추적(Path Tracing)** 알고리즘 기반의 물리적으로 정확한 렌더러다:

```
경로 추적 알고리즘:
  카메라 → 픽셀 → 광선 발사 → 표면 히트 →
  BRDF에 따라 여러 방향으로 반사 →
  재귀적 광선 추적 (수천 번) →
  평균내어 최종 색상 결정
```

이 방식은 전역 조명(Global Illumination), 간접 조명, 소프트 셰도우, 코스틱 등을 물리적으로 정확하게 시뮬레이션하므로, PBR 메터리얼의 물리적 속성이 실제와 가까운 결과를 만들어낸다.

## 🔬 핵심 개념 심층 분석

### BRDF — 물리 기반 렌더링의 수학적 핵심

BRDF (Bidirectional Reflectance Distribution Function)는 재질이 빛을 어떻게 반사하는지 수학적으로 기술하는 함수다.

```
fr(ωi, ωo) = 반사 복사 휘도(Radiance) / 입사 조도(Irradiance)
단위: sr⁻¹ (스테라디안의 역수)

입력:
  ωi — 입사광 방향 (단위 벡터)
  ωo — 관찰 방향 (단위 벡터)

출력:
  특정 방향으로 반사되는 빛의 비율
```

**PBR에서 사용하는 Cook-Torrance BRDF:**

```
fr(ωi, ωo) = [Diffuse Term] + [Specular Term]

Diffuse Term (난반사):
  fLambert = c/π
  c = 알베도(Base Color)

Specular Term (정반사):
  fCT = D(h) · F(ωo, h) · G(ωi, ωo, h) / (4 · (n·ωi) · (n·ωo))

  D — 법선 분포 함수 (Normal Distribution Function, NDF)
      GGX/Trowbridge-Reitz 모델 사용
      Roughness에 의해 제어

  F — 프레넬 방정식 (Fresnel Equation)
      Schlick 근사: F(θ) = F0 + (1-F0)(1-cosθ)^5
      관찰 각도에 따른 반사율 변화

  G — 기하 감쇠 함수 (Geometry Function)
      미세면이 서로 가리는 효과
      Smith's Shadowing-Masking Term 사용
```

### Fresnel 효과 — 왜 비스듬히 보면 반사가 강해지는가

```
정면(θ ≈ 0°): 반사율 = F0 (재질 고유값)
  비금속: ~4%, 금속: ~50~100%

비스듬히(θ → 90°): 반사율 → 100%
  모든 재질은 비스듬한 각도에서 강하게 반사

실생활 예:
  - 정면에서 보는 호수: 바닥이 보임 (낮은 반사)
  - 멀리서 보는 호수: 하늘이 반사 (강한 반사)
```

Schlick 근사법:
```
F(θ) = F0 + (1 - F0) × (1 - cos(θ))^5

F0는 재질에 따라 고정:
  물(Water): 0.02
  유리(Glass): 0.04
  플라스틱: 0.04
  금(Gold): [1.0, 0.766, 0.336]
  은(Silver): [0.972, 0.960, 0.915]
  철(Iron): [0.562, 0.565, 0.578]
```

### Microfacet Theory — Roughness의 물리적 기반

```
실제 표면 (마이크로 스케일):
  ┌───────────────────────────────┐
  │  거침(Rough):                 │
  │  /\/\/\/\/\/\/\/\/\/\/\/\/\   │  미세면이 무작위 방향
  │  각도가 다양 → 하이라이트 퍼짐 │
  │                               │
  │  매끄러움(Smooth):            │
  │  ─────────────────────────── │  미세면이 같은 방향
  │  같은 방향 → 날카로운 하이라이트│
  └───────────────────────────────┘
```

**GGX/Trowbridge-Reitz NDF:**
```
D(h) = α² / (π × ((n·h)² × (α²-1) + 1)²)

α = Roughness²  (제곱하는 이유: 지각적으로 선형으로 보이도록)
h = Half-Vector (조명 방향과 관찰 방향의 중간 벡터)
n = 표면 법선
```

### 2. 텍스처 해상도와 Mip Map

**해상도 표준:**

| 용도 | 권장 해상도 |
|------|------------|
| 모바일 게임 | 512×512 ~ 1024×1024 |
| 콘솔/PC 게임 | 2048×2048 ~ 4096×4096 |
| VFX/영화 | 4096×4096 ~ 8192×8192 |

**Mip Map:**
```
원본(4096):  ─→  2048  ─→  1024  ─→  512  ─→  256  ─→  ...

GPU가 카메라와의 거리에 따라 적절한 레벨 자동 선택:
  - 가까운 거리: 원본 해상도
  - 먼 거리: 저해상도 (메모리, 대역폭 절약)
  - 경계에서 선형 보간 (Linear Interpolation)
```

### 3. Albedo vs Base Color 차이

```
Albedo (순수 광학 개념):
  재질 자체의 반사율 (조명, 그림자 제거)
  AO, Specular 등 반사광 정보 미포함

Base Color (PBR 맥락):
  비금속: Albedo와 동일 (0.04~0.9 범위 권장)
  금속: F0 값 (정반사 색상)

이 둘은 비슷하지만 완전히 동일하지 않음
Base Color는 AO나 Specular 정보를 포함하지 않는 순수 재질 색상
```

### 4. Texture Compression

```
DXT1 (BC1):    4:1 압축, 불투명 텍스처 (Base Color 등)
DXT5 (BC3):    4:1 압축, 알파 채널 포함
BC5:           법선 맵 전용 (RG 채널만 저장, B는 재계산)
BC7:           고품질 범용 압축
ASTC:          모바일 플랫폼 표준

GPU는 압축된 포맷을 직접 디코딩 → VRAM 절약 + 대역폭 효율
```

### 5. Real-Time vs Offline 렌더링 비교

| 항목 | Real-Time (게임) | Offline (Arnold) |
|------|-----------------|-----------------|
| **목표 프레임** | 60fps+ | 상관없음 |
| **렌더링 방식** | 래스터화 + 근사 | 경로 추적 (정확) |
| **PBR 정확도** | 근사 BRDF | 수학적으로 정확한 BRDF |
| **전역 조명** | 근사 (베이크, SSAO) | 물리적으로 정확 |
| **텍스처 해상도** | 메모리 제한 | 거의 무제한 |
| **샘플링 노이즈** | 없음 | 샘플 수에 따라 노이즈 |

## 🎮 셰이더 디자이너 / 그래픽스 엔지니어 관점

### 실시간 셰이더에서 PBR 구현 관점

게임 엔진의 PBR 셰이더는 논문에서 다룬 오프라인 렌더러(Arnold)와 동일한 물리 법칙을 따르지만, 실시간 제약으로 인해 여러 근사(Approximation)를 도입한다.

**GLSL/HLSL에서 Cook-Torrance BRDF 구현 개요:**

```glsl
// Fresnel (Schlick 근사)
vec3 fresnelSchlick(float cosTheta, vec3 F0) {
    return F0 + (1.0 - F0) * pow(clamp(1.0 - cosTheta, 0.0, 1.0), 5.0);
}

// Normal Distribution (GGX)
float DistributionGGX(vec3 N, vec3 H, float roughness) {
    float a = roughness * roughness;
    float a2 = a * a;
    float NdotH = max(dot(N, H), 0.0);
    float NdotH2 = NdotH * NdotH;
    float denom = (NdotH2 * (a2 - 1.0) + 1.0);
    return a2 / (PI * denom * denom);
}

// Geometry (Smith's Schlick-GGX)
float GeometrySmith(vec3 N, vec3 V, vec3 L, float roughness) {
    float NdotV = max(dot(N, V), 0.0);
    float NdotL = max(dot(N, L), 0.0);
    float ggx1 = GeometrySchlickGGX(NdotV, roughness);
    float ggx2 = GeometrySchlickGGX(NdotL, roughness);
    return ggx1 * ggx2;
}

// 최종 PBR 조명 계산
vec3 Lo = vec3(0.0);
// ... 광원별 기여 합산
vec3 numerator = NDF * G * F;
float denominator = 4.0 * max(dot(N, V), 0.0) * max(dot(N, L), 0.0) + 0.0001;
vec3 specular = numerator / denominator;
```

### 텍스처 샘플링 최적화

PBR에서는 여러 텍스처(Base Color, Metallic, Roughness, Normal, AO)를 샘플링하므로 텍스처 대역폭이 병목이 될 수 있다.

**채널 패킹(Channel Packing) 기법:**

```
일반적인 패킹 방법 (ORM 텍스처):
  R 채널: AO (Ambient Occlusion)
  G 채널: Roughness
  B 채널: Metallic

장점:
  - 3개 텍스처 → 1개 텍스처로 통합
  - VRAM 절약 (3배)
  - 텍스처 유닛 사용량 감소 (샘플러 한계 주의)
  - 캐시 히트율 향상 (같은 좌표를 함께 샘플링)

주의:
  - AO는 sRGB 아닌 Linear 공간으로 샘플링해야 함
  - 패킹 규칙을 팀 내에서 문서화 필수
```

### Normal Map을 셰이더에서 올바르게 처리하기

```glsl
// Tangent Space에서 World Space로 Normal 변환
vec3 normalFromMap = texture(normalMap, TexCoords).rgb;
normalFromMap = normalFromMap * 2.0 - 1.0;  // [0,1] → [-1,1]

// TBN 매트릭스 구성 (탄젠트, 바이탄젠트, 법선)
vec3 T = normalize(TangentWS);
vec3 B = normalize(BitangentWS);
vec3 N = normalize(NormalWS);
mat3 TBN = mat3(T, B, N);

// 탄젠트 공간 법선 → 월드 공간 법선
vec3 worldNormal = normalize(TBN * normalFromMap);
```

### 절차적 생성을 셰이더에서 직접 구현하기

Substance Painter의 Generator가 텍스처를 굽는(Bake) 것과 달리, **셰이더에서 실시간으로 절차적 생성**도 가능하다:

```glsl
// 실시간 엣지 웨어 효과 (Curvature 근사)
float curvature = length(fwidth(worldNormal));  // 법선의 변화율 = 곡률 근사
float edgeWear = smoothstep(0.1, 0.5, curvature);

// 실시간 Dirt 효과 (AO 근사 + 중력 방향)
float dirtMask = (1.0 - ao) * clamp(-dot(worldNormal, vec3(0, 1, 0)), 0.0, 1.0);

// 결합
vec3 finalAlbedo = mix(baseAlbedo, metalAlbedo, edgeWear);
finalAlbedo = mix(finalAlbedo, dirtAlbedo, dirtMask * dirtAmount);
```

이 방법의 장점: VRAM 절약, 실시간 파라미터 변경 가능
단점: GPU 연산량 증가, Substance Painter 수준의 정교함 한계

### IBL (Image-Based Lighting)과의 연동

Arnold는 HDR 환경맵을 사용한 IBL로 전역 조명을 시뮬레이션한다. 실시간 엔진도 사전 계산된 IBL을 사용한다:

```
IBL 구성 요소:
  1. Irradiance Map (Diffuse IBL):
     환경의 모든 방향에서 오는 난반사 기여
     구면 조화 함수(Spherical Harmonics)로 근사 (9개 계수)

  2. Pre-filtered Environment Map (Specular IBL):
     Roughness별 여러 Mip Level에 사전 필터링
     Roughness = 0 → 선명한 반사 (Mip 0)
     Roughness = 1 → 흐릿한 반사 (높은 Mip Level)

  3. BRDF LUT (Look-Up Texture):
     Split-Sum 근사에 사용
     2D 텍스처에 NdotV와 Roughness에 따른 BRDF 스케일/바이어스 사전 계산
```

## 🎓 대학원생 수준 핵심 포인트

### 1. 에너지 보존 — PBR의 철학적 기반

```
물리 법칙: 반사된 총 에너지 ≤ 입사된 총 에너지

수식적 표현:
  ∫ fr(ωi, ωo) cos(θi) dωi ≤ 1  (모든 반사 방향에 대한 적분)

PBR에서의 구현:
  kS = Fresnel 계수 (정반사 비율)
  kD = (1 - kS) × (1 - metallic)  (난반사 비율)

  kS + kD ≤ 1  이 조건을 항상 만족해야 물리적으로 올바름
  금속(metallic=1): kD = 0 → 난반사 없음
```

### 2. Gamma Correction과 Linear Workflow

그래픽스 파이프라인에서 가장 흔한 실수 중 하나:

```
인간 눈의 감마 반응: 어두운 영역에 민감
모니터 출력: sRGB 공간 (γ ≈ 2.2)

작업 공간:
  텍스처 저장: sRGB (눈에 맞게 인코딩)
  셰이더 계산: Linear (물리 계산은 선형 공간에서 올바름)
  최종 출력: sRGB (모니터 표시를 위해 다시 인코딩)

변환:
  sRGB → Linear: value ^ 2.2 (샘플링 시)
  Linear → sRGB: value ^ (1/2.2) (출력 시, Gamma Correction)
```

**Base Color 텍스처**: sRGB 공간으로 저장 → 셰이더에서 Linear로 변환 후 사용
**Metallic/Roughness/AO**: Linear 공간으로 저장 → 변환 없이 직접 사용

### 3. Mikkt Space — 산업 표준 탄젠트 공간

법선 맵의 탄젠트 공간 계산 방법이 다르면 아티팩트가 발생한다. Morten Mikkelson이 제안한 **MikkTSpace**가 Substance Painter, Blender, Unreal Engine의 표준이다.

```
Maya와 다른 툴에서 탄젠트 공간 계산이 다를 수 있음
→ Substance Painter에서 베이킹할 때 "Compute Tangent Space Per Fragment" 옵션 주의
→ 타겟 렌더러와 동일한 탄젠트 공간 사용 필수
```

### 4. Metallic 워크플로우의 함정 — sRGB vs Linear F0

```
금속의 F0 값은 Base Color에서 읽어오는데:
  Base Color 텍스처: sRGB 공간으로 저장
  F0는 Linear 값이어야 함

셰이더에서 Base Color를 Linear로 변환 후 F0로 사용해야 정확한 결과
이를 놓치면 금속 표면의 정반사 색상이 부정확해짐
```

### 5. Height Map vs Normal Map 선택 기준

```
Normal Map:
  - GPU 비용: 낮음 (법선 벡터 직접 치환)
  - 실루엣 변화: 없음
  - 사용: 실시간, 법선만 변경

Height Map (Displacement):
  - GPU 비용: 높음 (테셀레이션 필요)
  - 실루엣 변화: 있음 (실제 메시 변형)
  - 사용: 오프라인/고품질 렌더링, 시뮬레이션

Parallax Mapping (게임 근사):
  - 실루엣 변화: 없음
  - 시차 효과로 깊이감 부여
  - 중간 비용의 타협안 (POM, Relief Mapping)
```

### 6. 절차적 생성의 확장 방향 — 딥러닝 기반 텍스처링

논문에서 향후 발전 방향으로 언급한 내용의 심화:

```
1. Neural Texture (NeRF 기반):
   카메라 포즈 → 뉴럴 네트워크 → 색상/밀도 예측
   전통적 텍스처 없이 3D 씬 표현

2. Text-to-Texture (DALL-E, Stable Diffusion):
   텍스트 프롬프트 → 3D 오브젝트 텍스처 자동 생성
   예) "rusty iron surface with scratches"

3. Material Acquisition (실제 물체 → PBR 파라미터):
   스마트폰 카메라로 촬영 → AI 분석 → Base Color/Metallic/Roughness 자동 추출
   Substance Sampler가 이 방향으로 발전
```

## 💻 컴퓨터 하드웨어 연결

### GPU 아키텍처와 텍스처 처리

#### 텍스처 캐시 (Texture Cache)

GPU의 텍스처 샘플링은 텍스처 캐시를 통해 최적화된다:

```
계층 구조:
  L1 텍스처 캐시 (수 KB, 픽셀 셰이더별)
  L2 텍스처 캐시 (수 MB, GPU 전체 공유)
  VRAM (GDDR6/HBM, 수 GB)
  시스템 RAM (느림, 반드시 VRAM에 올려야 함)

PBR에서의 영향:
  여러 텍스처(Base Color, Metallic, Roughness, Normal, AO)를
  같은 UV 좌표에서 샘플링 → 공간적 지역성(Spatial Locality) 우수

채널 패킹의 캐시 이점:
  ORM 텍스처(AO+Roughness+Metallic 합친 것)를 한 번 샘플링으로 3개 값 획득
  캐시 라인을 효율적으로 사용
```

#### VRAM 대역폭과 텍스처 대역폭

```
현대 GPU VRAM 대역폭:
  RTX 4090: ~1TB/s (GDDR6X)
  RX 7900 XTX: ~960GB/s (GDDR6)

텍스처 대역폭 계산 예시:
  4K 텍스처(4096×4096) × 4바이트(RGBA) = 64MB
  60fps × 5개 텍스처 × 64MB = 19.2GB/s
  → 최신 GPU의 VRAM 대역폭의 약 2% (여유 있음)

텍스처 압축의 효과:
  BC7 압축: 약 4:1 → 실제 VRAM 사용량과 대역폭 1/4로 감소
```

#### GPU 텍스처 유닛 (Texture Mapping Unit, TMU)

```
텍스처 샘플링은 TMU가 전담:
  RTX 4090: 304개 TMU

TMU 병렬 처리:
  여러 텍스처를 동시에 샘플링 가능
  PBR 다중 텍스처도 파이프라인을 통해 병렬 처리

Anisotropic Filtering의 TMU 비용:
  16x AF → 최대 16번 샘플링 → TMU 부하 증가
  하지만 비스듬한 각도의 텍스처 품질 대폭 향상
```

### 메모리 레이아웃 — Tiled vs Linear

```
Linear Layout (CPU 메모리):
  픽셀이 행(Row)별로 순서대로 저장
  │ Row 0 │ Row 1 │ Row 2 │ ...

  2D 공간적 인접성 없음 → 캐시 미스 발생

Tiled Layout (GPU 텍스처):
  픽셀을 타일(예: 4x4 블록) 단위로 저장
  ┌──┬──┐
  │타 │타 │  ← 2D 공간적으로 인접한 픽셀이 캐시 라인에 함께 존재
  │일 │일 │
  └──┴──┘

  인접 픽셀 접근 시 캐시 히트율 대폭 향상
  → 텍스처 필터링(Bilinear, Trilinear)에서 성능 이점
```

### Arnold 오프라인 렌더링과 CPU/GPU

```
CPU 경로 추적 (전통적 Arnold):
  다수의 CPU 코어로 픽셀 병렬 처리
  각 픽셀을 독립적으로 Monte Carlo 경로 추적
  메모리 계층: L1/L2/L3 캐시 → RAM

GPU 경로 추적 (Arnold GPU / OptiX):
  RT Core (레이 트레이싱 전용 하드웨어) 활용
  BVH (Bounding Volume Hierarchy) 가속 구조
  VRAM에 씬 데이터 + 텍스처 모두 올려야 함

  PBR 텍스처가 많을수록 VRAM 요구량 급증:
  4K × 5맵 × BC7 압축 = 약 80MB/오브젝트
  100개 오브젝트 → 8GB+ VRAM 필요
```

### Substance Painter의 GPU 활용

```
Baking 과정:
  High Poly → Low Poly 방향으로 광선 발사
  GPU의 레이 트레이싱 기능 활용 (RTX GPU에서 가속)

  Normal, AO, Curvature, Position, Thickness 맵을 GPU에서 병렬 베이킹

실시간 뷰포트:
  완전한 PBR 셰이더 실시간 미리보기
  IBL + 직접 조명 + 셰도우를 실시간으로 계산
  변경 사항을 즉시 반영 (레이어 수정 → GPU에서 재계산)
```

### Q1. Metallic 값이 0.5인 재질은 어떤 재질인가?

**A:** 물리적으로 순수한 재질은 금속(1) 또는 비금속(0)이다. Metallic 0.5는 물리적으로 정확한 재질이 아니며, **도금이 부분적으로 벗겨진 표면**이나 **산화된 금속** 같은 전환 영역을 표현할 때만 제한적으로 사용해야 한다. 특정 부분에서 금속과 비금속이 공존하는 재질(예: 녹슨 철)은 일반적으로 텍스처로 금속 부분(흰색)과 비금속 부분(검정)을 명확히 구분하는 것이 올바른 접근이다.

### Q2. Normal Map의 파란색이 지배적인 이유는?

**A:** Normal Map의 파란색(B 채널)은 법선의 Z 성분(표면에서 수직 방향)을 나타낸다. 평평한 표면에서 법선은 완전히 위를 향하므로 Z 성분이 1.0이고, 이를 0~1 범위로 인코딩하면 1.0이 되어 파란색(RGB: 128, 128, 255)으로 저장된다. 요철이 있는 부분만 R과 G 채널(X, Y 성분)이 변하므로, 대부분의 Normal Map은 파란색이 지배적으로 보인다.

### Q3. AO를 베이킹하는 것과 실시간 SSAO(Screen Space Ambient Occlusion)의 차이는?

**A:** 베이킹된 AO는 메시의 기하학에서 사전 계산된 정보로, 오브젝트 자체의 틈새나 구석에서의 차폐를 정확하게 표현하지만 **동적 오브젝트 간의 AO는 표현하지 못한다**. SSAO는 카메라 공간의 깊이 버퍼(Depth Buffer)를 분석해 실시간으로 오클루전을 계산하므로 **동적 오브젝트에도 적용**되지만, 화면 공간(Screen Space)에서만 계산하므로 카메라에 보이지 않는 영역의 AO는 부정확하다. 최상의 결과를 위해 **베이킹된 AO + SSAO를 함께 사용**하는 것이 일반적이다.

### Q4. Substance Painter에서 베이킹 실패/아티팩트가 발생하는 주요 원인과 해결책은?

**A:**

| 원인 | 증상 | 해결책 |
|------|------|--------|
| Low Poly가 High Poly 밖으로 튀어나옴 | Normal Map에 검은 얼룩 | Low Poly를 High Poly 내부에 완전히 넣거나 케이지 조정 |
| Max Ray Distance 너무 작음 | 일부 영역 베이킹 누락 | Max Distance 증가 |
| 스무딩 그룹 불일치 | 경계에서 법선 아티팩트 | High Poly와 Low Poly의 스무딩 그룹 동일하게 설정 |
| UV 겹침 | 엉뚱한 부분이 베이킹됨 | UV 언래핑 재작업, UDIM으로 분리 |
| 탄젠트 공간 불일치 | Normal Map이 다른 뷰어에서 다르게 보임 | 타겟 엔진의 탄젠트 공간 설정 확인 |

### Q5. Height Map을 Maya Arnold에서 사용할 때 Displacement Shader에 연결하는 이유는?

**A:** Height Map은 법선(방향)이 아닌 **실제 버텍스의 높이 변위**를 정의한다. 이를 표현하려면 메시를 실제로 변형(Displacement)해야 하므로, 셰이더의 색상/반사 채널이 아닌 **Displacement 채널**에 연결해야 한다. Arnold에서는 `aiDisplacement` 노드를 통해 Height Map을 셰이딩 그룹의 Displacement Shader에 연결한다. 정상적으로 작동하려면 메시에 **테셀레이션(Subdivision)**이 적용되어 있어야 하며, Arnold의 경우 렌더 타임에 자동으로 테셀레이션한다.

### Q6. 절차적 텍스처링이 AI 생성 텍스처와 다른 점은?

**A:** 절차적 텍스처링(Procedural Texturing)은 **수학적 알고리즘과 물리적 법칙**에 기반해 텍스처를 생성한다. 메시의 기하학 정보(Curvature, AO, Position)를 입력으로 받아 규칙에 따라 출력한다. 결과가 **예측 가능하고 파라미터로 완전히 제어**된다. 반면 AI 생성 텍스처(Stable Diffusion, DALL-E 기반)는 학습 데이터의 패턴을 학습해 새로운 이미지를 생성하므로, 창의적이고 사실적인 결과를 빠르게 얻을 수 있지만 **물리적 일관성이 보장되지 않고 파라미터 제어가 어렵다**. 최근에는 두 접근법을 결합 — AI가 스타일을 생성하고 PBR 알고리즘이 물리적 정확성을 보완 — 하는 방향으로 발전하고 있다.

### Q7. 이 논문의 워크플로우를 게임 개발에 직접 적용한다면 무엇을 바꿔야 하는가?

**A:**

| 항목 | 논문 (오프라인/VFX) | 게임 파이프라인 |
|------|---------------------|----------------|
| 렌더러 | Arnold (경로 추적) | Unreal/Unity (래스터화) |
| 텍스처 해상도 | 4K+ | 2K 이하 (메모리 제한) |
| 텍스처 포맷 | PNG/EXR | DXT/BC 압축 |
| AO 처리 | 베이킹만 사용 | 베이킹 AO + SSAO |
| Displacement | Subdivision + Height Map | Parallax Mapping (근사) |
| Normal Map 포맷 | OpenGL | 엔진에 따라 확인 필요 |
| 익스포트 프리셋 | Arnold (aiStandard) | Unreal/Unity 전용 프리셋 |
| LOD | 없음 | LOD 체계 + HLOD 필요 |

> [!summary] 논문 요약 핵심
> **물리 기반 렌더링(PBR)** 의 에너지 보존 법칙을 기반으로, **Substance Painter의 절차적 생성(Procedural Generation)** — 특히 Generator(Metal Edge Wear, Dirt)를 활용한 자동 마스크 생성 — 이 수작업 대비 물리적으로 더 정확하고 재사용 가능하며 파이프라인 확장성이 뛰어남을 실증적으로 보였다. Metallic/Roughness 워크플로우 → Substance Painter → Maya aiStandardSurface → Arnold 렌더링으로 이어지는 파이프라인이 현대 3D 콘텐츠 제작의 효과적인 표준임을 확인했다.
