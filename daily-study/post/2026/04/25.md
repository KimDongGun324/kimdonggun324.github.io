### 1-9. 기본적인 서식 맞추기 — 사람을 위한 코드

#### 렉서(Lexer)는 공백을 무시하지만 사람은 무시하지 않는다

C++ 컴파일러의 렉서(Lexer)는 소스 코드를 토큰으로 분리할 때 대부분의 공백을 무시한다. `int x=5;`와 `int x = 5;`는 토큰 관점에서 동일하다. 그러나 코드는 6개월 후의 나 자신과 동료를 위해 써야 한다.

```cpp
// 나쁜 서식 — 컴파일은 되지만 읽을 수 없다
int x=5;if(x>3){cout<<"hi";}

// 좋은 서식 — 의도가 즉시 보인다
int x = 5;
if (x > 3)
{
    cout << "hi";
}
```

#### 핵심 규칙 4가지

| 항목 | 규칙 | 이유 |
|------|------|------|
| **인덴팅** | 스페이스 4개 또는 탭 1개 (팀 내 통일) | 스코프 깊이를 시각화 |
| **중괄호** | Allman(새 줄) 또는 K&R(같은 줄) 중 하나로 통일 | 혼재 시 git diff 노이즈 |
| **줄 길이** | 80~120자 이내 권장 | 화면 분할·코드 리뷰 용이 |
| **하드코딩** | 매직 넘버 금지 → `const` / `constexpr` 상수 | 의미 불명, 변경 시 누락 위험 |

### 1-12. 헤더 가드가 필요한 이유 — 빌드 파이프라인의 핵심

#### Declaration vs Definition — 링킹 에러의 원인

C++ 빌드는 **전처리기 → 컴파일러 → 링커** 3단계로 이루어진다.

```
main.cpp  --[컴파일러]--> main.obj
add.cpp   --[컴파일러]--> add.obj   ← 프로젝트에서 Remove하면 이 단계가 생략
                          ↓
          [링커] main.obj + add.obj --> main.exe
          add.obj 없으면 → LNK2019: unresolved external symbol
```

- **컴파일러** 는 선언(Declaration)만 보고도 번역을 완료한다 — "이 함수가 나중에 연결되겠지"
- **링커** 가 실제 정의(Definition)를 찾지 못하면 LNK2019 링킹 에러 발생
- 컴파일 에러가 아니라 링킹 에러라는 점이 헷갈리는 핵심이다

#### `#include`는 텍스트 복사-붙여넣기

`#include`의 정체는 전처리기의 **텍스트 인클루전**이다. 이 사실이 중복 포함 문제의 뿌리다.

```
// 포함 구조
main.cpp → #include "mywork.h"  (mywork.h 내부에서 add.h를 include)
main.cpp → #include "add.h"     (직접 한 번 더)

// 전처리 후 main.cpp에 add 함수 본체가 두 번 삽입됨
// → ODR(One Definition Rule) 위반 → LNK2005: already defined
```

#### 해결 방법 1 — `#ifndef` 헤더 가드

```cpp
// add.h
#ifndef MY_ADD_H      // 매크로가 정의되지 않았다면
#define MY_ADD_H      // 지금 정의하고

inline int add(int a, int b) { return a + b; }

#endif                // 조건부 포함 종료
```

두 번째 `#include "add.h"` 시도 시 `MY_ADD_H`가 이미 정의돼 있으므로 전체 내용을 건너뛴다.

#### 해결 방법 2 — `#pragma once` (실무 권장)

```cpp
#pragma once   // 이 파일은 번역 단위당 한 번만 포함

inline int add(int a, int b) { return a + b; }
```

| 항목 | `#ifndef` 방식 | `#pragma once` |
|------|:--------------:|:--------------:|
| C++ 표준 준수 | ✅ | 비표준 (사실상 표준) |
| 코드량 | 3줄 필요 | 1줄 |
| 오타 위험 | 매크로 이름 오타 가능 | 없음 |
| 임베디드·구형 컴파일러 | 완벽 지원 | 일부 미지원 |

> [!important] ODR — One Definition Rule
> C++ 표준(§6.2): 같은 번역 단위 내에서 변수·함수·클래스는 단 하나의 정의만 가질 수 있다.
> `inline`, `constexpr`, 템플릿은 여러 번역 단위에 동일 정의가 허용되지만, 모든 정의가 **토큰 수준에서 동일**해야 한다.

### 6.4 배열과 선택 정렬 — 알고리즘 구현 첫 경험

#### 선택 정렬 핵심 아이디어

매 패스(Pass)마다 **정렬되지 않은 구간에서 최솟값을 선택**해 앞으로 가져온다.

```
초기: [ 3, 5, 2, 1, 4 ]

Pass 0: 0~4 범위에서 최솟값(1, index 3) 탐색 → swap(0, 3)
결과: [ 1, 5, 2, 3, 4 ]  ↑확정

Pass 1: 1~4 범위에서 최솟값(2, index 2) 탐색 → swap(1, 2)
결과: [ 1, 2, 5, 3, 4 ]     ↑확정

Pass 2: 2~4 범위에서 최솟값(3, index 3) 탐색 → swap(2, 3)
결과: [ 1, 2, 3, 5, 4 ]        ↑확정

Pass 3: 3~4 범위에서 최솟값(4, index 4) 탐색 → swap(3, 4)
결과: [ 1, 2, 3, 4, 5 ]           ↑확정
```

종료 조건이 `length - 1`인 이유: n개 원소에서 n-1번 패스면 정렬 완료 (마지막 1개는 자동으로 제자리).

#### Swap — 임시 변수(temp)의 역할

두 변수를 직접 덮어쓰면 한쪽 값이 사라진다. 반드시 임시 저장소가 필요하다.

```cpp
// temp 패턴 (원리 이해용)
int temp = array[a];    // (1) A 백업
array[a] = array[b];    // (2) B→A 복사
array[b] = temp;        // (3) 백업→B 복사

// 실무 권장
std::swap(array[a], array[b]);  // C++11 이상: move semantics 활용
```

#### 선택 정렬의 시간 복잡도

```
비교 횟수: (n-1) + (n-2) + ... + 1  = n(n-1)/2  → O(n²)
swap 횟수: 최대 n-1회 (고정)         → O(n)
공간 복잡도: O(1) 제자리 정렬(in-place)
```

swap 횟수가 O(n)으로 고정된다는 점이 선택 정렬의 특징이다. 원소 하나가 수백 바이트인 구조체 배열에서는 swap 비용이 크기 때문에 이 성질이 유리할 수 있다.
