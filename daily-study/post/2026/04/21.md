## 1️⃣ Today I Learned

### ✅ C++ 프로세스 메모리 구조 — 5대 세그먼트

교수님이 이번 강의에서 강조하신 핵심: *"포인터와 동적 할당이 왜 이렇게 복잡하게 만들어졌는지 — 컴퓨터 내부에서 CPU와 메모리를 어떻게 사용하는지 알면 납득이 된다."*

프로그램이 실행되면 OS는 해당 프로세스에 **가상 메모리(Virtual Memory)** 공간을 할당하고, 이를 5개의 **세그먼트(Segment)** 로 구분한다.

| 영역 | 성장 방향 | 저장 내용 |
|------|-----------|-----------|
| **Heap** | 위(↑)로 성장 | `new` / `malloc`으로 동적 할당한 데이터 |
| **Stack** | 아래(↓)로 성장 | 함수 호출 프레임 · 지역변수 · 매개변수 |
| **Data** | 고정 | 초기값이 있는 전역변수 / `static` 변수 |
| **BSS** | 고정 | 초기값이 없는 전역변수 / `static` 변수 (0 자동 초기화) |
| **Code (Text)** | 고정 | 컴파일된 기계어 명령어 |

```cpp
int g_i = 0;  // Data 세그먼트 (초기값 있음)
              // 프로그램 시작 ~ 종료까지 상주

int g_uninit;  // BSS 세그먼트 (초기값 없음, 0으로 자동 초기화)
```

> [!info] 전역변수의 생명주기
> 전역변수는 스택도 힙도 아닌 별도 세그먼트에 저장된다. 프로그램 시작 시 가장 먼저 생성되고, 종료 시 가장 늦게 사라진다.

### ✅ 스택 Stack — 함수 호출과 스택 프레임

#### 스택 프레임(Stack Frame)이란?

함수가 호출될 때마다 스택에 **스택 프레임** 이 하나씩 push되고, 함수가 `return`하면 자동으로 pop된다.

스택 프레임에는 다음이 포함된다:
- **지역변수(local variables)** — 함수 내 선언 변수
- **매개변수(parameters)** — 호출 시 전달된 인수의 복사본
- **반환 주소(return address)** — 함수 종료 후 돌아갈 명령어 위치
- **이전 프레임 포인터(saved frame pointer)** — 호출자 프레임의 기준점

#### 강의 예제 — 3단계 함수 호출 스택 변화

```cpp
#include <iostream>

int g_i = 0;         // Data 세그먼트

int second(int x)    // 매개변수 x → second()의 스택 프레임
{
    return 2 * x;
}

int first(int x)     // 매개변수 x, 지역변수 y → first()의 스택 프레임
{
    int y = 3;
    return second(x + y);
}

int main()
{
    using namespace std;
    int a = 1, b;    // a, b → main()의 스택 프레임
    b = first(a);
    cout << b << endl;  // 결과: 8 (second(1+3) = 2*4)
    return 0;
}
```

**호출 흐름에 따른 스택 변화 (LIFO — 회전초밥 접시 더미):**

```
[1] main() 시작
  Stack: [ main() | int a, b ]

[2] first(a) 호출
  Stack: [ first() | int x, int y ] ← 최상단
         [ main()  | int a, int b ]

[3] second(x+y) 호출
  Stack: [ second() | int x         ] ← 최상단
         [ first()  | int x, int y  ]
         [ main()   | int a, int b  ]

[4] second() return → pop
  Stack: [ first() | int x, int y ]
         [ main()  | int a, int b ]

[5] first() return → pop
  Stack: [ main() | int a, int b ]

[6] main() return → pop → 프로그램 종료 (OS 메모리 회수)
  Stack: (비어 있음)
```

**스택의 핵심 특성:**

| 특성 | 설명 |
|------|------|
| **자동 관리** | 함수 진입/반환 시 OS가 자동 할당/해제 |
| **빠른 속도** | SP(Stack Pointer) 레지스터 이동만으로 O(1) 처리 |
| **크기 제한** | 보통 1MB ~ 8MB (OS/설정에 따라 다름) |
| **LIFO 구조** | 나중에 push된 프레임이 먼저 pop |

### ✅ 스택 오버플로우 Stack Overflow

```cpp
int main()
{
    // int 4 bytes x 1,000,000 = 약 4MB
    // 스택 한계(보통 1~8MB) 초과 → 즉시 크래시!
    int array[1000000];  // Stack Overflow
    return 0;
}
```

> [!danger] Stack Overflow
> 스택 크기를 초과하면 프로그램이 즉시 크래시된다.
> - **Windows**: "Stack overflow" 예외
> - **Linux**: "Segmentation fault (core dumped)"

**주요 발생 원인:**

| 원인 | 설명 |
|------|------|
| **대형 지역 배열** | 스택에 수 MB짜리 배열 선언 |
| **무한 재귀** | 종료 조건 없는 재귀 → 프레임이 무한히 누적 |
| **과도하게 깊은 재귀** | 수십만 단계 재귀 호출 |

**해결: 힙으로 이동**

```cpp
// 나쁜 예 — Stack Overflow
void bad()
{
    int array[1000000];
}

// 좋은 예 — 힙 사용 (스마트 포인터)
void good()
{
    auto array = std::make_unique<int[]>(1000000);
}
```

### ✅ 힙 Heap — 동적 메모리 할당과 해제

#### new / delete 작동 원리 — 단계별 메모리 상태

```cpp
int main()
{
    // [1] 포인터 변수 ptr → 스택에 생성, nullptr로 초기화
    int *ptr = nullptr;

    // [2] 힙에 sizeof(int) x 1,000,000 = 약 4MB 할당
    //     OS가 연속된 빈 공간을 탐색하여 할당
    //     ptr에 힙 블록의 시작 주소 저장
    ptr = new int[1000000];

    // [3] 힙 데이터 사용
    ptr[0] = 42;

    // [4] 힙 메모리 해제 — OS에게 반납
    //     배열이면 반드시 delete[] 사용
    delete[] ptr;

    // [5] 댕글링 포인터(dangling pointer) 방지
    //     delete 이후에도 ptr은 이전 주소값을 보유 중
    //     nullptr 대입으로 해제된 주소 참조 사고 방지
    ptr = nullptr;

    return 0;
}
```

```
[2단계] ptr = new int[1000000] 후
  Stack: [ main() | int *ptr(0xABCD) ] ──→ Heap: [ ████ 4MB 할당 ████ ]

[4단계] delete[] ptr 후
  Stack: [ main() | int *ptr(0xABCD) ]      Heap: [ ? 해제됨(OS 반납) ? ]
         ↑ 아직 이전 주소 보유 — 위험 (댕글링 포인터)

[5단계] ptr = nullptr 후
  Stack: [ main() | int *ptr(null) ]        Heap: (해제 완료)
```

**new / delete 쌍 규칙 — 반드시 준수:**

| 할당 | 해제 |
|------|------|
| `new T` | `delete ptr` |
| `new T[n]` | `delete[] ptr` (대괄호 필수) |

> [!warning] 쌍을 맞추지 않으면
> `new[]`에 `delete`(대괄호 없이)를 사용하면 **미정의 동작(Undefined Behavior)** 발생 — 소멸자 미호출, 힙 메타데이터 손상 가능

**힙의 핵심 특성:**

| 특성 | 설명 |
|------|------|
| **수동 관리** | 프로그래머가 직접 `new` / `delete` 담당 |
| **대용량 가능** | 수백 MB ~ GB 단위 (물리 메모리 한도 내) |
| **위치 예측 어려움** | OS가 적합한 빈 공간을 선택 → 어디에 생길지 모름 |
| **유연한 생명주기** | `delete` 전까지 메모리 유지 |

### ✅ 메모리 누수 Memory Leak

힙에 할당한 메모리를 **`delete` 없이 포인터를 잃어버리는 것**. 포인터가 사라지면 해당 메모리 블록에 영원히 접근할 방법이 없다.

```cpp
void initArray()
{
    int *ptr2 = new int[1000];  // 힙에 4,000 bytes 할당
    // delete[] ptr2;           // 이 줄이 없으면 누수!
}                               // ptr2는 스택 프레임과 함께 소멸
                                // 힙의 4,000 bytes는 접근 불가 상태로 남음

int main()
{
    initArray();  // 반복 호출 시 메모리가 점점 쌓임
    return 0;
}
```

```
[initArray() 반환 후]
  Stack: [ main() ]  ← ptr2 소멸
  Heap:  [ ████ 4,000 bytes ████ ]  ← 누수! 영원히 해제 불가
```

> [!danger] 메모리 누수의 위험성
> 게임·서버 등 장기 실행 프로그램에서 누수가 반복되면 가용 메모리가 점점 줄어들고, 결국 **Out of Memory → 크래시** 로 이어진다. 다른 프로그램의 메모리까지 잠식한다.

**C++11 이후 권장 해결법 — RAII + 스마트 포인터:**

```cpp
#include <memory>

void safeInitArray()
{
    // unique_ptr: 스코프 종료 시 소멸자가 자동으로 delete[] 호출
    auto ptr2 = std::make_unique<int[]>(1000);
    // 함수 종료 → 소멸자 자동 실행 → 누수 없음
    // 예외 발생 시에도 안전하게 해제됨
}
```

**스마트 포인터 3종:**

| 종류 | 소유권 | 사용 시기 |
|------|--------|-----------|
| `std::unique_ptr` | 단독 소유 | 가장 일반적인 힙 관리 (권장) |
| `std::shared_ptr` | 공유 소유 (참조 카운팅) | 여러 곳에서 소유권 공유 시 |
| `std::weak_ptr` | 비소유 관찰 | `shared_ptr` 순환 참조 방지 |

### ✅ 그래픽스 엔지니어 셰이더 디자이너 관점

#### GPU 레지스터 파일과 스택의 유사성 — 레지스터 스필링(Register Spilling)

GPU에는 CPU의 스택에 해당하는 개념이 없다. 대신 SIMT 아키텍처에서 각 스레드(레인, lane)마다 **레지스터 파일(Register File)** 이 할당된다. GLSL/HLSL 셰이더의 지역변수는 GPU 레지스터에 저장되며, 레지스터가 부족해지면 느린 로컬 메모리(GPU의 힙에 해당)로 **스필(spill)** 된다. 이 순간 성능이 급락하므로, 셰이더 내 불필요한 지역변수 선언을 최소화하고 컴파일러의 레지스터 사용량 리포트를 모니터링하는 것이 실무에서 중요하다.

#### VRAM 누수 — CPU 메모리 누수와 동일한 원리

그래픽 리소스(텍스처, 버퍼, 메시)는 CPU의 `new`처럼 VRAM에 동적으로 할당된다.

| CPU | GPU (OpenGL) | GPU (Vulkan/DX12) |
|-----|-------------|-------------------|
| `new` | `glGenBuffers()` + `glBufferData()` | `vkAllocateMemory()` |
| `delete[]` | `glDeleteBuffers()` | `vkFreeMemory()` |

`delete`를 빠뜨리면 CPU에 메모리 누수가 생기듯, `glDeleteBuffers()`를 빠뜨리면 **VRAM 누수**가 발생한다. 씬 전환 시 반드시 해제하지 않으면 VRAM이 점점 증가하다 드라이버 크래시로 이어진다. Modern API(Vulkan, DX12)에서는 자원 관리가 더욱 명시적이므로 RAII 패턴이 더욱 중요하다.

#### 유니폼 버퍼(UBO)와 메모리 정렬 — 구조체 패딩의 GPU 버전

셰이더 uniform 변수는 CPU 힙 메모리에서 GPU로 전송된다. GLSL `std140` 규칙에 따라 변수는 **16 bytes 단위로 정렬**되어야 한다. 이는 CPU 구조체 패딩(struct padding)의 GPU 버전이다. 정렬을 맞추지 않으면 셰이더가 잘못된 메모리 위치에서 값을 읽어 **시각적 아티팩트(visual artifact)** 가 발생한다. `alignas(16)` 지시자로 CPU 측 정렬을 명시해야 한다.

#### 프레임 얼로케이터 — 게임 엔진의 힙 최적화 전략

고성능 게임 엔진(Unreal, Unity DOTS)은 매 프레임 수천 개의 렌더 커맨드를 생성/소멸한다. `new`/`delete`를 그대로 쓰면 힙 단편화와 할당 오버헤드로 프레임 레이트가 불안정해진다. 해결책은 **프레임 얼로케이터(Frame Allocator)**: 프레임 시작 시 큰 블록을 한 번에 힙에서 받아두고, 프레임 내 소규모 할당은 이 블록에서 순차 처리한 뒤, 프레임 종료 시 포인터를 시작점으로 되돌린다. O(1) 할당 + 단편화 없음을 동시에 달성하는 핵심 기술이다.

#### 셰이더 재귀와 GPU의 스택 부재

GLSL 4.0 이후 재귀 함수가 문법적으로 허용되나, 대부분의 GPU 드라이버는 **실제 재귀를 지원하지 않는다**. GPU 셰이더 스테이지에는 CPU와 같은 동적 스택이 없기 때문이다. 레이 트레이싱 셰이더(DXR, Vulkan RT)에서는 `TraceRay()` 재귀가 지원되지만 `MaxRecursionDepth`로 엄격히 제한된다. 이 제한을 초과하면 GPU 드라이버 수준에서 TDR(Timeout Detection and Recovery)이 발생한다.

## 2️⃣ Key Insights

- **포인터와 동적 할당의 존재 이유는 메모리 구조에 있다** — 스택이 빠르지만 크기가 제한적이기 때문에, 대용량 데이터는 힙을 사용할 수밖에 없다. 힙을 직접 다루려면 포인터가 필요하고, 수동 관리가 불가피하다. 이 구조를 이해하면 C++의 복잡성이 납득된다

- **스택은 자동이지만 힙은 수동이다** — 스택의 지역변수는 스코프가 끝나면 자동 해제된다. 하지만 힙은 프로그래머가 직접 `delete`를 호출해야 해제된다. 이 비대칭성이 메모리 누수의 근원이며, C++11의 스마트 포인터가 해결책이다

- **`delete` 이후 nullptr 대입은 선택이 아닌 필수 습관** — `delete` 후에도 포인터는 이전 주소를 보유한다. 이 댕글링 포인터(dangling pointer)를 역참조하면 미정의 동작(UB)이 발생한다. `nullptr` 대입으로 역참조를 시도할 경우 명확한 오류로 감지 가능해진다

- **`new[]`에는 반드시 `delete[]`** — 단순 실수처럼 보이지만 내부적으로 배열의 크기 정보를 힙 메타데이터에 저장하는 구조 차이가 있다. `delete`(대괄호 없음)를 쓰면 소멸자가 첫 원소에만 호출되거나 힙 메타데이터가 손상될 수 있다

- **Stack Overflow는 크기가 아니라 한계를 모르는 데서 발생한다** — 스택이 수 MB밖에 안 된다는 사실을 모르면, 4MB짜리 로컬 배열이 왜 크래시를 내는지 이해할 수 없다. 이제는 이유를 알고 힙으로 이동하는 판단을 내릴 수 있다

- **RAII가 C++ 메모리 관리의 정답이다** — 생성자에서 할당, 소멸자에서 해제를 묶는 RAII 패턴은 예외 발생 시에도 자원 누수를 방지한다. `std::unique_ptr`이 이 패턴의 표준 구현체다

- **그래픽스 엔지니어에게 힙 관리는 두 배의 영역이다** — CPU 힙(RAII, 스마트 포인터)과 GPU VRAM(OpenGL/Vulkan 리소스 관리)을 동시에 책임져야 한다. 원리는 동일하고, 누수의 결과도 동일하게 치명적이다
