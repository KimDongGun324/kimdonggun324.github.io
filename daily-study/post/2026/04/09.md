## 1️⃣ Today I Learned

### ✅ 링키지(Linkage) — 변수가 다른 파일에 보이는가?

C++은 여러 `.cpp` 파일을 각각 `.obj`로 컴파일한 뒤 링커가 하나의 `.exe`로 합친다.
**링키지** = 이 변수/함수가 링킹 단계에서 다른 파일과 연결될 수 있는가?

| 링키지 | 키워드 예시 | 접근 범위 |
|---|---|---|
| **없음** | `int a;` (함수 안) | 선언된 블록 `{}` 안 |
| **내부 연결** | `static int g_x;` / `const int g_x = 1;` | 같은 `.cpp` 파일 안에서만 |
| **외부 연결** | `int g_x;` / `extern int g_x;` | 모든 `.cpp` 파일에서 접근 가능 |

```cpp
int g_x;              // External Linkage (전역 변수 기본값)
static int g_x;       // Internal Linkage  (이 파일 한정)
const int g_x = 1;    // Internal Linkage  (const 기본값)
extern int g_x;       // External Linkage  (다른 파일 정의 참조, 메모리 없음)
extern int g_x(1);    // External Linkage  (이 파일에서 정의 + 외부 공개)
```

### ✅ 전역 변수 & `::` 전역 스코프 연산자

지역 변수와 전역 변수의 이름이 겹치면 **이름 숨기기(Name Hiding)** 발생.
이때 `::` 로 전역 변수에 명시적으로 접근할 수 있다.

```cpp
int value = 123;    // 전역 변수

int main() {
    cout << value;    // 123 (지역 선언 전 → 전역 참조)

    int value = 1;    // 지역 변수 → 전역 value가 가려짐!

    cout << ::value;  // 123 ← :: 로 전역 강제 접근
    cout << value;    // 1   ← 지역 변수
}
```

> [!warning] 전역 변수는 가급적 사용 금지
> 어디서든 값을 바꿀 수 있어서 파일이 많아지면 추적이 불가능해진다.
> 불가피할 경우 `g_` 접두사로 이름을 명확히 할 것 (`g_playerSpeed` 등)

### ✅ 정적 변수 (Static Duration Variable)

함수 내부의 `static` 변수는 **딱 한 번만 초기화**되고,
함수가 종료되어도 메모리가 유지된다.

```cpp
void doSomething() {
    static int a = 1;  // 최초 1회만 초기화, 이후 호출 시 건너뜀
    ++a;
    cout << a << endl;
}

int main() {
    doSomething();  // 출력: 2
    doSomething();  // 출력: 3
    doSomething();  // 출력: 4
    doSomething();  // 출력: 5
}
```

> [!info] "Static" = 메모리가 정적으로 고정된다는 뜻
> 일반 지역 변수: 함수 호출마다 메모리 할당 → 반납 반복
> static 변수: 프로그램 시작 시 메모리 고정 → 종료까지 같은 주소 유지
>
> **실용 사용처**: 함수 호출 횟수 카운팅(디버깅), 고유 ID 발급, 싱글톤 패턴

### ✅ extern 키워드 — 여러 파일 간 변수 공유

```cpp
// test.cpp — 변수를 "정의"하는 파일 (메모리 할당)
extern int a = 123;

void doSomething() { cout << "In test.cpp" << endl; }
```

```cpp
// main.cpp — 변수를 "사용"하는 파일
void doSomething();  // forward declaration (함수 몸체는 test.cpp에 있음)
extern int a;        // 선언만! 메모리는 test.cpp에 있음

int main() {
    doSomething();
    cout << a << endl;  // 123 출력 (test.cpp의 a를 링킹으로 연결)
}
```

> [!tip] 선언(Declaration) vs 정의(Definition)
> - **선언**: 존재를 알림, 메모리 없음 → `extern int a;`
> - **정의**: 메모리 할당 → `extern int a = 123;` 또는 `int a = 123;`
> - extern 변수는 **딱 하나의 파일**에서만 정의해야 함. 두 곳에서 정의하면 LNK2005 에러!

### ✅ extern const 패턴 — 상수를 효율적으로 여러 파일에 공유

**문제**: 헤더에 `const double pi = 3.14;` 를 쓰면 include한 파일마다 사본이 생긴다.

```cpp
// ❌ 비효율적: 파일마다 pi 복사본이 생김 (주소가 다름)
// My_Constants.h
namespace Constants { const double pi(3.141592); }
```

**해결**: 헤더엔 선언만, `.cpp`에서 딱 한 번 정의

```cpp
// ✅ My_Constants.h — 선언만
#pragma once
namespace Constants {
    extern const double pi;
    extern const double gravity;
}

// ✅ My_Constants.cpp — 딱 한 번 정의 (메모리 1개)
#include "My_Constants.h"
namespace Constants {
    extern const double pi(3.141592);
    extern const double gravity(9.8);
}
```

```
결과:
  &Constants::pi in main.cpp → 0x00A1
  &Constants::pi in test.cpp → 0x00A1  ← 같은 주소! 메모리 공유 ✅
```

### ✅ 링킹 에러(LNK Error) 핵심 2종

| 에러 | 원인 | 해결 |
|---|---|---|
| **LNK2019** Unresolved External | `extern` 선언은 있는데 어디에도 정의(초기화)가 없음 | 하나의 `.cpp`에서 정의 추가 |
| **LNK2005** Multiple Definition | 같은 `extern` 변수를 두 파일에서 모두 초기화 | 한 파일에서만 초기화, 나머지는 선언만 |

> [!warning] 링킹 에러가 나면 → Output 창부터!
> 컴파일 에러 목록이 아닌 **Output 창**에서 `LNK????` 번호를 확인한다.
> 컴파일 단계에서는 통과하고 링킹 단계에서 터지기 때문에 더 찾기 어렵다.

## 2️⃣ Key Insights

- **링키지는 빌드 과정의 언어** — `.cpp` → `.obj` → `.exe` 흐름을 모르면 Internal/External이 추상적으로 느껴진다
- **`static`의 핵심은 메모리** — "정적"이란 말이 의미하는 것은 메모리 주소가 고정된다는 것. 나중에 동적 할당을 배우면 대비 개념으로 확 와닿는다
- **`extern int a;`와 `extern int a = 1;`은 전혀 다르다** — 전자는 선언(메모리 없음), 후자는 정의(메모리 있음)
- **전역 변수의 진짜 위험** — 값이 어디서 바뀌는지 추적 불가. 파일이 수십 개 넘어가는 순간 디버깅 지옥
- **`extern const` 패턴** — 게임 엔진에서 수백 개 파일이 물리 상수를 공유할 때 메모리를 아끼는 실전 패턴
