### ✅ [Python] 딕셔너리 자료형 (02-5)

#### 딕셔너리란 — 세 가지 이름의 동일한 개념

강의에서 사물함(locker) 비유를 통해 딕셔너리를 직관적으로 설명했다. 사물함 번호(Key)를 알면 안의 내용(Value)을 바로 꺼낼 수 있는 것처럼, 딕셔너리는 **Key로 Value를 O(1) 시간에 조회**하는 자료구조다.

| 명칭 | 관점 | 설명 |
|------|------|------|
| **딕셔너리 (Dictionary)** | 파이썬 공식 | 단어로 뜻을 찾는 사전처럼 Key로 Value를 찾음 |
| **연관 배열 (Associative Array)** | CS 이론 | 정수 인덱스 대신 임의 Key로 접근하는 배열 |
| **해시 테이블 (Hash Table)** | 구현 방식 | 해시 함수로 Key → 저장 위치를 O(1)에 계산 |

```python
# 기본 생성 — 중괄호 {}, key: value 쌍
a = {1: '파랑구름', 2: '노랑구름', 3: '초록구름'}

# 문자열 key — 실무에서 가장 많이 씀
dic = {'name': 'pey', 'phone': '0119993323', 'birth': '1118'}
```

**Python 딕셔너리의 핵심 특성:**
- **순서 보장**: Python 3.7+부터 삽입 순서(insertion order) 유지
- **Key 유일성**: 같은 Key 두 번 쓰면 나중 값으로 **경고 없이** 덮어씀
- **Key 불변성**: Key는 반드시 hashable(불변) 타입 — 정수, 문자열, 튜플 가능 / **리스트 불가**
- **O(1) 조회**: 리스트의 O(n) 선형 탐색과 달리 해시 테이블 덕분에 평균 상수 시간 보장

#### 핵심 메서드 — items / keys / values

강의 코드에서 주석 처리(`#`)되어 있던 세 메서드가 딕셔너리 활용의 핵심이다.

```python
a = {1: '파랑구름', 2: '노랑구름', 3: '초록구름'}

print(a.items())   # dict_items([(1, '파랑구름'), (2, '노랑구름'), (3, '초록구름')])
print(a.keys())    # dict_keys([1, 2, 3])
print(a.values())  # dict_values(['파랑구름', '노랑구름', '초록구름'])
```

세 메서드가 반환하는 것은 복사본이 아닌 **"뷰(View)"** 다 — 원본 딕셔너리를 실시간으로 반영하는 창(window). 딕셔너리가 바뀌면 뷰도 즉시 반영된다. 리스트로 쓰려면 `list(a.keys())`로 명시적 변환이 필요하다.

#### for 루프와 딕셔너리 순회 — 언패킹(Unpacking)

강의의 핵심 코드 패턴:

```python
a = {1: '파랑구름', 2: '노랑구름', 3: '초록구름'}

for k, v in a.items():
    print("키는 : " + str(k))   # int key → str() 변환 필수, 안 하면 TypeError
    print("값은 : " + v)
```

`for k, v in a.items():`의 작동 원리:
1. `a.items()`가 `(1, '파랑구름')`, `(2, '노랑구름')` ... 같은 튜플 시퀀스 반환
2. 각 루프 차례에서 튜플이 자동으로 **언패킹** → 첫 원소가 `k`, 둘째가 `v`에 할당
3. `str(k)` — key가 정수(int)이므로 문자열 연결 시 명시적 타입 변환 필수

**4가지 순회 패턴 정리:**

```python
# 패턴 1: key만 (파이써닉, 기본)
for k in a:
    print(k)          # 1, 2, 3

# 패턴 2: value만
for v in a.values():
    print(v)          # '파랑구름', '노랑구름', '초록구름'

# 패턴 3: key + value 동시 (강의 핵심 패턴)
for k, v in a.items():
    print(k, v)

# 패턴 4: 인덱스 + key (enumerate 응용)
for i, k in enumerate(a.keys()):
    print(i, k)
```

#### key로 Value 얻기 — 직접 인덱싱 vs get()

```python
a = {1: '파랑구름', 2: '노랑구름', 3: '초록구름'}

# 직접 인덱싱 — 없는 key 접근 시 KeyError로 프로그램 중단
print(a[1])    # '파랑구름'
# print(a[5])  # KeyError: 5

# get() — 없으면 None 반환, 프로그램 계속 실행
print(a.get(5))              # None
print(a.get(5, '없는 구름')) # '없는 구름' — 기본값 지정 가능
```

| 상황 | 권장 방법 |
|------|-----------|
| key가 반드시 존재한다고 확신 | `a[key]` — 명확한 에러 발생이 오히려 좋음 |
| key가 없을 수도 있는 경우 | `a.get(key)` |
| API 응답 파싱, 필드 누락 가능성 | `a.get(key, 기본값)` 필수 |

#### key 존재 확인 — in 연산자

```python
a = {1: '파랑구름', 2: '노랑구름', 3: '초록구름'}

print(4 in a)   # False — key 4 없음
print(1 in a)   # True  — key 1 있음

# not in 도 사용 가능
print(4 not in a)   # True

# 중요: in은 key 집합만 검색, value 검색은 .values() 명시 필요
print('파랑구름' in a)           # False — key 중에 없음
print('파랑구름' in a.values())  # True  — value 중에 있음
```

`in` 연산자는 내부적으로 해시 테이블을 탐색하므로 **O(1)** — 리스트의 `in`이 O(n)인 것과 대조된다. 원소가 100만 개인 리스트 vs 딕셔너리에서 `in`을 쓰면 딕셔너리가 압도적으로 빠르다.

#### API와 JSON — 교수님이 강조한 실무 핵심

> *"서버에 요청을 보냈을 때 서버에서 응답이 오는 게 JSON 형태로 많이 들어온다. 그래서 이거를 어떻게 활용하느냐가 굉장히 많이 나온다. 그래서 딕셔너리는 꼭 알아둬야 한다."* — 강의 음성

JSON은 딕셔너리와 1:1 대응된다:

```python
import json

# API 응답(JSON 문자열) → 파이썬 딕셔너리
json_str = '{"name": "홍길동", "age": 25, "skills": ["Python", "C++"]}'
data = json.loads(json_str)   # loads: load string

# 안전한 필드 추출 패턴 — API 실무 표준
name   = data.get('name',   'Unknown')
email  = data.get('email',  '')         # 없는 필드 → 기본값
skills = data.get('skills', [])

# 중첩 딕셔너리 접근
response = {
    "가족관계": {"아버지": "홍판서", "어머니": "춘섬"},
    "특기": ["농구", "도술"]
}
print(response["가족관계"]["아버지"])   # "홍판서"
print(response["특기"][0])             # "농구"
```

**JSON ↔ Python 타입 대응:**

| JSON | Python |
|------|--------|
| `object {}` | `dict` |
| `array []` | `list` |
| `string` | `str` |
| `number` | `int` / `float` |
| `true` / `false` | `True` / `False` |
| `null` | `None` |

### ✅ [그래픽스 엔지니어 / 셰이더 디자이너 관점]

#### 머티리얼 파라미터 관리 — 딕셔너리가 곧 Material Properties

그래픽스 파이프라인에서 Material(재질)은 이름-값 쌍의 집합이다. `{"albedo": [1,0,0,1], "roughness": 0.3, "metallic": 0.0}` — 이것이 그대로 딕셔너리 구조다. Python 기반 렌더링 도구(PyOpenGL, ModernGL, wgpu-py)에서 셰이더 uniform 바인딩, render state 설정, 머티리얼 인스턴스 관리 모두 딕셔너리 + for 루프 패턴으로 자동화한다.

```python
# 셰이더 uniform 자동 바인딩 패턴 (ModernGL 스타일)
uniforms = {
    'u_time':       0.0,
    'u_resolution': [1920, 1080],
    'u_albedo':     [1.0, 0.0, 0.0, 1.0],
    'u_roughness':  0.3,
    'u_metallic':   0.0,
}

for name, value in uniforms.items():
    if name in shader.uniforms:
        shader[name].value = value  # 딕셔너리 key가 GLSL uniform 이름과 직접 대응
```

#### Houdini — kwargs 딕셔너리가 이벤트 시스템의 핵심

후디니 Python API에서 이벤트 핸들러(onCreated, onDeleted 등)는 항상 `kwargs` 딕셔너리를 인수로 받는다. `kwargs['node']`, `kwargs['parm_name']`, `kwargs['script_value']` 형태로 딕셔너리 키 접근이 상시 등장한다. Python SOP의 `geo.attribValue()`, `node.parm()` 시스템도 문자열 key로 속성에 접근하는 딕셔너리 철학을 따른다.

#### Substance Designer/Painter — 프리셋을 JSON 딕셔너리로 직렬화

Substance Designer의 Python API에서 노드 파라미터와 연결 정보는 딕셔너리로 관리된다. 커스텀 머티리얼 프리셋을 JSON 딕셔너리로 직렬화(serialize)해 재사용하고, `sd.api` 시스템이 `property name → value` 딕셔너리 매핑을 제공한다. Substance Painter에서도 `shelf.set_properties({"color": [1,0,0], "roughness": 0.5})` 형태의 딕셔너리 기반 파라미터 일괄 설정이 표준 패턴이다.

#### AOV 렌더 패스 관리 — 딕셔너리 컴프리헨션 실전

```python
# AOV(Arbitrary Output Variable) 렌더 패스 설정 관리
aov_config = {
    'beauty':  {'format': 'EXR32', 'enabled': True},
    'diffuse': {'format': 'EXR16', 'enabled': True},
    'depth':   {'format': 'EXR32', 'enabled': False},
    'normal':  {'format': 'EXR16', 'enabled': True},
}

# 활성화된 패스만 딕셔너리 컴프리헨션으로 필터링
active = {k: v for k, v in aov_config.items() if v['enabled']}

for pass_name, cfg in active.items():
    print(f"렌더링: {pass_name} ({cfg['format']})")
```

## 2️⃣ Key Insights

- **딕셔너리의 O(1) 조회는 해시 테이블 덕분** — `Key → hash(Key) → 버킷 인덱스 → Value`의 3단계 계산이 CPU 레지스터 연산으로 완결된다. 리스트가 원소 수에 비례해 느려지는 것과 달리, 딕셔너리는 원소가 100만 개여도 조회 시간이 동일하다

- **`get(key, default)` 패턴이 API 파싱의 표준** — 교수님이 강조한 JSON/API 연결이 핵심. 서버 응답 필드는 언제든 누락될 수 있으므로 `response['field']` 직접 접근은 KeyError 위험이 있다. `response.get('field', 기본값)` 패턴으로 방어적으로 파싱해야 한다

- **`in` 연산자의 대상이 key임을 명확히 이해** — `'파랑구름' in a`는 False다. value를 검색하려면 반드시 `a.values()`를 명시해야 한다. 이 구분을 놓치면 논리 버그가 발생하고 디버깅이 어렵다

- **딕셔너리 뷰(View)는 복사본이 아니다** — `a.keys()`, `a.values()`, `a.items()`는 딕셔너리 원본의 라이브 뷰를 반환한다. 원본이 바뀌면 뷰도 자동 반영된다. 스냅샷이 필요하면 `list(a.keys())`로 변환해야 한다

- **Key는 반드시 hashable(불변)이어야 한다** — 리스트를 Key로 쓰면 `TypeError: unhashable type: 'list'`가 발생한다. 좌표 쌍을 Key로 쓰려면 리스트 `[x, y]` 대신 튜플 `(x, y)`를 사용해야 한다. 셰이더 파라미터 테이블에서 텍스처 슬롯 번호나 uniform 이름을 Key로 쓸 때 이 제약이 적용된다

- **딕셔너리 컴프리헨션은 그래픽스 데이터 파이프라인의 강력한 도구** — AOV 필터링, 셰이더 uniform 선택적 바인딩, 머티리얼 파라미터 일괄 변환 등 그래픽스 파이프라인의 데이터 처리 패턴이 모두 딕셔너리 컴프리헨션 한 줄로 표현된다

- **JSON이 곧 Python 딕셔너리다** — `json.loads()`로 변환하면 그대로 딕셔너리가 된다. REST API, 씬 파일, 셰이더 설정 파일, 렌더 설정 파일이 모두 JSON 기반이므로, 딕셔너리를 완전히 이해하는 것이 모든 도구 통합의 기반이 된다
