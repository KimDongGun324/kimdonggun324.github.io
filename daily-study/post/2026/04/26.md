### 6.5 정적 다차원 배열 — Row-Major와 메모리 레이아웃

#### 2D 배열은 메모리에서 1차원이다

C++에서 `int arr[3][5]`는 논리적으로는 3행 5열이지만, 물리적 메모리는 15개의 int가 연속으로 배치된 **1차원 배열**이다.

```
논리적 표현 (3x5):         물리적 메모리 (Row-Major):
[ 0  1  2  3  4 ]   →   arr[0][0] arr[0][1] arr[0][2] arr[0][3] arr[0][4]
[ 5  6  7  8  9 ]        arr[1][0] arr[1][1] ... (행이 끝나도 메모리는 연속)
[10 11 12 13 14 ]        arr[2][0] ...
```

주소 계산 공식: `arr[row][col]` 의 주소 → `base + (row * 열수 + col) * sizeof(int)`

#### Row-Major vs Column-Major

| 언어/API | 레이아웃 | 의미 |
|----------|----------|------|
| C / C++ | Row-Major | 행 우선 — 같은 행이 메모리에 연속 |
| Fortran / OpenGL(GLSL `mat4`) | Column-Major | 열 우선 — 같은 열이 메모리에 연속 |
| DirectX(HLSL `matrix`) | Row-Major | C++과 동일 |

-> OpenGL, DX는 AI가 참고용으로 만들어줌

#### 3D 배열과 부분 초기화

```cpp
int arr3D[2][3][4] = { 0 }; // 전체 24개 원소를 0으로 초기화
// 나머지 명시되지 않은 원소는 자동으로 0으로 채워진다
```

### 6.6 C 언어 스타일의 배열 문자열 — null terminator와 cstring 함수

#### null terminator — 문자열의 끝을 표시하는 규약

C 스타일 문자열은 `char` 배열로 표현되며, 반드시 끝에 **null terminator(`'\0'`, ASCII 0)** 를 가져야 한다. 이것이 없으면 문자열 함수들이 배열 경계를 넘어 메모리를 읽는다.

```cpp
char str[6] = { 'H', 'e', 'l', 'l', 'o', '\0' }; // 직접 선언
char str2[]  = "Hello"; // 컴파일러가 자동으로 '\0' 추가 (크기 6)

// 메모리: H(72) e(101) l(108) l(108) o(111) \0(0)
```

#### 주요 cstring 함수 비교

| 함수 | 용도 | 위험도 | 안전 대안 |
|------|------|--------|-----------|
| `strcpy(dest, src)` | 문자열 복사 | 🔴 높음 | `strcpy_s(dest, SIZE, src)` |
| `strcat(dest, src)` | 문자열 이어붙이기 | 🔴 높음 | `strcat_s(dest, SIZE, src)` |
| `strcmp(a, b)` | 문자열 비교 | 🟢 안전 | — |
| `strlen(str)` | 문자열 길이 | 🟡 주의 | 루프 내 중복 호출 금지 |
| `cin.getline(buf, SIZE)` | 공백 포함 입력 | 🟢 안전 | — |

#### `strcmp` 반환값 해석

```
strcmp(source, dest) 반환값:
  0       → source와 dest가 완전히 동일
  양수(+) → source가 dest보다 사전순으로 뒤에 온다 (source > dest)
  음수(-) → source가 dest보다 사전순으로 앞에 온다 (source < dest)
```

#### 버퍼 오버플로우 — 가장 위험한 보안 취약점

```cpp
char buf[5];
strcpy(buf, "Hello World!"); // 12바이트를 5바이트 공간에 → 인접 메모리 덮어쓰기
// → 스택 스매싱(Stack Smashing), 리턴 주소 변조, 원격 코드 실행으로 이어질 수 있다
```

> [!danger] 실무 보안
> `strcpy`, `strcat` 는 현대 코드베이스에서 사용 금지 수준으로 취급된다. MSVC는 이를 경고(C4996)로 처리하며, `-D_CRT_SECURE_NO_WARNINGS` 없이는 컴파일이 중단된다. 항상 크기를 명시하는 `_s` 버전을 사용할 것.

### 6.7 포인터의 기본적인 사용법 — 주소를 다루는 변수

#### 포인터의 본질 — "주소를 담는 변수"

```cpp
int x{ 5 };

int* ptr_x = &x;   // &: 주소 연산자(address-of operator) — x의 주소를 가져옴
                   // *: 역참조 연산자(dereference operator) — 주소에 있는 값에 접근

std::cout << ptr_x;   // x의 주소 출력 (예: 00AFFD14)
std::cout << *ptr_x;  // x의 값 출력 (5) — 역참조
```

```
메모리 다이어그램:
┌─────────────────────────────────────────┐
│  x      │ 주소: 00AFFD14 │ 값: 5        │
│  ptr_x  │ 주소: 00AFFD18 │ 값: 00AFFD14 │ ← x의 주소를 값으로 가짐
└─────────────────────────────────────────┘
```

#### sizeof(포인터)는 항상 고정 크기

```cpp
int*    ptr1;  // sizeof: 4바이트(32비트) 또는 8바이트(64비트)
double* ptr2;  // sizeof: 동일 — 포인터 크기는 데이터 타입과 무관
char*   ptr3;  // sizeof: 동일

// 이유: 포인터는 "메모리 주소"를 담는 변수
//       주소의 크기는 시스템 주소 공간에 따라 고정 (32/64비트)
```

#### 포인터를 사용하는 두 가지 이유

1. **데이터 복사 없이 전달** — 크기가 큰 구조체를 함수에 전달할 때 포인터 8바이트만 복사
2. **여러 곳에서 같은 데이터 공유** — 모두 같은 주소를 가리키므로 한 곳의 수정이 전체에 반영

#### 위험: 초기화되지 않은 포인터

```cpp
int* ptr;        // 스택에 남은 쓰레기 주소가 들어감
*ptr = 5;        // 런타임 오류: "Run-Time Check Failure #3 - The variable 'ptr' is being used without being initialized."
```

> [!danger] 반드시 초기화
> 포인터 선언과 동시에 항상 `nullptr` 또는 유효한 주소로 초기화할 것. 쓰레기 주소를 역참조하면 감지가 매우 어려운 메모리 오염 버그가 발생한다.

### 6.7 널 포인터 Null Pointer — 포인터의 첫 번째 방어선

#### 세 가지 초기화 방법

```cpp
double* ptr1 = NULL;      // C 스타일 (레거시) — 정수 0 매크로
double* ptr2{ 0 };        // 정수 0 암묵적 변환 (레거시)
double* ptr3{ nullptr };  // C++11 이후 권장 — 타입 안전
```

| 방법 | 타입 | 정수 변환 | 오버로딩 안전 |
|------|------|-----------|--------------|
| `0` | `int` | 가능 | 위험 |
| `NULL` | 구현마다 다름 | 가능 | 위험 |
| `nullptr` | `std::nullptr_t` | 불가 | 안전 |

#### null 체크 패턴 — 역참조 전 반드시 검사

```cpp
void doSomething(double* ptr)
{
    if (ptr != nullptr)
    {
        std::cout << *ptr << std::endl;  // 유효한 주소만 역참조
    }
    else
    {
        std::cout << "Null ptr, do nothing" << std::endl;
    }
}
```

#### 핵심 발견 — 포인터는 값으로 전달된다

강의에서 `&ptr`을 출력해 직접 증명한 사실: 함수에 포인터를 전달하면 포인터 **변수 자체**가 스택의 새 위치에 복사된다.

```
실제 출력:
address of pointer variable in doSomething()  00F6FCA8  ← 항상 동일 (함수 스택)
address of pointer variable in doSomething()  00F6FCA8
address of pointer variable in main()         00F6FD8C  ← 다른 주소 (main 스택)
```

> [!important] 결론
> 함수 안에서 포인터 파라미터 자체를 다른 주소로 바꿔도 호출자의 포인터에는 영향이 없다. 호출자의 포인터까지 바꾸려면 **포인터의 포인터(`double**`)** 또는 **포인터 참조(`double*&`)** 를 사용해야 한다.

### 참고용! - AI가 생성해준 그래픽스 엔지니어 관점에서의 연결

이런게 있구나 정도만 확인하자

#### ① 행렬 레이아웃 불일치 — OpenGL vs DirectX

셰이더 엔지니어에게 Row-Major / Column-Major 차이는 실제 버그의 원천이다. C++에서 `glm::mat4`로 계산한 변환 행렬을 `glUniformMatrix4fv`로 GLSL에 넘길 때, 세 번째 파라미터(transpose)를 올바르게 설정하지 않으면 모든 오브젝트가 예상치 못한 위치에 그려진다. GLM은 기본적으로 Column-Major를 사용하므로 `GL_FALSE`를 전달하는 것이 맞지만, 자체 Row-Major 행렬 라이브러리를 만든다면 반드시 `GL_TRUE`로 전치해야 한다. 이 차이를 이해하는 것이 오늘 6.5에서 배운 Row-Major의 가장 직접적인 실무 활용이다.

#### ② glShaderSource — C 문자열 포인터의 포인터

`glShaderSource(shader, 1, &src, nullptr)` 에서 세 번째 파라미터 타입은 `const GLchar**`다. 이는 오늘 6.6에서 배운 C 문자열(`const char*`)의 포인터다. 셰이더 소스를 여러 개 이어붙일 때 `const char* sources[3] = { header, main_body, footer };` 처럼 배열을 만들고 `glShaderSource(shader, 3, sources, nullptr)`로 한 번에 전달한다. null terminator가 없는 char 배열을 넘기면 GL 드라이버가 메모리 끝을 넘어 읽어 정의되지 않은 동작이 발생한다.

#### ③ VBO 포인터와 오프셋 — glVertexAttribPointer

`glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, stride, (void*)0)` 의 마지막 파라미터는 `const void*` 타입이지만, VBO가 바인딩된 상태에서는 **버퍼 내 바이트 오프셋**을 의미한다. 이는 오늘 6.7 포인터 강의에서 배운 포인터의 주소 의미론이 API 설계에서 재활용된 사례다. 인터리브 버텍스 배열에서 위치(offset 0), 노멀(offset 12), UV(offset 24)를 구분할 때 이 오프셋 계산이 결정적이다.

#### ④ Vulkan 핸들과 nullptr — VK_NULL_HANDLE 패턴

Vulkan에서 `VkRenderPass`, `VkPipeline`, `VkBuffer` 등 모든 핸들은 선언 시 `VK_NULL_HANDLE`로 초기화하고, 사용 전 반드시 유효성을 확인해야 한다. 이는 오늘 6.7 널 포인터에서 배운 `if (ptr != nullptr)` 가드 패턴과 완전히 동일하다. 핸들 초기화를 빠뜨리면 유효성 검사를 통과해버려 이미 소멸된 핸들로 드로우콜을 제출하는 치명적 버그가 발생한다. 현대 Vulkan 코드베이스에서 `VK_NULL_HANDLE` 패턴은 선택이 아닌 강제 규약이다.

#### ⑤ 씬 그래프 순회와 null 종료 — 렌더러의 핵심

게임 엔진의 씬 그래프는 트리 구조로, 각 노드가 자식 노드를 포인터로 참조한다. 렌더러가 씬을 재귀적으로 순회할 때 `if (node->firstChild != nullptr)` 조건이 순회 종료를 결정한다. 포인터 초기화를 빠뜨린 채 `firstChild`에 쓰레기 주소가 남아 있으면 렌더러가 쓰레기 메모리를 노드로 해석해 순회 중 크래시가 발생한다. nullptr 초기화는 씬 그래프의 안전한 순회를 보장하는 최소 조건이다.
