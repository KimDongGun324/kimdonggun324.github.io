### ✅ [C++] using 문과 모호성 (4.3)

`using namespace`는 편리하지만, 동명 식별자가 여러 네임스페이스에 있으면 **모호성(Ambiguity)** 에러가 발생한다.

```cpp
namespace a { int my_var = 1; }
namespace b { int my_var = 2; }

using namespace a;
using namespace b;

cout << my_var; // ❌ error: 'my_var' is ambiguous
cout << a::my_var; // ✅ 명시적 지정
```

| 형태 | 예시 | 특징 |
|---|---|---|
| **네임스페이스 전체** | `using namespace std;` | 편리하지만 충돌 위험 |
| **특정 식별자만** | `using std::cout;` | 안전하고 명시적 |
| **블록 스코프 한정** | `{}` 안에서 `using` | 범위 밖으로 영향 없음 |

> [!warning] 헤더 파일에서 `using namespace` 금지
> 헤더를 include한 모든 파일에 전역 오염이 퍼진다.
> 헤더에서는 반드시 `std::cout`, `std::string` 형태로 명시적으로 작성할 것.

### ✅ [C++] auto 키워드와 자료형 추론 (4.4)

컴파일러가 초기값을 보고 자료형을 **자동 추론**. 컴파일 타임에 결정되므로 런타임 비용 없음.

```cpp
auto a = 123;       // int
auto d = 123.0;     // double
auto c = 1 + 2.0;   // double (int + double → 암시적 승격)
auto x;             // ❌ ERROR — 반드시 초기화 필요
```

**Trailing Return Type** — 함수 반환 타입을 뒤에 쓰는 문법 (C++11~)

```cpp
// 기존 방식
int add(int x, int y) { return x + y; }

// trailing return type
auto add(int x, int y) -> int { return x + y; }
```

> [!info] 함수 파라미터에는 auto 불가 → 템플릿 사용
> `void func(auto x)` — C++20 이전은 컴파일 에러.
> C++20 이전엔 `template<typename T> void func(T x)` 로 해결.

### ✅ [C++] 형변환 Type Conversion (4.5)

**암시적 형변환 2종류:**

| 종류 | 방향 | 안전성 | 예시 |
|---|---|---|---|
| **Numeric Promotion** | 작은 → 큰 타입 | ✅ 안전 | `char → int`, `float → double` |
| **Numeric Conversion** | 큰 → 작은 타입 | ⚠️ 위험 | `int → char`, `double → float` |

```cpp
int i = 30000;
char c = i;         // 30000 → 48 (비트 잘림, 반올림 아님!)

double d = 3.99;
int n = d;          // 3.99 → 3 (버림, 반올림 아님!)
```

**명시적 형변환 (Casting):**

```cpp
double d = 4.7;
int n = (int)d;            // C 스타일 — 지양
int n = int(d);            // 함수 스타일
int n = static_cast<int>(d); // ✅ C++ 권장 방식 — 명시적이고 안전
```

> [!warning] unsigned + signed 혼합 주의
> `unsigned int u = -10;` → u는 약 42억으로 해석됨.
> signed와 unsigned를 같이 연산하면 signed가 unsigned로 묵시적 변환된다.

### ✅ [C++] std::string 소개 (4.6)

`std::string`은 C 스타일 문자열(`char[]`)과 달리 **클래스(사용자 정의 자료형)**다.

| 항목 | C 스타일 | std::string |
|---|---|---|
| 타입 | `const char[N]` | `std::string` 클래스 |
| 길이 저장 | 없음 (null `\0` 로 판단) | 별도 멤버 변수에 저장 |
| 연산 | 불편 | `+`, `+=` 연산자 지원 |

```cpp
std::string s1 = "Hello";        // Copy 초기화
std::string s2("Hello");         // Direct 초기화
std::string s3{ "Hello" };       // Uniform 초기화

s1 += " World";                  // 문자열 연결 (operator overloading)
cout << s1.length();             // 멤버 함수 접근은 .() 점 연산자
```

**cin vs getline:**

```cpp
// cin은 공백에서 끊김
cin >> s1;        // "Hello World" 입력 → "Hello" 만 저장

// getline은 줄 전체 입력
std::getline(cin, s1);   // "Hello World" 전부 저장
```

> [!tip] cin 버퍼 문제
> `cin >> int` 후 `getline` 쓰면 버퍼에 남은 `\n` 때문에 빈 줄이 입력됨.
> 해결: `cin.ignore(std::numeric_limits<std::streamsize>::max(), '\n');`

### ✅ [언리얼] 머티리얼 기초 — 에셋 생성과 편집기

**머티리얼 생성**: 콘텐츠 브라우저 빈 공간 우클릭 → Material

**네이밍 컨벤션:**

| 에셋 | 접두사 | 예시 |
|---|---|---|
| 머티리얼 | `M_` | `M_RockWall` |
| 텍스처 | `T_` | `T_RockAlbedo` |
| 블루프린트 | `BP_` | `BP_Player` |
| 머티리얼 인스턴스 | `MI_` | `MI_RockWall_Wet` |

- 공백 금지, 구분은 `_` (언더스코어)
- 에셋 수정 후 별표(`*`) 표시 = **Dirty** 상태 (미저장)
- **UE 5.4 변화**: 메인 머티리얼 노드에서 기본값 인라인 직접 입력 가능 (5.3 이하에서는 Constant 노드 필수였음)

### ✅ [언리얼] 메인 머티리얼 노드 주요 입력값

| 입력 | 자료형 | 설명 |
|---|---|---|
| **Base Color** | Float3 (RGB) | 표면 기본 색상 |
| **Metallic** | Float1 | 금속성 (0=비금속, 1=금속) |
| **Specular** | Float1 | 정반사 강도 (기본 0.5) |
| **Roughness** | Float1 | 거칠기 (0=매끈, 1=거침) |
| **Normal** | Float3 (XYZ) | 표면 법선 방향 |
| **Emissive Color** | Float3 (RGB) | 자체 발광 |

> [!info] 언리얼 Roughness vs 유니티 Smoothness
> 같은 개념을 **반대 방향**으로 표현한다.
> **언리얼 Roughness 0 = 유니티 Smoothness 1** (매끈한 표면)

**노드 생성 4가지 방법:**
1. 빈 공간 **우클릭** → 검색
2. **팔레트** 창에서 드래그앤드롭
3. **키보드 단축키** + 클릭 (`1`=Constant, `3`=Constant3Vector, `T`=Texture Sample)
4. **콘텐츠 브라우저** 텍스처를 그래프로 드래그

### ✅ [언리얼] Float 자료형과 스위즐링(Swizzling)

**Float 자료형:**

```
Float1 → 값 1개 → Roughness, Metallic
Float2 → 값 2개 → UV 좌표
Float3 → 값 3개 → RGB 색상, XYZ 법선
Float4 → 값 4개 → RGBA
```

각 값을 **컴포넌트(Component)** 또는 **채널(Channel)** 이라 부른다.
`RGB` = `XYZ` = `UVW` — 같은 포맷, 쓰임새에 따라 이름만 다름.

**스위즐링** = 채널을 쪼개거나, 합치거나, 순서를 바꾸는 작업

| 노드 | 역할 | 예시 |
|---|---|---|
| **Component Mask** | 원하는 채널만 추출 | Float3(R,G,B) → Float1(R) |
| **Append Vector** | 채널 합치기 | Float1(R) + Float1(G) → Float2 |
| **Break** | 벡터 분해 | Float3 → R, G, B 각각 Float1 |
| **Split** | Break의 머티리얼 함수 버전 | (파란색 노드) |

> [!warning] Component Mask 채널 순서
> R, B 체크해서 Base Color(Float3)에 연결하면 R→Red, B→Green으로 매핑됨.
> **출력은 항상 R→G→B 순서로 채워지기 때문**.

## 2️⃣ Key Insights

- **`using namespace`는 편의 vs 안전의 트레이드오프** — 규모가 커질수록 명시적 `std::` 접두사가 훨씬 안전하다
- **auto는 타입을 숨기는 게 아니라 추론을 컴파일러에 위임하는 것** — 런타임 비용 없고, 초기값이 명확하면 적극 활용 가능
- **형변환에서 "버림"이지 "반올림"이 아니다** — `(int)3.99 == 3`, `(char)30000`은 48이라는 것을 항상 기억
- **unsigned는 음수가 없다는 약속** — signed와 혼합하는 순간 의도치 않은 대형 양수가 된다. 조건문 비교 시 특히 위험
- **std::string은 클래스다** — `+` 연산자가 동작하는 이유가 연산자 오버로딩이고, `.length()` 가 동작하는 이유가 멤버 함수이기 때문
- **언리얼 Float 자료형 = 셰이더의 기본 단위** — RGB든 XYZ든 UVW든 결국 같은 Float3이고, 어디다 쓰느냐가 다를 뿐
- **스위즐링은 셰이더에서 일상적인 작업** — 채널 패킹된 텍스처를 다루거나 UV를 조작할 때 매우 자주 쓰인다
