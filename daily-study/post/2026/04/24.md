### 6.1 배열 기초 — 선언 규칙과 핵심 패턴

#### 배열의 크기는 반드시 컴파일 타임 상수

C++ 스택 배열의 크기는 컴파일 타임에 확정되어야 한다. 컴파일러는 함수 진입 시점에 스택 프레임 전체를 `SUB RSP, N` 한 줄로 확보하기 때문이다.

```cpp
const int num_students = 5;
int students_scores[num_students]; // OK — const + 리터럴

int n;
cin >> n;
int arr[n]; // E0028: expression must have a constant value
```

| 선언 방식 | 컴파일 타임 상수? | 배열 크기 사용 가능? |
|----------|:---:|:---:|
| 리터럴 `int arr[5]` | ✅ | ✅ |
| `const int N = 5` | ✅ | ✅ |
| `constexpr int N = 5` | ✅ | ✅ |
| `cin >> n` 후 `const int N = n` | ❌ | ❌ |

> [!important] `constexpr` vs `const`
> `const`는 "수정 금지"이지 "컴파일 타임 확정"이 아니다. **컴파일 타임을 명시적으로 보장하려면 `constexpr`을 써야 한다.** 런타임 값으로 초기화하면 컴파일 오류가 나서 실수를 조기에 잡아준다.

#### `#define` vs `const` vs `constexpr` — 상수 선언 진화

```cpp
#define NUM_STUDENTS 100000   // 권장하지 않음 — 타입 없음, 스코프 없음, 디버거에서 이름 안 보임
const int NUM_STUDENTS = 5;   // 타입 안전, 스코프 있음
constexpr int NUM_STUDENTS = 5; // 컴파일 타임 보장까지 추가 (현대 C++ 권장)
```

#### enum으로 배열 인덱스에 이름 붙이기

마지막 멤버를 `NUM_STUDENTS`로 두면 배열 크기와 이름이 자동으로 연동된다.

```cpp
enum StudentName { JACKJACK, DASH, VIOLET, NUM_STUDENTS };
// JACKJACK=0, DASH=1, VIOLET=2, NUM_STUDENTS=3

int scores[NUM_STUDENTS] {};
scores[JACKJACK] = 0;
scores[DASH]     = 100;
// 새 학생 추가 시 enum에만 넣으면 배열 크기가 자동으로 증가
```

#### 중괄호 초기화와 sizeof

```cpp
int my_array[] { 1, 2, 3, 4, 5 }; // 크기 자동 추론, = 생략 가능 (C++11)
int arr[5] {};                     // 빈 중괄호 → 전부 0 초기화

// sizeof 관용구
int n = sizeof(my_array) / sizeof(my_array[0]); // 5
```

### 6.2 배열과 포인터 — 주소를 직접 찍어 확인하기

#### 배열 이름은 내부적으로 첫 번째 원소의 주소

```cpp
int students_scores[20] = { 1, 2, 3, 4, 5 };

cout << (int)students_scores     << endl; // 14548006
cout << (int)&students_scores    << endl; // 14548006 (동일)
cout << (int)&students_scores[0] << endl; // 14548006 (동일)
```

세 값이 모두 같다. 배열 이름은 첫 번째 원소를 가리키는 포인터로 **암시적 변환(Decay)** 된다. 단, 타입이 다르다: `students_scores`는 `int*`, `&students_scores`는 `int(*)[20]`.

> [!warning] x64에서 (int) 캐스팅 주의
> 포인터는 x64에서 8바이트. `(int)`로 캐스팅하면 상위 32비트가 잘린다.
> **`(uintptr_t)` 또는 `(size_t)`를 사용하는 것이 이식성에 맞다.**

#### 연속 메모리 확인 — 4바이트씩 증가

직접 실행한 결과:
```
scores[0] 주소: 14547996
scores[1] 주소: 14548000  (+4)
scores[2] 주소: 14548004  (+4)
sizeof(students_scores) in main: 80 (20 * 4)
```

#### 배열을 함수에 전달하면 포인터로 붕괴 (Array Decay)

```cpp
void doSomething(int students_scores[20]) { }
void doSomething(int students_scores[])   { } // 동일
void doSomething(int* students_scores)    { } // 동일 — 셋 다 같음!
```

- **전달되는 것:** 첫 번째 원소의 주소 하나 (4 또는 8 bytes)
- **전달되지 않는 것:** 나머지 79 bytes의 배열 데이터

```
sizeof in main        → 80  (배열 전체: int[20])
sizeof in doSomething → 4   (x86: 포인터 크기)
sizeof in doSomething → 8   (x64: 포인터 크기)
```

#### &students_scores (in doSomething) vs &students_scores[0]

```
main의 students_scores 주소:           11401780
doSomething의 &students_scores:        11401992 ← 포인터 변수 자체의 스택 주소 (다름!)
doSomething의 &students_scores[0]:     11401780 ← 원본 배열의 주소 (main과 동일)
```

doSomething의 `students_scores`는 `11401780`을 값으로 저장하는 **별도의 포인터 변수**다. 이 변수 자체의 위치는 doSomething 스택에 있어 주소가 다르다.

#### x86 vs x64 포인터 크기

| 플랫폼 | 포인터 크기 | 주소 공간 |
|--------|:-----------:|:--------:|
| x86 (32비트) | 4 bytes | 최대 4GB |
| x64 (64비트) | 8 bytes | 최대 16 EB |

Visual Studio에서 빌드 플랫폼을 x86 → x64로 바꾸면 `sizeof(pointer)`: 4 → 8로 바뀐다.

### 6.3 배열과 반복문

#### 변수 나열 방식의 비효율성과 배열 리팩토링

```cpp
// 비효율적: 100명이면 100줄
int score0 = 84, score1 = 92, score2 = 76, score3 = 81, score4 = 56;
int total = score0 + score1 + score2 + score3 + score4;

// 효율적: 100명이 되어도 초기화 리스트만 늘어남
int scores[] = { 84, 92, 76, 81, 56 };
int total = 0;
for (int i = 0; i < 5; ++i) total += scores[i];
```

#### sizeof로 배열 크기 안전하게 구하기

```cpp
const int num_students = sizeof(scores) / sizeof(scores[0]);
// sizeof(scores[0]) 사용 권장 — 타입이 바뀌어도 자동 대응
```

#### 루프 조건 — `<` vs `<=` (Off-by-One Error)

```
배열 크기 5, 유효 인덱스 0~4

i < 5  → 0, 1, 2, 3, 4       ✅ 정상
i <= 5 → 0, 1, 2, 3, 4, 5    ❌ i=5에서 OOB
```

**Off-by-One Error** 는 배열 처리에서 가장 흔한 버그 유형이다. 규칙: **항상 `i < N`.**


## 2️⃣ Key Insights — 오늘 배운 것 중 가장 중요한 것들

> [!success] 인사이트 1 — 배열 크기는 컴파일 타임 상수여야 한다
> `const int`는 값이 런타임에 결정되면 배열 크기로 쓸 수 없다. **`constexpr`이 컴파일 타임을 명시적으로 보장**한다. 이 차이를 모르면 E0028 오류를 만나고 이유를 모른 채 우회하게 된다.

> [!success] 인사이트 2 — 함수에서 sizeof가 배열 크기를 반환하지 않는다
> 배열이 함수 매개변수로 전달되면 포인터로 붕괴된다. `sizeof(arr)` in 함수는 **포인터 크기(4 or 8)** 를 반환한다. main에서 80이던 것이 함수 내에서 4가 되는 것을 직접 주소를 찍어 확인한 오늘 강의가 이 개념을 가장 명확히 설명해준 순간이었다.

> [!success] 인사이트 3 — 나누기 전에 형변환해야 소수가 보존된다
> `static_cast<double>(total) / n`과 `static_cast<double>(total / n)`은 결과가 다르다. 후자는 정수 나눗셈이 먼저 일어나 소수점 이하가 버려진다. **형변환은 나누기 직전 피연산자에** 적용해야 한다.

> [!success] 인사이트 4 — 배열 루프는 반드시 `i < N` 조건을 써야 한다
> `i <= N`은 Off-by-One Error를 만든다. 배열 인덱스가 0부터 시작하는 이유는 "첫 원소까지의 거리"가 0이기 때문이며, 마지막 유효 인덱스는 `N-1`이다. 이 원칙 하나로 범위 초과 버그의 대부분을 방지한다.
