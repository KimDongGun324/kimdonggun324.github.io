### PBR 5대 파라미터 완전 분석

#### Base Color — Metallic에 따라 역할이 완전히 바뀐다

Base Color는 이름과 달리 단순한 "기본 색상"이 아니다. **Metallic 값에 따라 셰이더 내부에서 완전히 다른 물리량**으로 해석된다.

| Metallic 값 | Base Color의 역할 | 물리적 의미 |
|:-----------:|:----------------:|:----------:|
| 0 (비금속) | **알베도(Albedo)** — 난반사 색상 | 빛이 내부로 투과·산란 후 나오는 색 |
| 1 (금속) | **F0** — 정반사 고유색 | 자유전자와 즉시 반응, 난반사 없음 |

```
비금속 셰이더 계산:
  최종 색상  = Diffuse(Base Color) + Specular(F0 고정 4%, Fresnel 적용)
  Base Color → 난반사 색상

금속 셰이더 계산:
  최종 색상  = Specular(F0 = Base Color, Fresnel 적용)
  Diffuse    = 0 (난반사 없음)
  Base Color → 정반사 고유색
```

> [!important] PBR 아티스트의 기준 — 올바른 Base Color 값 범위
> 현실에서 빛을 100% 흡수하거나 100% 반사하는 재질은 없다.
> **권장 범위: sRGB 30~240** (선형 약 0.02~0.9)
> 너무 밝거나 어두운 값은 비물리적(non-physical) 결과를 낸다.

---

#### Metallic — 반드시 0 또는 1만

Metallic은 픽셀이 **금속(1)인지 비금속(0)인지**를 정의하는 이진(Binary) 값이다.

```
Metallic = 0 → 비금속 (플라스틱, 나무, 돌, 피부, 천 등)
Metallic = 1 → 금속   (철, 금, 알루미늄, 크롬 등)
```

**왜 0.5 같은 중간값은 비물리적인가?**
현실의 재질은 금속이거나 비금속이다. 중간값은 물리적으로 존재하지 않는 재질이다. 단, 텍스처의 이웃 픽셀 보간(Bilinear Filtering) 과정에서 경계부에 자연스럽게 중간값이 생기는 것은 허용된다.

---

#### Roughness — 미세면(Microfacet)의 분포

Roughness는 표면의 **미세면(Microfacet) 분포**를 제어한다.

```
Roughness = 0 → 미세면 정렬 → 날카로운 정반사 하이라이트 (거울 느낌)
Roughness = 1 → 미세면 무작위 → 넓게 퍼진 하이라이트 (무광 느낌)
```

| 엔진 | 용어 | 0의 의미 | 1의 의미 |
|------|------|:--------:|:--------:|
| Unreal Engine | Roughness | 매끄러움 | 거침 |
| Unity | Smoothness | 거침 | 매끄러움 |

Unity의 Smoothness 텍스처를 Unreal에 가져올 때 **채널 반전**이 필요하다.

> [!tip] 당구공 현상 이해
> 비금속은 Roughness를 0으로 내려도 완전한 거울이 되지 않는다. 분자 단위에서 항상 기본적인 난반사가 존재하기 때문이다. 실무에서는 최솟값을 0.05 이상으로 설정하는 것이 권장된다.

---

#### Specular — 비금속 F0 리매핑

Specular는 **비금속에서 정반사 하이라이트 강도**를 제어한다. Unreal 내부에서 F0를 리매핑하는 방식으로 작동한다.

```
F0(실제 반사율) = 0.08 × Specular

Specular = 0.5 → F0 = 0.04 (4%)  ← 대부분 비금속의 현실 측정값
Specular = 0.0 → F0 = 0.0  (0%)  ← 물리적으로 불가능
Specular = 1.0 → F0 = 0.08 (8%)  ← 다이아몬드 수준
```

기본값 **0.5** 가 대부분의 비금속에서 물리적으로 정확하다. Metallic이 1이면 Specular 입력은 셰이더에서 무시된다 (금속의 F0는 Base Color에서 직접 읽음).

---

#### Emissive Color — HDR 자체 발광

Emissive는 조명 계산 결과에 **더해지는(Additive)** 방식으로 작동한다. 0~1을 초과하는 **HDR 값** 사용이 가능하며, Bloom Post-Process와 연동해 주변을 밝히는 것처럼 보인다.

```
마스크 텍스처(R) × 발광 색상(Float3) × 강도값(Float1) → Emissive 입력
```

특정 영역만 발광시키려면 그레이스케일 마스크를 곱연산(Multiply)으로 적용한다.

### 금속 vs 비금속 — 빛 반응의 물리적 원리

#### 난반사 vs 정반사

```
비금속 표면:
  입사광 → 표면 통과 → 내부 산란 → 무작위 방향 출사 = 난반사(Diffuse)
          → 표면에서 직접 반사 = 정반사(Specular, F0 ≈ 4%)

금속 표면:
  입사광 → 자유전자와 즉시 상호작용 → 정반사만 발생
  Diffuse = 0 (빛이 내부로 투과 불가)
```

#### Fresnel 효과

관찰 각도에 따라 반사율이 달라지는 현상. 비금속도 비스듬한 각도에서 보면 반사율이 급격히 상승한다.

```
정면(θ ≈ 0°):    반사율 ≈ F0 (비금속 약 4%)
비스듬(θ → 90°): 반사율 → 100% (모든 재질 공통)
```

물 표면을 위에서 보면 바닥이 보이지만, 수평으로 보면 하늘이 반사되는 현상이 바로 Fresnel 효과다. Unreal의 PBR 셰이더는 **Schlick 근사(Schlick Approximation)** 로 이를 자동 계산한다.

### 텍스처 채널 관리 — 컴포넌트 수 일치와 채널 패킹

#### 컴포넌트 수 일치 원칙

Unreal Material Editor에서 입력 슬롯이 요구하는 컴포넌트 수와 공급하는 컴포넌트 수를 정확히 맞춰야 한다.

| 파라미터 | 요구 컴포넌트 | 올바른 연결 |
|---------|:-----------:|:---------:|
| Base Color | 3 (RGB) | Texture.RGB 또는 Float3 |
| Metallic | 1 (Float) | Texture.R 채널 단독 |
| Roughness | 1 (Float) | Texture.R 채널 단독 |
| Specular | 1 (Float) | Texture.R 채널 단독 |
| Normal | 3 (RGB) | Normal Texture.RGB |
| Emissive | 3 (RGB) | Texture.RGB 또는 Float3 |

RGB를 단일 채널 슬롯에 연결하면 Unreal이 자동으로 R만 읽지만, GPU 레지스터에서 `vec3 → float` 변환이 발생해 비효율적이다.

#### ORM 채널 패킹 — VRAM 절약

실무에서는 단일 채널 데이터를 하나의 텍스처에 모아 저장한다.

```
ORM 텍스처 (1장으로 3장 역할):
  R 채널→ Ambient Occlusion
  G 채널→ Roughness
  B 채널→ Metallic

장점:
  텍스처 3장 → 1장 절약 (VRAM 약 1/3)
  텍스처 샘플러 유닛 절약 (Unreal 기본 제한 16개)
  같은 UV 샘플링 → L1/L2 캐시 효율 향상
```

### 현실 측정값 — 알베도 범위와 금속 F0

#### 비금속 알베도 현실 범위

| 재질 | sRGB 값 | 선형값 |
|------|:-------:|:-----:|
| 아스팔트 | ~0.05 | ~0.05 |
| 맨 흙 | ~0.13 | 0.02 |
| 콘크리트 | 0.3~0.4 | 0.07~0.13 |
| 갓 내린 눈 | 0.8~0.9 | 0.6~0.8 |

#### 주요 금속 F0 측정값

```
금(Gold):      R=1.000, G=0.766, B=0.336  (황금빛)
은(Silver):    R=0.972, G=0.960, B=0.915  (밝은 흰빛)
구리(Copper):  R=0.955, G=0.638, B=0.538  (구릿빛)
철(Iron):      R=0.562, G=0.565, B=0.578  (회색)
크롬(Chrome):  R=0.549, G=0.556, B=0.554  (은회색)
알루미늄:      R=0.913, G=0.921, B=0.925  (밝은 은색)
```

금처럼 고유색이 강한 금속일수록 채널 간 편차가 크다. 정확한 측정값을 Base Color에 입력해야 물리적으로 올바른 정반사 색이 나온다.

### Height · Normal · Parallax · Displacement 비교

이 기술들은 모두 **실제 폴리곤을 늘리지 않고 표면 입체감을 부여**하는 기법이다.

| 기법 | 실루엣 변화 | GPU 비용 | 주요 용도 |
|------|:----------:|:-------:|:--------:|
| Bump Map | ✗ | 낮음 | 레거시, 거의 미사용 |
| **Normal Map** | ✗ | 낮음 | 실시간 표면 디테일 표준 |
| Parallax (POM) | ✗ | 중간 | 지면·벽 시차 효과 |
| Displacement | ✅ | 높음 | 오프라인·고품질 렌더링 |

- **Bump Map:** 픽셀 밝기의 기울기(Gradient)로 법선을 교란. Normal Map에 대체됨.
- **Normal Map:** 법선 벡터를 RGB로 인코딩. 픽셀 단위로 음영을 속임.
- **Parallax Occlusion Mapping (POM):** Height Map + 레이마칭으로 카메라 각도에 따라 UV 이동.
- **Displacement:** Height Map으로 버텍스를 물리적으로 이동. 실루엣도 변하고 셀프-섀도우도 정확.

### Normal Map 채널 구조와 인코딩

Normal Map은 접선 공간(Tangent Space)에서의 법선 벡터를 RGB로 인코딩한다.

```
R 채널 → X축 (가로 방향 기울기)
G 채널 → Y축 (세로 방향 기울기)
B 채널 → Z축 (표면 수직, 카메라/관찰자 방향)
```

법선 벡터 범위는 -1~1이지만 텍스처는 0~1만 저장하므로 인코딩/디코딩이 필요하다.

```
저장(Pack):   normal_stored = normal * 0.5 + 0.5
읽기(Unpack): normal        = normal_stored * 2.0 - 1.0

중간값(0.5) → 법선 성분 0 (해당 축 기울기 없음)
흰색(1.0)   → 법선 성분 +1
검정(0.0)   → 법선 성분 -1
```

**왜 Normal Map은 파랗게 보이는가?**
평평한 표면의 법선은 (0, 0, 1). 인코딩하면 (0.5, 0.5, 1.0), 즉 RGB (128, 128, 255) → 파란색. 경사진 부분에서만 R과 G 채널에 변화가 생기므로 파란 배경에 붉은/초록 노이즈가 혼재하는 모습이 된다.

#### Unreal vs Unity — G 채널 방향 차이

```
Unreal (OpenGL 방식):   G 채널 = Y+ 방향 (위쪽 기울기)
Unity  (DirectX 방식):  G 채널 = Y+ 반전  (아래쪽 기울기)
```

| 변환 도구 | 방법 |
|-----------|------|
| 포토샵 | G 채널 선택 → Ctrl+I 반전 |
| Unreal | Texture 에셋 → Flip Green Channel |
| Substance Painter | 익스포트 시 타겟 엔진 선택 (자동 처리) |

> [!warning] Normal Map 이식 시 반드시 확인
> G 채널 방향을 확인하지 않으면 볼록한 부분이 오목하게, 오목한 부분이 볼록하게 보인다. "표면이 뒤집힌" 것처럼 보이면 G 채널 반전 문제다.

### sRGB vs Linear — 텍스처 색 공간 관리

#### sRGB의 원리

```
sRGB 인코딩:   저장값 = 선형값 ^ (1/2.2)
모니터 디코딩: 표시값 = 저장값 ^ 2.2    → 원래 선형값 복원
```

인간의 눈은 어두운 영역에 더 민감하다. sRGB는 이 지각 특성에 맞춰 어두운 영역에 더 많은 비트를 할당한다.

#### 텍스처별 sRGB 설정 기준

| 텍스처 종류 | sRGB 설정 | 이유 |
|------------|:--------:|:----:|
| Base Color | ✅ 켜짐 | 색상 데이터, 감마 보정 필요 |
| Emissive Color | ✅ 켜짐 | 색상 데이터 |
| Normal Map | ❌ 꺼짐 | 수치 벡터 데이터, 보정 금지 |
| Roughness | ❌ 꺼짐 | 수치 데이터, 보정 시 왜곡 |
| Metallic | ❌ 꺼짐 | 수치 데이터 |
| AO | ❌ 꺼짐 | 수치 데이터 |

**sRGB를 잘못 설정하면 무슨 일이 일어나는가?**

```
Roughness 텍스처 + sRGB 켜짐 (잘못된 설정):
  저장값 0.5 → GPU가 0.5^2.2 ≈ 0.22 로 읽음
  → Roughness가 실제보다 훨씬 낮아짐
  → 재질이 예상보다 훨씬 매끄럽게(광택 있게) 보임
```

### UE5 Lumen과 Ambient Occlusion 변화

**Static Lighting 허용(Allow Static Lighting)** 설정에 따라 AO 동작이 달라진다.

```
Static Lighting ON (기본값):
  AO 슬롯 → Lumen이 라이트맵에 양보
  AO 텍스처 효과가 거의 사라짐

Static Lighting OFF (Lumen 전용):
  AO 슬롯 → 정상 동작
  Lumen GI와 조합해 정밀한 간접광 차폐 표현
```

Lumen 환경에서 AO를 제대로 활용하려면 **Allow Static Lighting을 끄는 것**이 전제조건이다.

## 2️⃣ Key Insights — 오늘 배운 것 중 가장 중요한 것들

> [!success] 인사이트 1 — Base Color의 이중 역할
> Base Color는 단순한 색상이 아니다. **Metallic 값에 따라 알베도(비금속) 또는 F0(금속)** 로 완전히 다르게 해석된다. 같은 슬롯이 두 가지 물리량을 담당하는 이 구조를 이해해야 PBR 머티리얼을 제대로 제작할 수 있다.

> [!success] 인사이트 2 — sRGB 설정 하나가 렌더링을 망친다
> Roughness 텍스처에 sRGB를 켜면 `0.5 → 0.22` 로 왜곡된다. 단순한 체크박스 하나가 **전체 머티리얼의 물리적 정확성**을 무너뜨린다. 데이터 텍스처와 색상 텍스처를 구분하는 기준을 반드시 내재화해야 한다.

> [!success] 인사이트 3 — Normal Map은 법선 벡터 수식의 텍스처화
> `normal_stored = normal * 0.5 + 0.5` 라는 Pack/Unpack 공식을 이해하면 Normal Map이 왜 파랗게 보이는지, Unreal과 Unity의 G 채널이 왜 반대인지 자동으로 이해된다. 이 공식은 HLSL 셰이더에서 직접 구현할 때도 동일하게 사용된다.

> [!success] 인사이트 4 — GBuffer와 PBR 파라미터의 연결
> PBR 파라미터는 Deferred Rendering의 GBuffer를 통해 Lighting Pass로 전달된다. Base Color·Metallic·Roughness·Normal 각각이 어떤 GBuffer 채널에 어떤 형식으로 저장되는지 알아야 셰이더 엔지니어 수준의 이해가 완성된다.
