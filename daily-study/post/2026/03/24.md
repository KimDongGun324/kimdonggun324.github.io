## 1. 심볼릭 상수란?

**의미 있는 이름을 붙인 상수.** 코드에 숫자를 직접 쓰는 대신 이름으로 대체한다.

```cpp
// ❌ 매직 넘버 — 30이 무엇인지 알 수 없다
int price = num_item * 30;

// ✅ 심볼릭 상수 — 이름 자체가 의미를 설명한다
const int price_per_item = 30;
int price = num_item * price_per_item;
```

C++에서 심볼릭 상수를 만드는 방법은 **3가지**다:

| 방법 | 키워드 | 권장 여부 |
|---|---|---|
| 전처리기 매크로 | `#define` | ❌ 비권장 |
| 상수 선언 | `const` | ✅ |
| 컴파일 타임 상수 | `constexpr` | ✅✅ 최권장 |

---

## 2. `#define` — 왜 C++에서 안 쓰는가

```cpp
#define PRICE_PER_ITEM 30
// 전처리기가 컴파일 이전에 PRICE_PER_ITEM을 전부 30으로 텍스트 치환
// 컴파일러는 PRICE_PER_ITEM이라는 이름 자체를 모른다
```

### 사용하지 않는 3가지 이유

**① 타입이 없다**

```cpp
#define PRICE_PER_ITEM 30        // 30이 int인지 double인지 알 수 없음
const int price_per_item = 30;   // int라는 타입 정보가 명확 → 컴파일러가 검사 가능
```

**② 스코프(범위)가 없다**

```cpp
#define MAX 100  // 선언 이후 파일 끝까지 모든 곳에 영향을 줌

void funcA() { int x = MAX; }  // 영향받음
void funcB() { int y = MAX; }  // 영향받음 — 의도한 것인지 알 수 없음
```

`const`는 선언된 `{ }` **블록 안에서만** 유효:

```cpp
int main() {
    const int price_per_item = 30;  // main() 블록 안에서만 유효
}
// 다른 함수에는 전혀 영향 없음 → 안전하고 예측 가능
```

**③ 디버깅이 어렵다**

```
전처리 전:  int price = num_item * PRICE_PER_ITEM;
전처리 후:  int price = num_item * 30;   ← 이름이 완전히 사라짐
```

> [!warning] 디버깅 시나리오
> `price` 값이 이상할 때 디버거를 갖다 대면 숫자 `30`만 보인다.
> 이 30이 어디서 왔는지 알려면 소스 코드를 직접 뒤져야 한다.
> `const`였다면 디버거가 `price_per_item = 30` 이라고 바로 보여준다.

## 3. `const` — 기본 심볼릭 상수

### 기본 사용법

```cpp
const int    price_per_item = 30;
const double gravity        = 9.8;
```

### const의 핵심 목적 — 실수 방지

다른 프로그래머가 **실수로 값을 바꾸는 것을 컴파일러가 막아준다.**

```cpp
const double gravity = 9.8;
gravity = 2.0;  // ❌ 컴파일 에러 — 컴파일러가 즉시 차단
```

> [!danger] `const_cast`로 강제 변경 — 절대 비권장
> C++에는 `const`를 강제로 무력화하는 방법이 있지만 **사용하지 말 것.**
> 값이 바뀌어야 하는 상황이라면 처음부터 변수로 선언하거나 다른 기법을 사용하자.

### 반드시 선언 시 초기화해야 한다

값을 나중에 바꿀 수 없으므로, **선언과 동시에 초기값을 넣어야 한다.**

```cpp
const int x;        // ❌ 컴파일 에러 — 초기값 없음
const int x = 10;   // ✅ 선언과 동시에 초기화
```

### const 위치 — 앞과 뒤

일반 변수 선언에서 `const`는 앞이든 뒤든 **의미가 같다.** 관습적으로 앞에 붙인다.

```cpp
const double gravity = 9.8;   // ✅ 일반적 스타일 (앞) — 권장
double const gravity = 9.8;   // ✅ 동일한 의미 (뒤)
```

> [!caution] 포인터에서는 위치가 달라지면 의미가 달라진다
> 지금은 "앞에 붙이는 습관"만 들여두고,
> 포인터·레퍼런스를 배울 때 위치에 따른 차이를 다시 공부한다.

## 4. `constexpr` — 컴파일 타임 상수 (C++11~)

### `const`와의 핵심 차이: 값이 언제 결정되는가

```cpp
// ── const: 런타임 상수도 가능 ────────────────────────────────
int number;
cin >> number;
const int special_number(number);  // ✅ 실행 중에 값이 결정됨 — OK

// ── constexpr: 반드시 컴파일 타임 ────────────────────────────
constexpr int my_const(123);       // ✅ 컴파일 시점에 이미 확정

constexpr int x = number;         // ❌ 컴파일 에러 — 런타임 값은 받을 수 없음
```

| 구분 | 값 결정 시점 | 런타임 값 수용 | 성능 |
|---|---|---|---|
| `const` | 런타임 or 컴파일 타임 | ✅ 가능 | 보통 |
| `constexpr` | 반드시 컴파일 타임 | ❌ 불가 | 우수 (런타임 비용 없음) |

> [!tip] 선택 기준
> - 값이 코드에 고정 → `constexpr` 우선
> - 값이 실행 중에 결정 (사용자 입력 등) → `const`

## 5. 함수 파라미터에 `const` 사용하기

### 파라미터가 전달되는 원리

함수를 호출하면, 전달한 값이 파라미터 변수에 **복사**되어 들어온다.

```cpp
void printNumber(int my_number)  // 123이 my_number로 복사됨
{
    cout << my_number << endl;
}
printNumber(123);
```

### 왜 파라미터에 `const`를 붙이는가

파라미터는 함수의 **입력**이다. 내부에서 몰래 바꾸면 코드의 의도를 파악하기 어렵다.

```cpp
// ❌ 나쁜 예 — 입력값을 함수 안에서 바꿔버림
void printNumber(int my_number)
{
    my_number = 456;            // 원래 입력(123)이 사라짐
    cout << my_number << endl;  // 456 출력 — 호출자는 전혀 예상 못 함
}

// ✅ 좋은 예 — const로 막고, 바꿔야 하면 복사해서 사용
void printNumber(const int my_number)
{
    // my_number = 456;  ← ❌ 컴파일 에러로 차단
    int n = my_number;  // ✅ 복사 후 자유롭게 사용
    n = 456;
    cout << n << endl;
}
```

### `const int&` 파라미터 — 오픈소스에서 자주 보임

```cpp
void printNumber(const int& my_number)
//                         ^ 참조(Reference) — 복사 없이 원본을 바라봄
```

- `const` → 원본을 읽기만 할 수 있음 (수정 불가)
- `&` → 복사 없이 원본 참조 (큰 객체 전달 시 성능 유리)
- 레퍼런스는 뒤에서 자세히 배운다 → **지금은 오픈소스에서 이런 형태가 나온다는 것만 인지**

> [!tip] 파라미터에 const를 붙이는 이유 요약
> - "이 파라미터는 읽기 전용"임을 코드만 보고 즉시 파악 가능
> - 실수로 파라미터를 수정하는 버그를 컴파일 타임에 원천 차단
> - 실무 코드에서 매우 흔하게 쓰이므로 반드시 익숙해지자

## 6. 상수를 헤더 파일 + `namespace`로 관리하기

상수가 많아지면 코드 곳곳에 흩어지기 쉽다.
→ **전용 헤더 파일**에 모아두고 `namespace`로 묶어 관리하는 것이 표준 방식.

### `MY_CONSTANTS.h` 작성

```cpp
#pragma once  // 헤더 가드 — 같은 헤더가 중복 include되는 것을 방지

namespace constants
{
    constexpr double pi       = 3.14159265358979;
    constexpr double avogadro = 6.02214076e23;    // 아보가드로 수
    constexpr double gravity  = 9.80665;           // 중력 가속도 (m/s²)
}
```

### `main.cpp`에서 사용

```cpp
#include <iostream>
#include "MY_CONSTANTS.h"

using namespace std;

int main()
{
    double radius;
    cin >> radius;

    double circumference = 2.0 * radius * constants::pi;
    //                                    ^^^^^^^^^^^ namespace::상수명

    cout << circumference << endl;
    return 0;
}
```

### `::` 스코프 해결 연산자

```cpp
constants::pi   // constants namespace의 pi
std::cout       // std namespace의 cout  ← 이미 익숙한 것과 같은 원리
```

### 헤더 파일 관리의 장점

| 장점 | 설명 |
|---|---|
| **재사용성** | 여러 `.cpp`에서 `#include`만 하면 바로 사용 |
| **일괄 수정** | 값을 바꿔야 할 때 헤더 한 곳만 수정 → 전체 반영 |
| **가독성** | 어떤 상수가 있는지 한눈에 파악 |
| **충돌 방지** | `namespace`로 다른 라이브러리와 이름 충돌 원천 차단 |

> [!example] 실전 예시 — 게임 엔진
> ```cpp
> namespace Physics {
>     constexpr double gravity  = 9.8;
>     constexpr double friction = 0.85;
> }
> // 중력을 달 기준(1.62)으로 바꿔야 한다면?
> // → MY_CONSTANTS.h 한 줄만 수정하면 전체 프로그램에 즉시 반영
> ```

## 📌 전체 요약

```
심볼릭 상수 (Symbolic Constants)
│
├── #define  ❌ 비권장
│   ├── 타입 없음  → 타입 검사 불가, 버그 위험
│   ├── 스코프 없음 → 파일 전체에 영향
│   └── 디버거에 이름이 사라짐 → 디버깅 어려움
│
├── const  ✅
│   ├── 블록 스코프 — 선언된 { } 안에서만 유효
│   ├── 선언 시 반드시 초기화
│   ├── 런타임 값도 수용 가능
│   ├── 위치: 앞뒤 상관없음 (포인터에서는 다름, 나중에 학습)
│   └── 파라미터에도 사용 → 입력값 보호, 실무에서 매우 흔함
│
├── constexpr  ✅✅ (C++11~)
│   ├── 반드시 컴파일 타임에 값 확정
│   └── 런타임 비용 없음 → 성능 우수, 값이 고정이면 최우선 사용
│
└── 상수가 많아지면
    └── 헤더 파일(MY_CONSTANTS.h) + namespace 로 관리
        └── constants::pi 처럼 명시적으로 사용
