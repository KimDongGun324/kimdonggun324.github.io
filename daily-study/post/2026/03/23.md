# C++ 리터럴 상수 (Literal Constants)

---

## 📄 강의 예제 코드

```cpp
#include <iostream>

int main()
{
    using namespace std;

    // ── 타입 접미사 ──────────────────────────────────────────
    unsigned int i = 5u;      // u  → unsigned int
    long         n2 = 5L;     // L  → long
    double       d  = 6.0e-10;// e  → 지수 표기법 (6.0 × 10⁻¹⁰)

    // ── 진수 표기 ─────────────────────────────────────────────
    // int x = 012;            // 8진수  → 10진수 10  (앞에 0)
    // int x = 0xF;            // 16진수 → 10진수 15  (앞에 0x)
    int x = 0b1010;            // 2진수  → 10진수 10  (앞에 0b, C++14~)

    // 0011 1010 1111  ← 긴 이진수는 읽기 힘들다
    // 16진수로 바꾸면: 0x3AF  ← 훨씬 짧고 읽기 쉽다

    // ── 자릿수 구분자 (C++14~) ────────────────────────────────
    int bin = 0b1011'1111'1010'1010;  // ' 는 컴파일러가 무시 → 가독성만 향상
    int big = 1'000'000;

    // ── 매직 넘버 vs 심볼릭 상수 ──────────────────────────────
    // ❌ 나쁜 예: 10이 무엇인지 알 수 없다
    // int price = num_items * 10;

    // ✅ 좋은 예: 이름 자체가 의미를 설명한다
    const int price_per_item = 10;   // 심볼릭 상수
    int num_items = 123;
    int price     = num_items * price_per_item;

    cout << x << endl;

    return 0;
}
```

## 1. 리터럴 상수 (Literal Constant) 란?

소스 코드에 **직접 값을 써 넣은 것**. 변수처럼 이름이 없고, 값 그 자체가 상수다.

```cpp
int x = 42;      // 42 가 리터럴 상수
double d = 3.14; // 3.14 가 리터럴 상수
```

리터럴에는 두 가지 속성이 있다:
- **값(Value)**: 실제 숫자나 문자
- **타입(Type)**: 컴파일러가 기본값으로 추론하거나, 접미사로 명시

---

## 2. 타입 접미사 (Type Suffix)

리터럴 뒤에 접미사를 붙여 타입을 **명시적으로 지정**한다.
붙이지 않으면 컴파일러가 기본 타입(`int`, `double`)으로 추론한다.

```cpp
unsigned int i = 5u;      // u / U → unsigned int
long         n = 5L;      // l / L → long
float        f = 3.14f;   // f / F → float
double       d = 6.0e-10; // e     → 지수 표기법 (= 6.0 × 10⁻¹⁰)
```

| 접미사       | 타입               | 예시        |
| --------- | ---------------- | --------- |
| `u` / `U` | `unsigned int`   | `5u`      |
| `l` / `L` | `long`           | `5L`      |
| `f` / `F` | `float`          | `3.14f`   |
| `e` / `E` | 지수 표기 (`double`) | `6.0e-10` |

> [!tip] 접미사가 중요한 이유
> `float pi = 3.14;` 는 `3.14`가 `double`이라 암묵적 변환이 일어난다.
> `float pi = 3.14f;` 로 쓰면 처음부터 `float`로 처리 → 변환 없음, 경고 없음.

---

## 3. 진수 표기법 (Number Base Notation)

C++은 10진수 외에도 **8진수, 16진수, 2진수**를 리터럴로 표현할 수 있다.

```cpp
int a = 012;    // 8진수  → 앞에 0    → 10진수 10
int b = 0xF;    // 16진수 → 앞에 0x   → 10진수 15
int c = 0b1010; // 2진수  → 앞에 0b   → 10진수 10  (C++14~)
```

### 변환 원리

```
8진수  012   → 1×8¹ + 2×8⁰       = 8 + 2      = 10
16진수 0xF   → F = 15             = 15
2진수  0b1010→ 1×2³ + 0×2² + 1×2¹ + 0×2⁰ = 8+0+2+0 = 10
```

### 진수 대응표

| 10진수 | 2진수 | 8진수 | 16진수 |
|---|---|---|---|
| 0 | 0000 | 0 | 0 |
| 8 | 1000 | 10 | 8 |
| 9 | 1001 | 11 | 9 |
| 10 | 1010 | 12 | A |
| 11 | 1011 | 13 | B |
| 15 | 1111 | 17 | F |

> [!warning] 실수로 앞에 0 붙이지 말 것!
> ```cpp
> int x = 012;  // 10진수 12가 아니라 8진수 → 실제 값은 10!
> ```
> 8진수로 쓰고 싶을 때**만** 앞에 `0`을 붙인다.

---

## 4. 16진수를 실무에서 많이 쓰는 이유

이진수는 비트가 많아지면 읽기가 매우 힘들다. 16진수는 **4비트 = 16진수 1자리**이므로 훨씬 짧아진다.

```
2진수:  0011  1010  1111   ← 12자리, 읽기 힘들다
16진수:    3     A     F   ← 3자리, 깔끔하다
```

- 오픈소스 코드, 시스템 프로그래밍에서 자주 등장
- **다른 사람의 코드를 읽으려면 반드시 알아야 한다**
- 직접 이진수를 쓰는 것보다 16진수가 훨씬 간결

---

## 5. 자릿수 구분자 (Digit Separator) — C++14~

긴 숫자를 `'`(작은따옴표)로 끊어 읽기 쉽게 만들 수 있다.
**컴파일러는 `'`를 완전히 무시**하므로 값에 영향 없음.

```cpp
int bin = 0b1011'1111'1010'1010;  // 4비트씩 끊어 읽기 편함
int big = 1'000'000;              // 백만 → 10진수에도 사용 가능
int hex = 0xFF'EC'D1'B2;          // 16진수에도 사용 가능
```

> [!info] 왜 유용한가?
> 이진수를 직접 다루는 하드웨어/임베디드 코드에서 특히 유용하다.
> 모르는 상태에서 남의 코드에서 보면 "왜 빌드가 되지?" 할 수 있으니 알아두자.

## 6. 매직 넘버 (Magic Number)

> [!danger] 매직 넘버는 쓰지 말 것
> 의미 불명의 숫자를 코드에 하드코딩하면, 나중에 본인도 이해 못 한다.

### 정의

**소스 코드에 직접 박혀 있는, 의미를 알 수 없는 숫자(또는 문자열).**
"왜 이 숫자인지 모르지만 마법처럼 동작한다"는 데서 이름이 유래했다.

```cpp
// ❌ 10이 무엇인지 전혀 알 수 없다
int price = num_items * 10;

// ❌ 더 나쁜 경우 — 같은 숫자가 전혀 다른 의미로 반복된다
int price    = num_items * 10;  // 여기서 10 = 개당 가격
int discount = total - 10;      // 여기서 10 = 할인 금액 → 완전히 다른 의미!
```

### 왜 문제인가?

| 문제점 | 설명 |
|---|---|
| **가독성 저하** | 숫자만 봐서는 의미를 전혀 알 수 없음 |
| **유지보수 어려움** | 값이 바뀌면 코드 전체를 검색해서 일일이 수정해야 함 |
| **중복·혼동 위험** | 같은 숫자가 다른 의미로 여러 곳에 쓰일 수 있음 |
| **버그 유발** | 관련 없는 동일 숫자를 실수로 함께 수정하는 사고 발생 |

## 7. 심볼릭 상수 (Symbolic Constant) — 매직 넘버의 해결책

의미 있는 **이름을 붙인 상수**. 이름 자체가 주석 역할을 한다.

### 방법 1: `const` — 기본

```cpp
const int price_per_item = 10;           // 이름이 의미를 설명
int price = num_items * price_per_item;  // 코드 의도가 명확
```

- 컴파일러가 **타입 검사** 수행
- 선언 후 **값 변경 불가** (변경 시 컴파일 에러)
- **블록 스코프** 적용 — 선언된 `{ }` 안에서만 유효
- **런타임 값**도 받을 수 있음

```cpp
int input;
cin >> input;
const int max_val = input;  // 런타임에 결정된 값도 OK
```

#### const의 필요성 — 실수 방지

`const`의 핵심 목적은 **나 혹은 다른 프로그래머가 실수로 값을 바꾸는 것을 컴파일러가 막아주는 것**이다.

```cpp
const double gravity = 9.8;

// 다른 프로그래머가 실수로 바꾸려 하면?
gravity = 2.0;  // ❌ 컴파일 에러 — 컴파일러가 막아준다
```

> [!warning] 특수 방법으로 강제 변경하는 것은 절대 비권장
> C++에는 `const_cast` 등 `const`를 강제로 무력화하는 방법이 있지만,
> **어떠한 경우에도 사용하지 말 것**.
> 중력이 변해야 하는 상황이라면 처음부터 변수로 두고 주석으로 설명하거나,
> 다른 프로그래밍 기법을 사용하는 것이 옳다.

#### const 위치 — 앞과 뒤 중 어디에?

일반 변수 선언에서는 `const`를 앞에 붙이나 뒤에 붙이나 **의미가 같다.**

```cpp
const int x = 10;   // ✅ 일반적 스타일 (앞에 붙임 — 권장)
int const x = 10;   // ✅ 동일한 의미 (뒤에 붙임)
```

> [!caution] 포인터에서는 위치가 달라지면 의미가 달라진다
> 지금은 같다고 이해하면 되고,
> 포인터·레퍼런스를 배울 때 `const` 위치에 따른 차이를 다시 다룬다.
> → **지금은 항상 앞에 붙이는 습관을 들이자.**

#### const는 반드시 초기화해야 한다

상수는 선언 후 값을 바꿀 수 없으므로, **선언할 때 반드시 초기값을 넣어야 한다.**
초기화 없이 선언하면 컴파일 에러가 발생한다.

```cpp
const int x;        // ❌ 컴파일 에러 — 초기화 없음
const int x = 10;   // ✅ 반드시 이렇게
```

> [!info] 당연한 이유
> 값을 바꿀 수 없는데 초기값도 없으면, 그 상수는 영원히 쓸모가 없다.
> 컴파일러가 논리적으로 이를 막아준다.

#### const를 함수 파라미터에 사용하기

함수 파라미터에도 `const`를 붙여 **함수 내부에서 파라미터 값을 바꾸지 못하게** 막을 수 있다.

```cpp
// ✅ 파라미터에 const — 함수 내부에서 my_number를 바꿀 수 없음
void print_number(const int my_number)
{
    cout << my_number << endl;
    // my_number = 999;  ← ❌ 컴파일 에러
}
```

**왜 파라미터에 const를 붙이는가?**

```cpp
// ❌ const 없는 경우 — 다른 프로그래머가 이렇게 짤 수 있다
void print_number(int my_number)
{
    my_number = 456;   // 파라미터 값을 함수 안에서 바꿔버림
    cout << my_number << endl;  // 원래 의도와 다른 출력
}
```

파라미터는 함수의 **입력**이다. 입력값을 함수 내부에서 몰래 바꾸면 입출력 관계가 불명확해진다.
→ **파라미터를 바꿔야 할 때는 복사해서 사용하는 것이 원칙이다.**

```cpp
void print_number(const int my_number)
{
    int n = my_number;  // ✅ 복사해서 바꾼다
    n = 456;
    cout << n << endl;
}
```

> [!tip] 파라미터에 const를 붙이는 것이 좋은 습관인 이유
> - 함수를 보는 사람이 "이 파라미터는 읽기 전용"임을 즉시 파악한다
> - 실수로 파라미터를 수정하는 버그를 컴파일 타임에 차단한다
> - 실무 코드에서 파라미터에 `const`가 붙은 경우가 매우 많으니 익숙해지자

### 방법 2: `constexpr` — C++11~, 최권장

```cpp
constexpr int price_per_item = 10;  // 컴파일 타임에 값이 확정됨
```

- **컴파일 타임 상수** — 반드시 컴파일 시점에 값이 결정되어야 함
- 런타임 비용 없음 → **성능상 유리**
- `const`보다 더 강한 제약 = 더 안전
- 배열 크기 등 컴파일 타임 상수가 필요한 곳에 사용 가능

```cpp
constexpr int max_size = 100;
int arr[max_size];  // ✅ 가능
```

### 방법 3: `#define` — 구식, 비권장

```cpp
#define PRICE_PER_ITEM 10  // 전처리기 매크로 (관습적으로 대문자)
```

- C 스타일의 옛날 방법
- **타입 없음** → 타입 검사 불가 → 버그 위험
- **스코프 없음** → 파일 전체에 영향
- **디버거에 표시 안 됨** → 디버깅 어려움
- C++에서는 `const`/`constexpr`이 있으므로 **사용 지양**

## 8. `#define`의 문제점 — 심화

### 문제 1: 디버거에서 값이 보이지 않는다

`#define`은 전처리기가 컴파일 **이전에** 텍스트를 단순 치환한다.
컴파일러는 `PRICE_PER_ITEM`이라는 이름 자체를 모른다.

```
전처리 전: int price = num_items * PRICE_PER_ITEM;
전처리 후: int price = num_items * 30;   ← 이름이 사라짐
```

> [!warning] 디버깅 시나리오
> `price` 값이 이상할 때, 디버거를 갖다 대면 그냥 숫자 `30`만 보인다.
> `30`이 어디서 왔는지 알려면 소스 코드에서 `#define`을 직접 찾아야 한다.
> `const`/`constexpr`이었다면 디버거가 이름과 값을 함께 보여준다.

### 문제 2: 스코프(Scope)가 파일 전체다

`#define`은 선언 이후 **해당 `.cpp` 파일 끝까지** 모든 코드에 영향을 준다.

```cpp
#define MAX 100  // ← 이 줄부터 파일 끝까지 모든 MAX가 100으로 치환됨

void funcA() { int x = MAX; }  // 영향받음
void funcB() { int y = MAX; }  // 영향받음 — 의도한 건지 모름!
void funcC() { int z = MAX; }  // 영향받음
```

반면 `const`는 **선언된 블록 안에서만** 유효하다:

```cpp
void funcA() {
    const int max = 100;  // ← 이 { } 안에서만 유효
    int x = max;
}
// funcB, funcC에는 전혀 영향 없음 → 안전하고 예측 가능
```

> [!info] 스코프(Scope)란?
> 변수·상수가 **살아있는 범위**. `{ }`가 경계.
> `const`/`constexpr`은 스코프를 따르지만, `#define`은 스코프를 **무시**한다.

## 9. 상수 명명 관습 (Naming Convention)

| 방법 | 관습 | 예시 | 비고 |
|---|---|---|---|
| `#define` | **전체 대문자 + 언더스코어** | `PRICE_PER_ITEM` | 코드 사이에서 눈에 띔 |
| `const`/`constexpr` | `k` 접두사 | `kGravity`, `kPi` | 물리/게임 엔진에서 자주 사용 |
| `const`/`constexpr` | **일반 소문자** | `gravity`, `pi` | 현대 C++ 권장 스타일 |

> [!tip] 현대 C++에서 일반 소문자를 쓰는 이유
> 나중에 상수 → 변수로 전환할 때 **이름을 바꿀 필요가 없다**.
> 대문자 매크로 이름은 변수로 바꾸면 어색해서 전부 리네이밍해야 한다.
> ```cpp
> // const 제거만으로 쉽게 변수로 전환 가능
> constexpr double price_per_item = 10.0;
> // double price_per_item = 10.0;  ← 이름 그대로 재사용
> ```

## 10. 상수를 헤더 파일 + namespace로 관리하기

프로그램 규모가 커지면 상수가 여러 파일에 흩어진다.
→ **전용 헤더 파일**에 모아두고 `namespace`로 묶는 것이 표준 방식.

### 파일 구조

```
project/
├── main.cpp
├── MyConstants.h    ← 상수 전용 헤더
└── physics.cpp
```

### `MyConstants.h` 작성

```cpp
#pragma once

namespace MyConstants
{
    constexpr double pi       = 3.14159265358979;
    constexpr double avogadro = 6.02214076e23;   // 아보가드로 수
    constexpr double gravity  = 9.80665;          // 중력 가속도 (m/s²)
    constexpr int    max_players = 4;
}
```

### `main.cpp` 에서 사용

```cpp
#include <iostream>
#include "MyConstants.h"

int main()
{
    using namespace std;

    double radius;
    cin >> radius;

    // MyConstants:: 로 어느 namespace인지 명시
    double circumference = 2.0 * radius * MyConstants::pi;
    cout << circumference << endl;

    return 0;
}
```

### `::` 스코프 해결 연산자

```cpp
MyConstants::pi       // MyConstants namespace 의 pi
std::cout             // std namespace 의 cout  ← 이미 익숙한 것과 같은 원리
```

### 헤더 파일 관리의 장점

| 장점 | 설명 |
|---|---|
| **재사용성** | 여러 `.cpp`에서 `#include`만 하면 바로 사용 |
| **일괄 수정** | 중력 상수를 바꾸면 헤더 한 곳만 수정 → 전체 반영 |
| **가독성** | 어떤 상수가 있는지 한눈에 파악 |
| **충돌 방지** | `namespace`로 이름 충돌 원천 차단 |

> [!NOTE]
> > [!example] 게임 엔진 활용 예
> > ```cpp
> > // GameConstants.h
> > namespace Physics {
> >     constexpr double gravity  = 9.8;
> >     constexpr double friction = 0.85;
> > }
> > namespace Screen {
> >     constexpr int width  = 1920;
> >     constexpr int height = 1080;
> > }
> >
> > // 사용
> > double force = mass * Physics::gravity;
> > ```
> > 중력을 달 기준(1.62)으로 바꿔야 한다면? → 헤더 한 줄만 수정!

## 📌 전체 흐름 요약

```
리터럴 상수 (Literal Constants)
│
├── 1. 타입 접미사     u, L, f, e  →  타입을 명시적으로 지정
│
├── 2. 진수 표기
│   ├── 0    →  8진수   (주의: 실수로 붙이면 버그!)
│   ├── 0x   →  16진수  (오픈소스에서 자주 등장, 반드시 읽을 줄 알아야)
│   └── 0b   →  2진수   (C++14~, 이진수 직접 표현)
│
├── 3. 자릿수 구분자   ' (C++14~)  →  컴파일러 무시, 가독성만 향상
│
└── 4. 매직 넘버 문제
    └── 심볼릭 상수로 해결
        ├── #define    →  ❌ 타입없음·스코프없음·디버깅불가  (비권장)
        ├── const      →  ✅ 블록 스코프, 런타임 값 가능
        ├── constexpr  →  ✅✅ 컴파일 타임 상수, 최권장 (C++11~)
        └── 상수 다수  →  헤더 파일 + namespace 로 모아서 관리
